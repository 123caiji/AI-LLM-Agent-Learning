## 一、AI 推理服务总览

### 1.1 基础定义

**AI 推理服务(Inference Server):** 加载训练好的模型权重,接收客户端请求,在 GPU/CPU 上执行前向计算,返回模型输出的服务端程序。

### 1.2 AI 推理 vs 传统 Web 服务

| 维度 | 传统 Web 服务 | AI 推理服务 |
|------|---------------|-------------|
| **计算资源** | CPU 为主 | GPU 为主 |
| **请求耗时** | 毫秒级 | 秒级~分钟级 |
| **批处理** | 一般不需要 | 关键优化(提升吞吐数倍) |
| **内存占用** | 可控 | 模型 + KV Cache 显存大 |
| **冷启动** | 秒级 | 分钟级(模型加载) |
| **流式输出** | 一般无 | 关键能力(SSE) |
| **请求长度** | 固定/小 | 差异极大(几 Token ~ 百万 Token) |
| **状态管理** | 通常无状态 | 有状态(KV Cache) |
| **负载均衡** | 轮询/最少连接 | 需考虑 KV Cache 亲和性 |

### 1.3 推理服务的核心能力

```
┌──────────────────────────────────────────────────────────┐
│                  AI 推理服务核心能力                       │
│                                                          │
│  [加载层]                                                 │
│    ├── 模型权重加载(GPU/CPU/混合)                        │
│    ├── 量化加载(INT8/INT4/NF4)                          │
│    ├── 多 GPU 分布式加载(TP/PP)                          │
│    └── LoRA 多租户加载                                    │
│                                                          │
│  [推理层]                                                 │
│    ├── Pre-fill(预填充,处理 Prompt)                    │
│    ├── Decode(解码,生成 Token)                         │
│    ├── 批处理(Continuous Batching)                      │
│    ├── 流式输出(SSE)                                    │
│    └── 解码算法(Greedy/Beam/Sampling/Top-K/Top-P)     │
│                                                          │
│  [服务层]                                                 │
│    ├── OpenAI 兼容 API                                    │
│    ├── 鉴权(API Key/Bearer)                            │
│    ├── 限流(QPS/Token/并发)                            │
│    ├── 负载均衡                                          │
│    └── 健康检查                                          │
│                                                          │
│  [优化层]                                                 │
│    ├── KV Cache 管理                                     │
│    ├── Prefix Cache                                      │
│    ├── Speculative Decoding                              │
│    ├── Chunked Prefill                                   │
│    └── Flash Attention                                   │
│                                                          │
│  [监控层]                                                 │
│    ├── Prometheus Metrics                                │
│    ├── Token 计数                                        │
│    ├── 延迟指标(TTFT/TPOT)                              │
│    └── GPU 利用率                                        │
└──────────────────────────────────────────────────────────┘
```

### 1.4 推理服务的两大阶段

```
用户输入 "What is AI?"
    ↓
┌──────────────────────────────────────────────┐
│  [阶段 1] Prefill(预填充)                  │
│  - 一次性并行处理所有输入 Token              │
│  - 计算 KV Cache                            │
│  - 计算量大,瓶颈是 GPU 计算                │
│  - 输出:第一个 Token(对应 TTFT 指标)      │
└──────────────────────────────────────────────┘
    ↓
┌──────────────────────────────────────────────┐
│  [阶段 2] Decode(解码)                     │
│  - 自回归逐 Token 生成                      │
│  - 复用 KV Cache,只算新 Token              │
│  - 计算量小,瓶颈是显存带宽                │
│  - 输出:剩余 Tokens(对应 TPOT 指标)       │
└──────────────────────────────────────────────┘
    ↓
完整输出: "Artificial Intelligence is..."
```

**关键洞察:**
- **Prefill 是 Compute-bound(计算密集):** GPU 利用率高
- **Decode 是 Memory-bound(显存带宽密集):** GPU 利用率低
- **Chunked Prefill 优化:** 把长 Prompt 切成小块与 Decode 一起调度,提升 GPU 利用率

## 三、vLLM 深度

### 3.1 基础定义

**全称:** Very Large Language Model(原意)
**定义:** 一个高吞吐、低延迟的开源 LLM 推理引擎,核心创新是 PagedAttention,工业界部署最广泛。

**官网:** https://docs.vllm.ai/
**GitHub:** https://github.com/vllm-project/vllm(40k+ star 且持续增长)
**论文:** "Efficient Memory Management for Large Language Model Serving with PagedAttention"(Kwon et al., 2023-09)
**论文链接:** https://arxiv.org/abs/2309.06180(SOSP 2023)

### 3.2 PagedAttention 深度

#### 3.2.1 传统 KV Cache 的问题

```
传统 KV Cache 分配:
┌─────────────────────────────────────────────┐
│ 请求 A(短): [█████ _____________ ____________]│
│             已用   内部碎片    外部碎片       │
│                                              │
│ 请求 B(长): [███████████████████ ____________]│
│             已用                外部碎片     │
│                                              │
│ 请求 C(中): [███████████ ____________________]│
│             已用   内部碎片                  │
└─────────────────────────────────────────────┘
问题: 
- 内部碎片:预留空间未用满(最大可达 80% 浪费)
- 外部碎片:请求结束后留下碎片
- 显存利用率只有 20-40%
```

#### 3.2.2 PagedAttention 的解决方案

```
借鉴 OS 虚拟内存:
┌──────────────────────────────────────────────┐
│  PagedAttention 内存管理                      │
│                                              │
│  [逻辑视图 - 每个请求看到连续的 KV]            │
│  请求 A: [Block 7][Block 12][Block 23]      │
│  请求 B: [Block 3][Block 8][Block 15][Block 19]│
│                                              │
│  [物理视图 - 实际显存分块存储]                 │
│  Block 0  Block 1  Block 2  Block 3          │
│  Block 4  Block 5  Block 6  Block 7          │
│  Block 8  Block 9  Block 10 Block 11         │
│  Block 12 Block 13 Block 14 Block 15         │
│  ...                                         │
│                                              │
│  [Block Table(页表)]                         │
│  请求 A → [7, 12, 23]                        │
│  请求 B → [3, 8, 15, 19]                     │
└──────────────────────────────────────────────┘
```

**核心思想:**
- KV Cache 分成固定大小的 **Block(块)**,通常 16 个 Token 为一块
- 每个 Block 可放在显存任意位置
- **Block Table** 记录逻辑 Block 到物理 Block 的映射
- 按需分配,用完即回收
- **共享:** 多个请求共享相同前缀的 Block(Copy-on-Write)

**效果:**
- 显存利用率从 20-40% 提升到 ~96%
- 支持更多并发请求
- 吞吐提升 2-4 倍
- 支持长上下文请求

#### 3.2.3 PagedAttention 工作流程

```
1. 请求到达,初始化 Block Table
   请求 A "What is AI?"
   ↓
2. Prefill 阶段,计算 KV Cache
   - 分配 Block 7
   - 写入 Token 1-4 的 K/V
   ↓
3. Decode 阶段,生成新 Token
   - Block 7 还有空间,继续写入
   - 满 16 Token,分配 Block 12
   ↓
4. 请求完成,释放 Block 7 和 12
   - Block 回收到空闲池
   ↓
5. 共享前缀场景
   - 请求 A 和请求 B 共享 System Prompt
   - 都指向 Block 7(共享)
   - 不同部分用 Copy-on-Write
```

### 3.3 Continuous Batching(连续批处理)

#### 3.3.1 传统 Static Batching 的问题

```
传统批处理:
请求 A: [生成 Token 1][生成 Token 2][结束].....等待其他请求
请求 B: [生成 Token 1][生成 Token 2][生成 Token 3][生成 Token 4][结束]
请求 C: [生成 Token 1][结束]...................等待

问题: 短请求要等长请求完成,浪费 GPU 时间
```

#### 3.3.2 Continuous Batching 的解决

```
Continuous Batching(迭代级调度):
迭代 1: 请求 A | 请求 B | 请求 C | 请求 D
迭代 2: 请求 A | 请求 B | 请求 C | 请求 D
迭代 3: 请求 A(完成)| 请求 B | 请求 C | 请求 D | 请求 E(新加入)
迭代 4: ────── | 请求 B | 请求 C(完成)| 请求 D | 请求 E | 请求 F(新)
迭代 5: ────── | 请求 B | ────── | 请求 D | 请求 E | 请求 F | 请求 G(新)

特点:
- 每次迭代(生成一个 Token)后检查
- 完成的请求立即移除
- 新请求随时加入
- GPU 始终满载
- 吞吐提升 5-10 倍
```

#### 3.4.1 vLLM V1 引擎(2025 起默认,必知)

V1 是 vLLM 2025 年 1 月发布的新引擎架构(alpha),随后成为**默认引擎**,V0 已移除。
核心差异:

| 维度 | V0(旧) | V1(2025 起默认) |
|------|--------|------------------|
| 调度器 | prefill/decode 分离调度,功能间兼容性差 | 统一调度器,每步按 `{请求: token 数}` 分配 |
| Chunked Prefill | 可选、与部分功能互斥 | **默认开启且不可关闭** |
| Prefix Caching | 需 `--enable-prefix-caching` 手动开 | **默认开启**(无需加参数) |
| Speculative Decoding | 实验性 | 原生支持(EAGLE/Medusa/n-gram) |
| CPU 开销 | 高(Python 调度瓶颈) | 低(多进程架构,CPU/GPU 解耦) |
| 多模态 | 支持有限 | 架构级支持 |
| 开关方式 | 默认 | 旧版本用 `VLLM_USE_V1=1` 显式开;新版本默认即 V1 |

**对老教程的影响(照抄报错点):**
- `--enable-chunked-prefill` / `--no-enable-chunked-prefill`:V1 中该开关无意义(恒开)
- `--max-num-batched-tokens`:V1 语义变为"每步调度预算",默认值已调大
- 遇到参数报错先查对应版本的 `vllm serve --help`,vLLM 迭代快、参数变动频繁

### 3.5 vLLM 完整部署示例

#### 3.5.0 环境与下载前置(国内开发者必看)

**环境要求:** Linux(推荐 Ubuntu 22.04)+ NVIDIA GPU + CUDA 12.x;

```bash

    --tensor-parallel-size 8 \
    --gpu-memory-utilization 0.9 \
    --max-model-len 32768 \

# 多机部署(2 台 8 GPU 机器,TP=8 × PP=2)
# vLLM 没有 --ray-workers / --node-rank / --master-address 参数!
# 正确做法:先用 Ray 把两台机器组成集群,再只在 head 节点上启动 vllm serve,
# vLLM 会通过 Ray 自动发现并使用全部 16 张卡。

# 机器 1(head 节点):先启动 Ray head

# 机器 2(worker 节点):加入集群(<主节点IP> 换成机器 1 的内网 IP)
ray start --address='<主节点IP>:6379'

# 验证集群(在机器 1 上执行,应看到 2 个节点、16 张 GPU)
ray status

# 只在机器 1(head)上启动 vLLM,TP=8 跨单机、PP=2 跨两台机器
vllm serve meta-llama/Llama-3-70B-Instruct \
    --tensor-parallel-size 8 \
    --pipeline-parallel-size 2 \
    --distributed-executor-backend ray \
    --gpu-memory-utilization 0.9

# 要点:
# - 两台机器的 vLLM / CUDA / 模型路径必须一致(建议用同一 Docker 镜像)
# - 节点间需要高速网络(建议 InfiniBand / RoCE),PP 跨机通信量小,优先跨机用 PP
# - K8s 环境可用 vLLM 官方 Helm Chart 或 llm-d 自动管理 Ray 集群
```

```python
from openai import OpenAI, AsyncOpenAI
import asyncio

# 同步调用
client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="dummy"
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3-8B-Instruct",
    messages=[{"role": "user", "content": "What is AI?"}],
    max_tokens=512,
    temperature=0.7,
    stream=True
)

# 流式输出
for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)

# 异步批量调用
async def batch_inference():
    async_client = AsyncOpenAI(
        base_url="http://localhost:8000/v1",
        api_key="dummy"
    )
    tasks = [
        async_client.chat.completions.create(
            model="meta-llama/Llama-3-8B-Instruct",
            messages=[{"role": "user", "content": f"Question {i}"}]
        ) for i in range(100)
    ]
    results = await asyncio.gather(*tasks)
    return results
```

## 六、TensorRT-LLM 深度

### 6.1 基础定义

**定义:** NVIDIA 推出的 LLM 推理优化引擎,基于 TensorRT,针对 NVIDIA GPU 做极致优化,性能最高但学习曲线陡峭。

**GitHub:** https://github.com/NVIDIA/TensorRT-LLM
**文档:** https://nvidia.github.io/TensorRT-LLM/

### 6.2 核心特性

| 特性 | 说明 |
|------|------|
| **In-flight Batching** | 比连续批处理更进一步的调度 |
| **INT4/INT8/FP8** | 全精度量化支持 |
| **Plugin** | 自定义算子插件 |
| **Tensor Parallelism** | 多 GPU 张量并行 |
| **Pipeline Parallelism** | 流水线并行 |
| **KV Cache 量化** | INT8/FP8 KV Cache |
| **CUDA Graph** | 减少内核启动开销 |
| **Graph Rewriting** | 算子融合优化 |

### 6.3 TensorRT-LLM 部署流程

**方式一(推荐,2025 主流):`trtllm-serve` 一条命令起服务**

新版 TRT-LLM(≥ 0.17 / 1.x)默认走 PyTorch 后端,**无需手动构建引擎**,
直接给 HF 模型 ID 或本地路径即可启动 OpenAI 兼容服务:

```bash
# 1. 安装(需要 NVIDIA GPU + 匹配的 CUDA 驱动)
pip install tensorrt-llm

# 2. 启动 OpenAI 兼容服务(首次运行会自动编译/优化,需要等待)
trtllm-serve serve "meta-llama/Llama-3.1-8B-Instruct" \
    --host 0.0.0.0 \
    --port 8000 \
    --tp_size 1

# 3. 调用(与 vLLM 相同的 OpenAI 接口)
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "meta-llama/Llama-3.1-8B-Instruct",
       "messages": [{"role": "user", "content": "你好"}]}'

# 多卡: --tp_size 8;更多旋钮见 trtllm-serve serve --help
# 也支持 NVIDIA 官方 FP4 镜像,如 nvidia/Llama-3.1-8B-Instruct-FP8
```

**方式二(传统 TRT 后端):`trtllm-build` 离线构建引擎**

适合需要深度定制(C++ 部署、Triton 集成)的场景;流程长、版本敏感,
完整参数以官方文档为准(https://nvidia.github.io/TensorRT-LLM/):

```bash
# 转换 checkpoint → 构建引擎 → 用 Triton Inference Server 部署
python convert_checkpoint.py --model_dir /path/to/llama-3.1-8b \
    --output_dir /tmp/llama-3.1-8b-trt --dtype float16
trtllm-build --checkpoint_dir /tmp/llama-3.1-8b-trt \
    --output_dir /tmp/llama-3.1-8b-engine \
    --max_batch_size 32 --max_input_len 2048 --max_output_len 512
```

> ⚠️ 老教程里的 `--gemm_plugin` / `--use_gemm_plugin` / `--use_paged_context_kv_cache`
> 等参数在新版本已改名或移除,照抄必报错——以你安装版本的
> `trtllm-build --help` 输出为准。新手强烈建议从方式一入手。

### 6.4 vLLM vs TensorRT-LLM 性能对比

| 指标 | vLLM | TensorRT-LLM | 说明 |
|------|------|--------------|------|
| **吞吐量** | 高 | 通常更高(NVIDIA 内核级优化) | 优势在 FP8/FP4、新硬件上放大 |
| **首 Token 延迟(TTFT)** | 低 | 低 | 两者同档 |
| **显存效率** | 高(PagedAttention) | 高(Paged KV Cache) | 相当 |
| **部署难度** | 低(pip 即用) | 中(trtllm-serve 已大幅简化) | 引擎构建流仍陡峭 |
| **硬件绑定** | NVIDIA/AMD/CPU 均可 | 仅 NVIDIA | 异构环境选 vLLM |

> 具体数字随版本、硬件、batch 配置差异极大,**以上仅为定性对比**;
> 权威数据以 vLLM blog 与 NVIDIA TensorRT-LLM 官方 benchmark 为准,
> 选型时建议用自有流量模型实测(vLLM 自带 `vllm bench serve`)。

## 七、SGLang 深度

### 7.1 基础定义

**全称:** SGLang(Structured Generation Language)
**定义:** LMSYS 推出的 LLM 推理框架,核心创新是 RadixAttention(基数树前缀缓存)和结构化生成。

**GitHub:** https://github.com/sgl-project/sglang
**论文:** "SGLang: Efficient Execution of Structured Language Model Programs"(Zheng et al., 2024)

### 7.2 核心创新:RadixAttention

```
传统前缀缓存:线性查找,效率低
RadixAttention:基数树组织,自动共享前缀

Radix Tree 示例:
                    [System Prompt]
                   /              \
            [User A]            [User B]
            /     \                |
      [Context1][Context2]    [Context3]
            |
        [Response]

效果:
- 自动识别共享前缀
- 共享 KV Cache
- 多轮对话加速明显
- 少样本(Few-shot)加速
```

### 7.3 SGLang 部署

```bash
# 或 OpenAI 兼容
python -m sglang.launch_server \
    --model-path meta-llama/Llama-3-8B-Instruct \
    --port 30000 \
    --tp 2 \
    --enable-cache-report

# 结构化生成(JSON)
curl http://localhost:30000/generate \

            }
        }
    }'
```

## 八、llama.cpp 深度

### 8.1 基础定义

**定义:** C/C++ 实现的轻量 LLM 推理库,可在 CPU/消费级 GPU 上运行大模型,核心是 GGUF 量化格式。

**GitHub:** https://github.com/ggerganov/llama.cpp(70k+ star)

### 8.2 GGUF 量化格式

| 量化等级 | 位宽 | 大小(7B) | 质量 | 适用 |
|----------|------|----------|------|------|
| **Q8_0** | 8-bit | 7GB | 接近原版 | 高质量 |
| **Q6_K** | 6-bit | 5.5GB | 几乎无损 | 推荐 |
| **Q5_K_M** | 5-bit | 4.8GB | 损失很小 | 平衡 |
| **Q4_K_M** | 4-bit | 4.1GB | 损失小 | **推荐** |
| **Q4_0** | 4-bit | 3.8GB | 损失明显 | 极致压缩 |
| **Q3_K_M** | 3-bit | 3.3GB | 损失明显 | 资源紧张 |
| **Q2_K** | 2-bit | 2.7GB | 损失大 | 测试 |

### 8.3 llama.cpp 使用

```bash

git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp

# CPU 编译
cmake -B build
cmake --build build --config Release -j

# CUDA 编译(需已装 CUDA Toolkit)
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j

# 下载模型(HuggingFace GGUF 版本)
huggingface-cli download TheBloke/Llama-2-7B-Chat-GGUF \
    llama-2-7b-chat.Q4_K_M.gguf \
    --local-dir ./models

# 命令行对话(二进制在 build/bin/ 目录)
./build/bin/llama-cli \
    -m ./models/llama-2-7b-chat.Q4_K_M.gguf \
    -p "What is AI?" \
    -n 512 \
    -t 8

# 启动服务(OpenAI 兼容)
./build/bin/llama-server \
    -m ./models/llama-2-7b-chat.Q4_K_M.gguf \
    --host 0.0.0.0 \
    --port 8080 \
    -c 4096 \
    -t 8 \
    -ngl 32  # GPU 层数
```

## 九、分布式推理

### 9.1 分布式推理的必要性

```
模型规模 vs 单 GPU 显存:
- 7B 模型 FP16:  14GB  → 单卡 A10/T4 可跑
- 13B 模型 FP16: 26GB  → 单卡 A100-40G 可跑
- 70B 模型 FP16: 140GB → 至少 2 张 A100-80G
- 175B 模型 FP16: 350GB → 至少 5 张 A100-80G
- 671B 模型 FP16: 1.3TB → 至少 16 张 H200
```

### 9.2 四种并行策略

#### 9.2.1 数据并行(Data Parallelism, DP)

```
[GPU 0] 完整模型 + 请求 A
[GPU 1] 完整模型 + 请求 B
[GPU 2] 完整模型 + 请求 C
[GPU 3] 完整模型 + 请求 D

特点:
- 每张 GPU 完整副本
- 处理不同请求
- 提升吞吐,不提升单请求速度
- 适合:模型小,请求量大
```

#### 9.2.2 张量并行(Tensor Parallelism, TP)

```
单层 Attention 计算:
[GPU 0] 算 Q/K/V 的一半
[GPU 1] 算 Q/K/V 的另一半
        All-Reduce 合并结果

特点:
- 层内切分
- 单请求加速
- 通信开销大(All-Reduce)
- 适合:单机多 GPU(NVLink)
- 推荐:TP ≤ 8(单机内)
```

#### 9.2.3 流水线并行(Pipeline Parallelism, PP)

```
[GPU 0] Layer 1-10
   ↓
[GPU 1] Layer 11-20
   ↓
[GPU 2] Layer 21-30
   ↓
[GPU 3] Layer 31-40

特点:
- 按层切分
- 单请求加速
- 通信开销小(只传激活)
- 适合:多机部署
- 问题:Pipeline Bubble(气泡)
```

#### 9.2.4 专家并行(Expert Parallelism, EP)

```
MoE 模型,8 个专家:
[GPU 0] Expert 1, 2
[GPU 1] Expert 3, 4
[GPU 2] Expert 5, 6
[GPU 3] Expert 7, 8

特点:
- 仅适用于 MoE 模型
- 不同专家放不同 GPU
- 路由器决定 Token 去哪个 GPU
- 配合 TP/PP 使用
- DeepSeek-V3 用 EP + TP
```

#### 9.2.5 上下文并行(Context Parallelism, CP)

```
长上下文(1M Token):
[GPU 0] 处理 Token 1-250K 的 Attention
[GPU 1] 处理 Token 250K-500K 的 Attention
[GPU 2] 处理 Token 500K-750K 的 Attention
[GPU 3] 处理 Token 750K-1M 的 Attention
        Ring Attention 合并

特点:
- 解决超长上下文显存问题
- 沿 Sequence 维度切分
- Ring Attention / DeepSpeed Ulysses
- 代表:Llama 3 1M 上下文训练
```

### 9.3 并行策略对比

| 并行 | 切分维度 | 通信 | 适用 | 代表 |
|------|---------|------|------|------|
| **DP** | 数据(请求) | 小(梯度) | 多请求吞吐 | 大规模服务 |
| **TP** | 张量(层内) | 大(All-Reduce) | 单机多卡 | vLLM 默认 |
| **PP** | 层(垂直) | 小(激活) | 多机 | Megatron-LM |
| **EP** | 专家 | 中(路由) | MoE | DeepSeek |
| **CP** | 序列 | 中(Ring) | 超长上下文 | Llama 3 |

### 9.4 组合并行策略

```
典型组合:TP + PP + DP
- TP=8(单机 8 GPU,层内切分)
- PP=2(2 台机器,层间切分)
- DP=4(4 个副本,数据并行)
- 总 GPU: 8 × 2 × 4 = 64

3D 并行:
[TP=8 单机 NVLink] → [PP=2 跨机 InfiniBand] → [DP=4 多副本]
```

### 9.5 Prefill/Decode 分离(P/D 分离,2025 生产部署最重要演进)

**为什么分离:** Prefill(算力瓶颈,吃 FLOPs)和 Decode(带宽瓶颈,吃显存带宽)
对硬件的需求完全不同,混跑在同一 GPU 上会互相干扰——长 prompt 的 prefill
会拖住所有 decode 请求的 TPOT。分开部署后可以:

- **独立扩缩:** prefill 实例和 decode 实例按各自负载单独扩容
- **独立选型:** prefill 用算力强的卡,decode 用带宽大的卡(甚至不同代际混部)
- **SLO 达标率提升:** DistServe(OSDI'24)实测在同等资源下可支撑数倍的请求速率

```
[客户端] → [Router/网关]
              ├── prefill 集群(算力型,算完 KV 就传走)
              │        │  KV Cache 传输(NIXL / Mooncake / LMCache)
              │        ▼
              └── decode 集群(带宽型,接手 KV 后逐 Token 生成)
```

**生态一句话定位:**

| 项目 | 定位 |
|------|------|
| **llm-d** | Kubernetes 原生的 P/D 分离编排框架(Red Hat/Google 等发起),vLLM 上游生产化路径 |
| **NVIDIA Dynamo** | NVIDIA 的分布式推理框架,内置 P/D 分离、KV 感知路由、NIXL KV 传输 |
| **LMCache** | KV Cache 存储/复用层,跨实例共享 KV(也是 P/D 分离的 KV 传输选项之一) |
| **NIXL** | NVIDIA 的点对点 KV 传输库(RDMA/NVLink),vLLM `--kv-transfer-config` 可用 |
| **Mooncake** | 月之暗面开源的 KVCache-centric 架构,vLLM 已集成其传输引擎 |
| **DistServe** | P/D 分离的开山论文(OSDI'24,https://arxiv.org/abs/2401.09670) |

**vLLM 侧最小入口(实验性):**
```bash
# prefill 实例与 decode 实例分别启动,通过 --kv-transfer-config 指定 KV 传输通道
vllm serve <model> --kv-transfer-config \
    '{"kv_connector":"NixlConnector","kv_role":"kv_producer"}'   # prefill 侧
vllm serve <model> --kv-transfer-config \
    '{"kv_connector":"NixlConnector","kv_role":"kv_consumer"}'   # decode 侧
```
> 生产环境建议直接用 llm-d / Dynamo 编排,手写双实例只用于验证概念。

## 十三、负载均衡

### 13.1 基础定义

**负载均衡(Load Balancer):** 在多个后端实例之间分配请求,提高可用性和吞吐量。

### 13.2 负载均衡算法

| 算法 | 原理 | 特点 | 适用场景 |
|------|------|------|---------|
| **轮询(Round Robin)** | 依次分配 | 简单公平 | 同质化后端 |
| **加权轮询** | 按权重分配 | 考虑性能差异 | 异构后端 |
| **最少连接** | 分配给连接最少的 | 动态适应 | 长连接 |
| **IP Hash** | 按 IP 哈希 | 同 IP 同后端 | 会话保持 |
| **URL Hash** | 按请求 URL 哈希 | 同 URL 同后端 | 缓存友好 |
| **最快响应** | 分配给响应最快的 | 性能最优 | 对延迟敏感 |
| **随机** | 随机分配 | 简单 | 简单场景 |

### 13.3 AI 场景特殊负载均衡策略

```
传统策略不适用,需要 AI 专用策略:

1. [KV Cache 亲和性]
   - 同一会话的请求路由到同一实例
   - 复用 KV Cache,降低延迟
   - 实现:基于 Session ID Hash

2. [Token 感知]
   - 考虑请求 Token 数,而非简单请求数
   - 长 Token 请求分配给空闲实例
   - 实现:维护每个实例的 Token 负载

3. [模型感知]
   - 不同实例加载不同模型
   - 根据请求的模型路由
   - 实现:模型-实例映射表

4. [排队感知]
   - 考虑实例的队列长度
   - 优先分配给队列短的实例
   - 实现:实时上报队列长度

5. [成本感知]
   - 不同渠道不同成本
   - 优先低成本渠道
   - 实现:渠道成本表
```

### 13.4 Nginx 负载均衡示例

```nginx
# vLLM 多实例负载均衡(开源版 Nginx 可用)
upstream vllm_backend {
    # 最少连接策略
    least_conn;
    
    # max_fails + fail_timeout 是【开源版】的被动健康检查:
    # 30s 内失败 3 次 → 临时摘除该节点 30s,之后自动恢复
    server vllm-1:8000 weight=1 max_fails=3 fail_timeout=30s;
    server vllm-2:8000 weight=1 max_fails=3 fail_timeout=30s;
    server vllm-3:8000 weight=2 max_fails=3 fail_timeout=30s backup;
    
    # 长连接支持
    keepalive 32;
}

server {
    listen 80;
    
    # 流式响应支持
    proxy_buffering off;
    proxy_cache off;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    
    location /v1/ {
        proxy_pass http://vllm_backend;
        proxy_read_timeout 600s;
        proxy_send_timeout 600s;
    }
}
```

> ⚠️ 注意:`health_check interval=10s fails=3 passes=2;` 是 **NGINX Plus(商业版)**
> 的主动健康检查指令,开源版写进去会直接启动失败。
> 开源版方案:
> 1. **被动健康检查**(上面示例的 `max_fails`/`fail_timeout`,零成本够用);
> 2. 需要主动探测时,换 **HAProxy**(`check` 指令原生支持)或 **Envoy**(主动健康检查),
>    或给开源 Nginx 打 `nginx_upstream_check_module` 补丁(维护成本高,不推荐);
> 3. K8s 环境直接用 Service + readinessProbe,不依赖 LB 层健康检查。

## 十四、限流

### 14.1 基础定义

**限流(Rate Limit):** 限制请求的速率,保护后端服务不被过载,保证服务质量。

### 14.2 限流算法详解

#### 14.2.1 固定窗口

```
窗口 1 (0-1s):  A A A A A A (6 个,超出 5 限制,第 6 个拒绝)
窗口 2 (1-2s):  B B B B B (5 个)
窗口 3 (2-3s):  C C C C C C C (7 个,超出,拒绝后 2 个)

问题: 边界突发 - 0.9s 5 个 + 1.1s 5 个 = 1 秒内 10 个请求
```

#### 14.2.2 滑动窗口

```
当前时间 1.5s,窗口 [0.5, 1.5]
统计这 1 秒内的请求数,与阈值比较

特点:
- 比固定窗口平滑
- 实现稍复杂(需记录时间戳)
```

#### 14.2.3 令牌桶(Token Bucket)

```
        ┌─────────────┐
        │  令牌桶      │
        │  ┌─┐┌─┐┌─┐  │  ← 以固定速率 r 补充令牌
        │  └─┘└─┘└─┘  │
        │  ┌─┐┌─┐┌─┐  │  ← 桶容量 C(允许突发)
        │  └─┘└─┘└─┘  │
        └──────┬──────┘
               ↓
        请求来时取一个令牌
        - 有令牌:放行
        - 无令牌:拒绝/等待

特点:
- 允许突发(C 个请求瞬间打来)
- 平均速率 r
- 最常用
```

#### 14.2.4 漏桶(Leaky Bucket)

```
请求流入 ────→ ┌────────────┐
               │   漏 桶    │ ────→ 以固定速率 r 流出
               │  ████████  │
               │  ████████  │
               └────────────┘
                  满则溢出(拒绝)

特点:
- 输出速率严格固定
- 不允许突发
- 适合需要平滑流量的场景
```

### 14.3 Python 令牌桶实现

```python
import time
import threading
from collections import defaultdict

class TokenBucket:
    def __init__(self, capacity: int, refill_rate: float):
        """
        capacity: 桶容量(允许突发)
        refill_rate: 每秒补充令牌数
        """
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = capacity
        self.last_refill = time.time()
        self.lock = threading.Lock()
    
    def consume(self, tokens: int = 1) -> bool:
        with self.lock:
            now = time.time()
            # 补充令牌
            elapsed = now - self.last_refill
            self.tokens = min(
                self.capacity,
                self.tokens + elapsed * self.refill_rate
            )
            self.last_refill = now
            
            if self.tokens >= tokens:
                self.tokens -= tokens
                return True
            return False

# 用户级限流
class UserRateLimiter:
    def __init__(self, capacity: int, refill_rate: float):
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.buckets = defaultdict(lambda: TokenBucket(capacity, refill_rate))
    
    def check(self, user_id: str) -> bool:
        return self.buckets[user_id].consume()

# 使用
limiter = UserRateLimiter(capacity=10, refill_rate=1)  # 10 突发,1/秒
for _ in range(15):
    allowed = limiter.check("user_001")
    print(f"Allowed: {allowed}")
```

### 15.1 基础定义

**鉴权(Authentication & Authorization):**
- **Authentication(认证):** 确认"你是谁"
- **Authorization(授权):** 确认"你能做什么"

### 15.2 鉴权方式对比

| 方式 | 说明 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **API Key** | 字符串密钥 | 简单 | 安全性低 | 服务端调用 |
| **Bearer Token** | Authorization: Bearer xxx | 通用 | 需配合 HTTPS | REST API |
| **JWT** | 自包含 Token | 无状态 | 撤销难 | Web 应用 |
| **OAuth 2.1** | 授权框架 | 灵活 | 复杂 | 第三方授权 |
| **mTLS** | 双向 TLS | 极安全 | 证书管理 | 内部服务 |
| **HMAC** | 签名 | 防篡改 | 实现复杂 | API 签名 |

### 15.3 OAuth 2.1 详解(MCP 推荐)

#### 15.3.1 OAuth 2.1 基础

**OAuth 2.1:** OAuth 2.0 的整合升级版,强制 PKCE,简化流程,2024+ 推荐使用。

**核心概念:**
| 概念 | 说明 |
|------|------|
| **Resource Owner** | 资源所有者(用户) |
| **Client** | 客户端应用 |
| **Authorization Server** | 授权服务器 |
| **Resource Server** | 资源服务器(API) |
| **Authorization Code** | 授权码 |
| **Access Token** | 访问令牌 |
| **Refresh Token** | 刷新令牌 |
| **PKCE** | Proof Key for Code Exchange,防授权码拦截 |

#### 15.3.2 OAuth 2.1 + PKCE 流程

```
1. 客户端生成 code_verifier 和 code_challenge
   code_verifier = 随机字符串
   code_challenge = SHA256(code_verifier) (S256)

2. 客户端 → 授权服务器
   GET /authorize?
       response_type=code&
       client_id=xxx&
       redirect_uri=xxx&
       scope=xxx&
       code_challenge=xxx&
       code_challenge_method=S256

3. 用户登录授权

4. 授权服务器 → 客户端(回调)
   GET /callback?code=AUTH_CODE

5. 客户端 → 授权服务器(换 Token)
   POST /token
   {
       grant_type: "authorization_code",
       code: AUTH_CODE,
       redirect_uri: xxx,
       client_id: xxx,
       code_verifier: code_verifier
   }

6. 授权服务器验证 code_verifier
   - 计算 SHA256(code_verifier)
   - 与 step 2 的 code_challenge 比对
   - 验证通过 → 返回 Access Token

7. 客户端 → 资源服务器
   GET /api/resource
   Authorization: Bearer ACCESS_TOKEN
```

#### 15.4.2 JWT Python 示例

```python
import jwt
import time

SECRET = "your-secret-key"

# 生成 JWT
def create_token(user_id: str, role: str) -> str:
    payload = {
        "sub": user_id,
        "role": role,
        "iat": int(time.time()),
        "exp": int(time.time()) + 3600  # 1 小时过期
    }
    return jwt.encode(payload, SECRET, algorithm="HS256")

# 验证 JWT
def verify_token(token: str) -> dict:
    try:
        return jwt.decode(token, SECRET, algorithms=["HS256"])
    except jwt.ExpiredSignatureError:
        raise Exception("Token expired")
    except jwt.InvalidTokenError:
        raise Exception("Invalid token")

# 使用
token = create_token("user_123", "admin")
payload = verify_token(token)
print(payload)  # {'sub': 'user_123', 'role': 'admin', ...}
```

```python
        payload = jwt.decode(token, SECRET, algorithms=["HS256"])
        return payload
    except jwt.PyJWTError:
        raise HTTPException(status_code=401, detail="Invalid token")
```

### 17.2 KV Cache 详解

**计算公式:**
```
KV Cache 显存 = 2 × num_layers × hidden_size × kv_heads / total_heads × seq_len × batch_size × dtype_bytes

(等价写法:2 × num_layers × kv_heads × head_dim × seq_len × batch_size × dtype_bytes,
 hidden_size × kv_heads/total_heads = kv_heads × head_dim;务必用 KV 头数,别用 Q 头数)

例:Llama-3-70B
- num_layers = 80
- hidden_size = 8192
- kv_heads = 8(GQA)
- total_heads = 64
- seq_len = 8192
- batch_size = 32
- dtype = FP16(2 bytes)

KV Cache = 2 × 80 × 8192 × 8/64 × 8192 × 32 × 2
         = 2 × 80 × 8192 × 0.125 × 8192 × 32 × 2
         = 86 GB(超过模型本身!)

单请求基线(与 09 章 10.2 同口径):
Llama-3-8B, seq_len=32K, batch=1, BF16:
  2 × 32 × 8 × 128 × 32768 × 1 × 2 ≈ 4.3 GB
由此线性外推:batch ×2 → 8.6 GB;seq_len ×2 → 8.6 GB
```

```python
# - 所以 Prompt Caching 适合"同一长前缀被反复使用"的场景,单次调用反而更贵
print(f"Input tokens: {response.usage.input_tokens}")
print(f"Cache creation: {response.usage.cache_creation_input_tokens}")  # 这部分按 1.25× 计费
print(f"Cache read: {response.usage.cache_read_input_tokens}")          # 这部分按 0.1× 计费
```

### 17.4 Semantic Cache 实现

```python
# pip install "redis>=5.0" numpy openai
# 依赖 Redis Stack(带 RediSearch 向量检索):
#   docker run -p 6379:6379 redis/redis-stack:latest
import hashlib
import redis
import numpy as np
from openai import OpenAI
from redis.commands.search.query import Query

r = redis.Redis()
client = OpenAI()

def get_embedding(text: str) -> np.ndarray:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return np.array(response.data[0].embedding, dtype=np.float32)

def semantic_cache_get(query: str, threshold: float = 0.95) -> str | None:
    """语义缓存查询"""
    query_emb = get_embedding(query)
    query_vec = query_emb.tobytes()
    
    # Redis Vector Search(需先建 cache_idx 向量索引,见下方注释)
    result = r.ft('cache_idx').search(
        Query(f'*=>[KNN 1 @embedding $query_vec AS score]')
            .return_fields('response', 'score')
            .dialect(2),
        query_params={'query_vec': query_vec}
    )
    
    if result.docs:
        score = 1 - float(result.docs[0].score)  # 余弦距离转相似度
        if score >= threshold:
            return result.docs[0].response
    return None

def semantic_cache_set(query: str, response: str):
    """写入语义缓存"""
    embedding = get_embedding(query)
    # 用 sha256 做稳定 key;内置 hash() 跨进程不稳定(Python 随机化),勿用
    key = hashlib.sha256(query.encode("utf-8")).hexdigest()[:16]
    r.hset(f"cache:{key}", mapping={
        'query': query,
        'response': response,
        'embedding': embedding.tobytes()
    })

# 首次使用需建索引(HNSW 向量索引,余弦距离):
# from redis.commands.search.field import TextField, VectorField
# from redis.commands.search.indexDefinition import IndexDefinition, IndexType
# r.ft('cache_idx').create_index(
#     [TextField('query'), TextField('response'),
#      VectorField('embedding', 'HNSW',
#                  {'TYPE': 'FLOAT32', 'DIM': 1536, 'DISTANCE_METRIC': 'COSINE'})],
#     definition=IndexDefinition(prefix=['cache:'], index_type=IndexType.HASH)
# )

# 使用
def cached_llm_call(query: str) -> str:
    # 1. 语义缓存查询
    cached = semantic_cache_get(query)
    if cached:
        return cached
    
    # 2. 调用 LLM
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": query}]
    )
    answer = response.choices[0].message.content
    
    # 3. 写入缓存
    semantic_cache_set(query, answer)
    
    return answer
```

### 19.5 常见故障 Runbook(症状 → 原因 → 处置)

**(1) CUDA OOM(启动或运行中 `torch.OutOfMemoryError`)**

- **症状:** vLLM 启动失败,或高并发时请求报错 OOM
- **原因:** KV Cache 池 + 权重超过显存;`max_model_len`/`max_num_seqs` 设太大
- **处置:**
  ```bash
  # 1. 降显存水位线(默认 0.9)
  --gpu-memory-utilization 0.85
  # 2. 砍并发与上下文
  --max-num-seqs 64 --max-model-len 8192
  # 3. KV Cache 量化(Hopper/Blackwell 可用)
  --kv-cache-dtype fp8
  # 4. 还不行 → 换更小模型/量化版,或 TP 加卡
  ```

**(2) 冷启动 502(网关已起,模型没加载完)**

- **症状:** 服务启动后前几分钟请求全 502/超时
- **原因:** 模型加载 + CUDA Graph 捕获需数分钟,负载均衡已把流量打进来了
- **处置:** K8s 配 readinessProbe 探 `/health`(vLLM 就绪才返回 200);
  Nginx 层靠 `max_fails` 被动摘除;启动命令加 `--load-format safetensors`、
  模型放本地 NVMe 减少加载时间;预热后再接入流量

**(3) KV Cache 打满,请求排队(`num_requests_waiting` 飙升)**

- **症状:** TTFT 变长,`vllm:gpu_cache_usage_perc` 接近 100%,waiting 队列增长
- **原因:** 并发请求 KV 总量超过 KV 池;长上下文请求占着不放
- **处置:** 调大 `--gpu-memory-utilization`;降 `--max-model-len`;
  开 prefix caching 复用公共前缀;扩容副本或上 P/D 分离(9.5);
  对超长请求单独路由到长上下文专用实例

**(4) gated repo 401/403(下载 Llama/Gemma 失败)**

- **症状:** `huggingface-cli download` 或 vLLM 启动报 401 Unauthorized
- **原因:** 模型需先在 HF 网页申请访问权限(Llama 需 Meta 审批),且未配置 token
- **处置:** 网页申请 → `huggingface-cli login` 粘贴 token;
  国内走镜像也要 token(gated 不因镜像免鉴权);
  或干脆改用 ModelScope 上的官方镜像(Qwen/DeepSeek/GLM 无此问题)

**(5) 显存碎片化(跑着跑着 OOM,重启就好)**

- **症状:** 长时间运行后逐渐 OOM,`nvidia-smi` 显示有空闲但分配失败
- **原因:** 频繁加载/卸载 LoRA、动态 shape 导致碎片;旧版 vLLM 的 V0 引擎更明显
- **处置:** 升级到 V1 引擎(统一块管理);LoRA 场景固定 `--max-loras`;
  定期滚动重启实例;`--enforce-eager` 排查是否 CUDA Graph 相关(仅调试用,生产勿开)

**(6) 吞吐上不去(GPU 利用率只有 30%)**

- **症状:** `nvidia-smi` 利用率低,QPS 低
- **原因:** 并发不够(continuous batching 吃不饱);CPU 瓶颈(旧版 V0);
  `--max-num-seqs` 太小;单请求串行调用
- **处置:** 客户端改异步并发;调大 `--max-num-seqs` 与 `--max-num-batched-tokens`;
  升级 V1 引擎;确认没有开 `--enforce-eager`

```python
def route_model(query: str, user_tier: str) -> str:
    """根据请求特征路由到不同模型"""
    
    # 1. 用户级别
    if user_tier == "free":
        return "llama-3-8b"  # 小模型,免费
    
    # 2. 查询复杂度
    complexity = estimate_complexity(query)
    if complexity < 0.3:
        return "llama-3-8b"  # 简单 → 小
    elif complexity < 0.7:
        return "llama-3-70b"  # 中等 → 中
    else:
        return "gpt-4o"  # 复杂 → 大
    
    # 3. 任务类型
    if "code" in query.lower():
        return "deepseek-coder"
    if "math" in query.lower():
        return "o3-mini"
    if "image" in query:
        return "gpt-4o"  # 多模态
    
    # 4. 延迟敏感
    if user_tier == "premium":
        return "claude-4-sonnet"  # 快+强

def estimate_complexity(query: str) -> float:
    """估算查询复杂度"""
    score = 0.0
    if len(query) > 200: score += 0.2
    if "?" in query: score += 0.1
    if any(w in query for w in ["分析", "推理", "推导"]): score += 0.3
    if any(w in query for w in ["简单", "是什么"]): score -= 0.2
    return max(0.0, min(1.0, score))
```

## 二十二、官方建议与最佳实践

### 22.1 vLLM 官方建议

> 来源:https://docs.vllm.ai/en/latest/

- **生产环境必开:** `--enable-prefix-caching`(前缀缓存)
- **长上下文必开:** Chunked Prefill(v0.5+ 默认开)
- **TP 配置:** 单机内 TP,跨机用 PP
- **量化选择:** FP8 > AWQ > GPTQ > INT4(质量从高到低)
- **显存调优:** `--gpu-memory-utilization 0.9` 平衡
- **批处理:** `--max-num-seqs 256`(生产环境)
- **监控:** 必须配 Prometheus + Grafana

### 22.2 Ollama 官方建议

> 来源:https://github.com/ollama/ollama

- **量化选择:** Q4_K_M(质量/大小最佳平衡)
- **并发:** `OLLAMA_NUM_PARALLEL=4`(默认 1)
- **保持时间:** `OLLAMA_KEEP_ALIVE=5m`(避免反复加载)
- **Flash Attention:** `OLLAMA_FLASH_ATTENTION=1`(如支持)
- **生产环境:** 不建议,适合开发/测试

### 22.3 Anthropic API 建议

> 来源:https://docs.anthropic.com/

- **Prompt Caching:** 长前缀反复使用必开:命中部分按 0.1× 计费(省 90%);注意首次写入按 1.25×(贵 25%),单用一次反而更贵
- **批量 API:** 非实时场景用 Batch API,降 50%
- **流式:** 长输出用 SSE,改善体验
- **Tool Use:** 结构化任务用 Tool Calling
- **模型选择:** Sonnet 平衡,Haiku 快速,Opus 复杂

### 22.4 OpenAI API 建议

> 来源:https://platform.openai.com/docs

- **模型选择:** GPT-4o-mini(便宜)/ GPT-4o(强)
- **结构化输出:** 用 `response_format: json_schema`
- **Function Calling:** 复杂任务拆解为工具调用
- **批量 API:** 异步任务,降 50%
- **流式:** 用 SSE,改善 UX
- **Embedding:** text-embedding-3-small(便宜)/ large(质量)
