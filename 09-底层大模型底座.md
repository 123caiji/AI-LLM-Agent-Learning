## 二、Transformer 架构(LLM 的基石)

### 2.1 基础定义

**定义:** 基于 Attention 机制的神经网络架构,2017 年 Google 在《Attention Is All You Need》论文中提出,是所有现代 LLM 的基础。

**论文:** Vaswani et al., "Attention Is All You Need", 2017
**论文链接:** https://arxiv.org/abs/1706.03762

### 2.2 Transformer 核心结构

```
┌────────────────────────────────────────────┐
│              Transformer 架构               │
│                                            │
│  输入 Embedding + 位置编码                  │
│         ↓                                  │
│  ┌─────────────────────────────────────┐   │
│  │  Multi-Head Self-Attention          │   │
│  │  (多头自注意力)                      │   │
│  └─────────────────────────────────────┘   │
│         + 残差连接 & LayerNorm              │
│         ↓                                  │
│  ┌─────────────────────────────────────┐   │
│  │  Feed-Forward Network (FFN)         │   │
│  │  (前馈网络)                         │   │
│  └─────────────────────────────────────┘   │
│         + 残差连接 & LayerNorm              │
│         ↓                                  │
│  (重复 N 层)                               │
│         ↓                                  │
│  输出层(Linear + Softmax)                  │
└────────────────────────────────────────────┘
```

### 2.3 Attention 机制详解

**Self-Attention 公式:**

```
Attention(Q, K, V) = softmax(Q·K^T / √d_k) · V

其中:
- Q (Query): 查询矩阵
- K (Key): 键矩阵
- V (Value): 值矩阵
- d_k: 键的维度(用于缩放)
```

**直觉理解:**
- 每个 Token 作为 Query,去和所有其他 Token 的 Key 做点积,得到注意力分数
- 用 softmax 归一化为概率分布
- 用这个分布对所有 Token 的 Value 加权求和,得到当前 Token 的新表示

### 2.4 Multi-Head Attention(MHA)

把 Q/K/V 分成多个头,每个头独立做 Attention,最后拼接:

```
Multi-Head(Q, K, V) = Concat(head_1, ..., head_h) · W_O

其中 head_i = Attention(Q·W_Q^i, K·W_K^i, V·W_V^i)
```

**优势:**
- 不同头关注不同方面(语法、语义、共指等)
- 表达能力更强
- 可并行计算

### 2.5 Attention 变体(优化)

| 变体 | 全称 | 特点 | 代表模型 |
|------|------|------|---------|
| **MHA** | Multi-Head Attention | 标准多头,Q/K/V 都有 h 个头 | GPT-3、Llama |
| **MQA** | Multi-Query Attention | Q 多头,K/V 共享 1 个 | PaLM、StarCoder |
| **GQA** | Grouped-Query Attention | Q 分组,每组共享 K/V | Llama 2 70B、Llama 3 |
| **MLA** | Multi-head Latent Attention | KV 联合低秩压缩 | DeepSeek-V2/V3 |

**对比:**
- **MHA:** 质量最高,KV Cache 大
- **MQA:** KV Cache 最小,质量略降
- **GQA:** 折中,KV Cache 小,质量接近 MHA
- **MLA:** DeepSeek 创新,极致压缩 KV Cache

### 2.6 位置编码(Position Encoding)

Transformer 本身是位置无关的,需要显式注入位置信息:

| 方法 | 全称 | 特点 | 代表模型 |
|------|------|------|---------|
| **Absolute PE** | 绝对位置编码 | 学习/正弦的位置向量 | GPT-2、BERT |
| **RoPE** | 旋转位置编码 | 通过旋转矩阵注入相对位置 | Llama、Qwen |
| **ALiBi** | 线性偏置注意力 | Attention 加距离偏置 | BLOOM、MPT |
| **NoPE** | 无位置编码 | 靠 Attention Mask 隐式建模 | 部分 2024 研究 |

**RoPE 优势:**
- 相对位置信息
- 长度外推性(可扩展到更长上下文)
- 计算高效

### 2.7 FFN(前馈网络)

```
FFN(x) = Dropout(Linear_2(GELU(Linear_1(x))))

或 SwiGLU 变体:
FFN(x) = (Swish(x·W_1) * (x·W_gate)) · W_2
```

**变体:**
| 变体 | 激活函数 | 代表模型 |
|------|---------|---------|
| 标准 FFN | ReLU/GELU | BERT、GPT-2 |
| SwiGLU | Swish + Gate | Llama、Qwen |
| GeGLU | GELU + Gate | T5 |

### 2.8 LayerNorm 与 RMSNorm

| 方法 | 公式 | 特点 | 代表模型 |
|------|------|------|---------|
| **LayerNorm** | (x - mean) / sqrt(var + eps) · γ + β | 减均值除标准差 | BERT、GPT-2 |
| **RMSNorm** | x / sqrt(mean(x²) + eps) · γ | 只除 RMS,不减均值 | Llama、Qwen |

**RMSNorm 优势:** 计算更快,效果接近。

### 2.9 Pre-Norm vs Post-Norm

```
Post-Norm: x = LayerNorm(x + Sublayer(x))   # 原版 Transformer
Pre-Norm:  x = x + Sublayer(LayerNorm(x))   # 现代主流
```

**Pre-Norm 优势:** 训练更稳定,梯度更好,几乎所有现代 LLM 用 Pre-Norm。

## 三、Tokenizer(分词器)深度

### 3.1 基础定义

**定义:** 将文本转换为 Token 序列的工具,是 LLM 的前端处理模块。

### 3.2 分词算法对比

| 算法 | 代表 | 原理 | 特点 |
|------|------|------|------|
| **BPE** | GPT、tiktoken | 合并高频字节对 | 最常用、确定性强 |
| **WordPiece** | BERT | 基于似然合并 | Google 系常用 |
| **Unigram** | Llama、T5 | 概率选最优子词集 | 多语言好 |
| **SentencePiece** | 多种 | 不依赖空格的分词 | 多语言、亚洲语言好 |

### 3.3 BPE(字节对编码)详解

**过程:**
1. 初始词汇表:所有单个字符(或字节)
2. 迭代:找到频率最高的相邻字符对,合并为一个新 Token
3. 重复:直到词汇表达到目标大小

**示例:**
```
语料: low, lower, newest, widest

第 1 轮: 找最高频对 (e, s) → 合并为 es
第 2 轮: 找最高频对 (es, t) → 合并为 est
第 3 轮: 找最高频对 (l, o) → 合并为 lo
第 4 轮: 找最高频对 (lo, w) → 合并为 low
...

最终词汇表包含: low, est, new, wide, ...
```

**优势:**
- 可以处理任何文本(包括 OOV 词)
- 词汇表大小可控
- 压缩效率高

### 3.4 SentencePiece

Google 开源的分词库,不依赖空格,对中文/日文等无空格语言友好:

```python
import sentencepiece as spm

# 训练
spm.SentencePieceTrainer.train(
    input='corpus.txt',
    model_prefix='m',
    vocab_size=32000,
    character_coverage=0.9995
)

# 使用
sp = spm.SentencePieceProcessor()
sp.load('m.model')
tokens = sp.encode("你好,世界")  # [1234, 5678, 9012]
text = sp.decode(tokens)  # "你好,世界"
```

### 3.5 tiktoken(OpenAI)

OpenAI 的 BPE 实现,Rust 编写,极快:

```python
import tiktoken

# GPT-4o 的编码器
enc = tiktoken.encoding_for_model("gpt-4o")
tokens = enc.encode("Hello, world!")  # [9906, 11, 1917, 0]
text = enc.decode(tokens)  # "Hello, world!"

# Token 数
n_tokens = len(enc.encode("Hello, world!"))  # 4
```

### 3.6 Token 与字符的大致比例

| 语言 | 比例 | 说明 |
|------|------|------|
| 英文 | ~4 字符 = 1 Token | 约 0.75 个单词 |
| 中文 | ~1.5~2 字 = 1 Token | 中文 Token 效率较低 |
| 代码 | ~3-4 字符 = 1 Token | 取决于语言 |
| Emoji | 1-3 Token | 视具体字符 |

**重要意义:**
- 中文用户 Token 消耗是英文的 2-3 倍
- 中文模型(如 Qwen、DeepSeek)对中文 Token 优化
- Token 效率影响 API 成本

### 3.7 Token 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **整词** | 完整的词 | `hello`、`world` |
| **子词** | 词的一部分 | `un`、`happ`、`iness` |
| **单字符** | 单个字符 | `a`、`!`、数字 |
| **特殊 Token** | 控制符 | `<|endoftext|>`、`<s>`、`<|im_start|>` |

### 3.8 特殊 Token 详解

| Token | 含义 | 用途 |
|-------|------|------|
| `<|endoftext|>` | 文本结束 | OpenAI 系 |
| `<s>` | 序列开始 | Llama、BERT |
| `</s>` | 序列结束 | Llama、BERT |
| `<|im_start|>` | 消息开始 | ChatGPT |
| `<|im_end|>` | 消息结束 | ChatGPT |
| `[INST]` | 指令开始 | Llama 2 Chat |
| `[/INST]` | 指令结束 | Llama 2 Chat |
| `<|user|>` | 用户消息 | Qwen |
| `<|assistant|>` | 助手消息 | Qwen |

## 四、VLM(视觉大模型)与多模态

### 4.1 VLM 基础定义

**全称:** Vision Language Model

**定义:** 同时理解文本和图像的多模态模型,可以输入图像并回答相关问题。

### 4.2 VLM 的能力

- **图像描述:** 用文字描述图片内容
- **视觉问答:** 根据图片回答问题
- **OCR:** 识别图片中的文字
- **图表理解:** 理解图表、流程图
- **代码生成:** 从截图生成代码
- **视频理解:** 视频内容理解
- **图像推理:** 视觉逻辑推理

### 4.3 VLM 架构

```
┌──────────────────────────────────────────┐
│              VLM 架构                     │
│                                          │
│  [图像] → [Vision Encoder] → 图像 Token   │
│              (CLIP / ViT)                │
│                    ↓                     │
│  [文本] → [Tokenizer] → 文本 Token ──→ 合并 │
│                                          │
│                    ↓                     │
│            [多模态 LLM]                   │
│            (融合处理)                     │
│                    ↓                     │
│              [文本输出]                   │
└──────────────────────────────────────────┘
```

### 4.4 主流 VLM 模型

| 模型 | 厂商 | 特点 | 开源 |
|------|------|------|------|
| **GPT-4o / GPT-4V** | OpenAI | 综合能力最强 | ❌ |
| **Claude 4 Opus/Sonnet** | Anthropic | 长文本+图像 | ❌ |
| **Gemini 2.5 Pro** | Google | 多模态原生 | ❌ |
| **LLaVA** | 开源社区 | 开源、可微调 | ✅ |
| **Qwen-VL** | 阿里 | 开源、中文好 | ✅ |
| **InternVL** | 上海 AI Lab | 开源、强 | ✅ |
| **Pixtral** | Mistral | 开源、MoE | ✅ |
| **CogVLM** | 智谱 | 开源、Agent 强 | ✅ |

### 4.5 MMM(多模态大模型)

**全称:** Multi-Modal Model

**定义:** 支持多种模态(文本、图像、音频、视频等)输入输出的模型。

**模态类型:**

| 模态 | 说明 | 示例 |
|------|------|------|
| **文本** | 自然语言 | 对话、写作 |
| **图像** | 视觉信息 | 图片理解、生成 |
| **音频** | 声音 | 语音识别、语音合成 |
| **视频** | 动态视觉 | 视频理解、生成 |
| **3D** | 三维空间 | 3D 模型生成 |

### 4.6 MMM 发展趋势

- **统一模型:** 一个模型处理所有模态(GPT-4o、Gemini)
- **任意模态到任意模态:** 文本→图像、音频→文本、图像→视频
- **更强的推理:** 跨模态推理能力
- **实时交互:** 低延迟的多模态对话(GPT-4o 实时语音)
- **端侧部署:** 小型多模态模型(Phi-3-Vision)

## 五、Embedding 与 Reranker

### 5.1 Embedding 模型

**定义:** 将文本(或其他数据)转换为高维向量表示的模型,向量的距离反映语义相似度。

**用途:**

| 用途 | 说明 |
|------|------|
| **语义搜索** | 根据向量相似度检索相关文档 |
| **RAG** | 知识库检索的核心 |
| **聚类** | 对文本进行语义聚类 |
| **分类** | 基于向量的文本分类 |
| **推荐** | 基于语义相似度的推荐 |
| **去重** | 语义去重 |
| **零样本分类** | 用 Embedding 做零样本分类 |

**主流 Embedding 模型详见 08-RAG知识库体系.md 第六章。**

### 5.2 Reranker 模型

**定义:** Cross-Encoder 模型,对初筛结果重新排序,提升检索精度。

**原理:**
```
[Bi-Encoder](用于初检,快但粗)
    Query → Encoder → Query Vector
    Doc   → Encoder → Doc Vector
    → Cosine 相似度

[Cross-Encoder](用于重排,慢但精)
    [Query, Doc] → Encoder → Score
```

**主流 Reranker:**
| 模型 | 来源 | 特点 |
|------|------|------|
| **bge-reranker-v2-m3** | 智源 | 多语言、中文优秀 |
| **Cohere Rerank 3** | Cohere | 商用、强 |
| **Jina Reranker v2** | Jina | 开源 |
| **Voyage Rerank-2** | Voyage AI | 商用顶级 |
| **ColBERT v2** | 开源 | 延迟交互 |
| **RankGPT** | 开源 | LLM 做 rerank |

## 六、训练流程:Pre-training → SFT → 对齐

### 6.1 LLM 训练三阶段

```
┌──────────────────────────────────────────────────────────────┐
│                  LLM 训练全流程                                │
│                                                              │
│  [阶段 1] Pre-training(预训练)                                │
│    数据: 万亿级 Token,无监督                                  │
│    任务: Next Token Prediction                               │
│    产出: Base Model(基座模型)                                │
│    成本: 极高(千万到亿美金)                                  │
│         ↓                                                    │
│  [阶段 2] SFT(监督微调)                                      │
│    数据: 10万-100万 指令-回答对                               │
│    任务: 指令遵循                                             │
│    产出: SFT Model(对话模型)                                │
│    成本: 中(千到万美金)                                     │
│         ↓                                                    │
│  [阶段 3] 对齐(RLHF/DPO/KTO)                                │
│    数据: 10万级 人类偏好对                                    │
│    任务: 偏好对齐                                             │
│    产出: Aligned Model(对齐模型)                            │
│    成本: 中(千到万美金)                                     │
│         ↓                                                    │
│  [发布] Chat / Instruct 版模型                                │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 Pre-training(预训练)

**定义:** 在大规模无监督语料上训练,让模型学习语言的统计规律。

**数据来源:**
- 网页爬取(Common Crawl、C4)
- 书籍(Books3、Gutenberg)
- 代码(GitHub、The Stack)
- 学术论文(arXiv、S2ORC)
- 百科(Wikipedia)
- 对话数据(Reddit、Stack Exchange)

**核心任务:** Next Token Prediction(预测下一个 Token)

```
输入: "今天天气"
目标: "很"
损失: CrossEntropy(预测分布, 真实 Token)
```

**训练成本:**
| 模型规模 | 训练成本(GPU 小时) | 大致美金 |
|---------|---------------------|---------|
| 7B | 80K-170K H100 | $1M-2M |
| 70B | 1M-6M H100 | $10M-60M |
| 175B | 10M+ H100 | $100M+ |
| 671B(DeepSeek-V3) | 2.79M H800 | $5.6M(优化后) |

### 6.3 SFT(监督微调)

**全称:** Supervised Fine-Tuning

**定义:** 使用标注好的指令-回答对来微调模型,让模型学会遵循指令。

**数据格式:**

```json
{
  "instruction": "翻译成英文",
  "input": "你好世界",
  "output": "Hello world"
}
```

**数据来源:**
- 人工标注(高质量但贵)
- 开源数据集(Alpaca、Vicuna、ShareGPT)
- Self-Instruct(模型生成)
- Evol-Instruct(WizardLM,进化式生成)

**数据规模:** 10万-100万条

**训练方式:**
- 全参微调(更新所有参数)
- LoRA 微调(只训练 LoRA 适配器,主流)
- QLoRA 微调(4bit 量化 + LoRA)

### 6.4 RLHF(人类反馈强化学习)

**全称:** Reinforcement Learning from Human Feedback

**论文:** InstructGPT(Ouyang et al., 2022-03)
**论文链接:** https://arxiv.org/abs/2203.02155

**三阶段:**

```
阶段 1: SFT(已在前面介绍)
    ↓
阶段 2: RM(奖励模型训练)
    数据: 人类偏好排名
    任务: 给回答打分
    输出: Reward Model
    ↓
阶段 3: RL(强化学习)
    算法: PPO(Proximal Policy Optimization)
    目标: 最大化奖励模型打分
    约束: KL 散度(防止偏离 SFT 模型太远)
```

**RM 训练数据:**

```json
{
  "prompt": "什么是 RAG?",
  "response_A": "RAG 是检索增强生成...",
  "response_B": "RAG 是一种技术...",
  "preference": "A > B"
}
```

**PPO 算法核心:**

```
1. 用当前策略生成回答
2. 用 RM 给回答打分
3. 计算优势函数(Advantage)
4. 用 PPO 更新策略,clip 限制更新幅度
5. 加 KL 惩罚,防止偏离参考模型
```

### 6.5 DPO(直接偏好优化)

**论文:** Rafailov et al., "Direct Preference Optimization: Your Language Model is Secretly a Reward Model", 2023-05
**论文链接:** https://arxiv.org/abs/2305.18290

**核心思想:** 直接用偏好数据优化策略,不需要训练奖励模型和 PPO。

**损失函数:**

```
L_DPO = -log σ(β · [log(π(y_w|x)/π_ref(y_w|x)) - log(π(y_l|x)/π_ref(y_l|x))])

其中:
- π: 当前策略
- π_ref: 参考策略(SFT 模型)
- y_w: 偏好回答(win)
- y_l: 不偏好回答(lose)
- β: 温度参数
```

**优势:**
- 更简单:不需要 RM 和 PPO
- 更稳定:训练更容易收敛
- 效果相当:在很多任务上和 RLHF 相当
- 更省资源:不需要同时维护 4 个模型(SFT/RM/Policy/Value)

### 6.6 其他对齐方法

| 方法 | 说明 | 特点 |
|------|------|------|
| **KTO** | Kahneman-Tversky Optimization | 心理学理论启发,只需二元反馈 |
| **ORPO** | Odds Ratio Preference Optimization | 单阶段,无需 SFT |
| **RLAIF** | AI 反馈的 RL | 用 AI 替代人类标注 |
| **Constitutional AI** | 宪法 AI(Anthropic) | 用原则自我对齐 |
| **SimPO** | Simple Preference Optimization | 简化 DPO,无参考模型 |
| **IPO** | Identity Preference Optimization | DPO 的正则化变体 |
| **RSO** | Rejection Sampling Optimization | 拒绝采样优化 |

### 6.7 SFT vs RLHF vs DPO 对比

| 维度 | SFT | RLHF | DPO |
|------|-----|------|-----|
| **阶段** | 基础对齐 | 进阶对齐 | 进阶对齐 |
| **数据** | 指令-回答对 | 人类偏好 + RM | 人类偏好 |
| **训练方式** | 监督学习 | PPO 强化学习 | 直接优化 |
| **复杂度** | 低 | 高 | 中 |
| **效果** | 基础 | 最好 | 接近 RLHF |
| **成本** | 低 | 高 | 中 |
| **稳定性** | 高 | 中(PPO 难调) | 高 |
| **需要 RM** | 否 | 是 | 否 |
| **需要参考模型** | 否 | 是 | 是 |

## 七、LoRA(低秩适配)深度

### 7.1 基础定义

**全称:** Low-Rank Adaptation

**论文:** Hu et al., "LoRA: Low-Rank Adaptation of Large Language Models", 2021-06
**论文链接:** https://arxiv.org/abs/2106.09685

**定义:** 一种高效的微调技术,通过在模型权重上添加低秩矩阵来微调,只训练少量参数。

### 7.2 LoRA 的核心思想

```
原始权重: W (d × d)
                  ↓ 冻结,不训练
LoRA 增量: ΔW = A · B
           其中 A: d × r
                B: r × d
                r << d (如 r=8)
                  ↓ 只训练 A 和 B

最终: h = W·x + (A·B)·x
        ↑       ↑
        原始   LoRA 增量
```

**数学基础:**
- 假设微调的权重变化 ΔW 是低秩的
- 用两个小矩阵 A (d×r) 和 B (r×d) 近似 ΔW
- 参数量从 d² 降到 2dr(r 通常 8-64)

**示例:**
- 原始 d=4096,参数量 = 4096² = 16.7M
- LoRA r=8,参数量 = 2 × 4096 × 8 = 65K
- 压缩比:256x

### 7.3 LoRA 的优势

| 优势 | 说明 |
|------|------|
| **参数少** | 只训练原模型 0.1%~1% 的参数 |
| **速度快** | 训练时间短 |
| **成本低** | 需要的显存少(只需存 LoRA 权重的梯度) |
| **可插拔** | 可以切换不同的 LoRA 适配器 |
| **不遗忘** | 不影响原始模型能力(原始权重冻结) |
| **可合并** | 推理时可以合并到原始权重,无推理开销 |
| **可组合** | 多个 LoRA 可以叠加 |

### 7.4 LoRA 的关键参数

| 参数 | 说明 | 典型值 |
|------|------|--------|
| **rank(r)** | 低秩矩阵的秩 | 8/16/32/64 |
| **alpha** | 缩放因子(实际缩放 = alpha/rank) | 通常 = 2 * rank |
| **dropout** | Dropout 概率 | 0.05 |
| **target_modules** | 应用 LoRA 的层 | q_proj, v_proj, k_proj, o_proj, gate_proj, up_proj, down_proj |

**rank 选择:**
- r=8:轻量任务,小数据集
- r=16-32:中等任务,通用
- r=64-128:复杂任务,大数据集

### 7.5 LoRA 变体

| 变体 | 说明 | 优势 |
|------|------|------|
| **QLoRA** | 4bit 量化 + LoRA | 极致省显存(7B 模型只需 6GB) |
| **LoRA+** | 改进的 LoRA,A/B 不同学习率 | 更好的效果 |
| **DoRA** | 权重分解 + LoRA | 方向和大小分开 |
| **AdaLoRA** | 自适应秩的 LoRA | 重要层分配更高秩 |
| **VeRA** | Vector-based LoRA | 共享矩阵,参数更少 |
| **LongLoRA** | 长上下文 LoRA | 高效扩展上下文 |
| **GaLore** | Gradient Low-Rank Projection | 优化器级别的低秩 |

### 7.6 LoRA 代码示例(PEFT 库)

```python
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM

# 1. 加载基座模型
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B",
    torch_dtype=torch.bfloat16,
    device_map="auto"
)

# 2. 配置 LoRA
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj"
    ]
)

# 3. 应用 LoRA
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# 输出: trainable params: 6,553,600 || all params: 8,072,204,288 || trainable%: 0.08%

# 4. 训练(与正常训练相同)
trainer = Trainer(model=model, ...)
trainer.train()

# 5. 保存(只保存 LoRA 权重,几十 MB)
model.save_pretrained("./lora_output")
# 加载时:base_model + PeftModel.from_pretrained(base, "./lora_output")
```

### 7.7 QLoRA(量化 LoRA)

**论文:** Dettmers et al., "QLoRA: Efficient Finetuning of Quantized LLMs", 2023-05
**论文链接:** https://arxiv.org/abs/2305.14314

**核心创新:**
- NF4(NormalFloat 4-bit)量化:信息论最优的 4-bit 量化
- Double Quantization:量化常数也量化,进一步省内存
- Paged Optimizer:用 CPU offload 处理显存峰值
- 基座模型 4-bit 冻结,LoRA 用 BF16 训练

**显存需求:**
| 模型 | 全参微调 | LoRA | QLoRA |
|------|---------|------|-------|
| 7B | 120GB+ | 20GB | 6GB |
| 13B | 240GB+ | 40GB | 12GB |
| 70B | 1200GB+ | 200GB | 48GB |

## 八、量化(Quantization)深度

### 8.1 基础定义

**定义:** 降低模型权重和激活的数值精度,减少内存占用和加速推理。

### 8.2 数值精度详解

| 精度 | 位数 | 取值范围 | 特点 | 用途 |
|------|------|----------|------|------|
| **FP32** | 32 | ±3.4e38 | 单精度,高精度 | 训练基准 |
| **FP16** | 16 | ±65504 | 半精度,易溢出 | 训练/推理 |
| **BF16** | 16 | ±3.4e38 | 大脑浮点,动态范围大 | 训练(更稳定) |
| **FP8(E4M3)** | 8 | ±448 | 8 位浮点 | 推理(新兴) |
| **FP8(E5M2)** | 8 | ±57344 | 8 位浮点,大动态范围 | 推理 |
| **INT8** | 8 | -128~127 | 8 位整数 | 推理 |
| **INT4** | 4 | -8~7 | 4 位整数 | 推理(极致压缩) |
| **NF4** | 4 | -1~1 | NormalFloat 4-bit | QLoRA |
| **INT2** | 2 | -2~1 | 2 位(实验性) | 极致压缩 |

### 8.3 量化技术对比

| 技术 | 全称 | 说明 | 代表实现 | 特点 |
|------|------|------|----------|------|
| **GPTQ** | Generalized Post-Training Quantization | 基于二阶信息的量化 | AutoGPTQ | 精度高,需要校准数据 |
| **AWQ** | Activation-aware Weight Quantization | 激活感知权重量化 | AutoAWQ | 速度快,精度好 |
| **GGUF** | GGML Universal Format | llama.cpp 的量化格式 | llama.cpp | 边缘部署友好 |
| **bitsandbytes** | bitsandbytes | 8/4 位量化库 | bitsandbytes | 简单,集成 HF |
| **SmoothQuant** | SmoothQuant | 平滑激活的量化 | MIT | 解决激活异常值 |
| **SpQR** | Sparse-Quantized Representation | 稀疏量化 | ITHKG | 极致压缩 |
| **QLoRA NF4** | NF4 | 信息论最优 4-bit | QLoRA | 训练时量化 |

### 8.4 量化的类型

```
按时机:
├── PTQ(Post-Training Quantization)
│   └── 训练后量化,无需重训
└── QAT(Quantization-Aware Training)
    └── 量化感知训练,训练时模拟量化

按对象:
├── Weight-Only Quantization
│   └── 只量化权重(激活保持高精度)
└── Weight+Activation Quantization
    └── 权重和激活都量化
```

### 8.5 量化的权衡

```
精度高 ←──────→ 精度低
FP32 → BF16 → FP16 → INT8 → INT4 → INT2
速度慢 ←──────→ 速度快
体积大 ←──────→ 体积小
```

**经验法则:**
- 追求质量:FP16/BF16
- 平衡:INT8
- 极致压缩:INT4(质量略有损失,但通常可接受)
- 边缘部署:INT4 GGUF

### 8.6 量化代码示例

```python
# 1. bitsandbytes(INT8/INT4)
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",

    bnb_4bit_use_double_quant=True

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B",
    quantization_config=bnb_config

# 2. GPTQ
from auto_gptq import AutoGPTQForCausalLM

    "TheBloke/Llama-3-8B-GPTQ",
    use_safetensors=True

# 3. AWQ
from awq import AutoAWQForCausalLM

# 4. GGUF(用 llama.cpp)
# 命令行: ./main -m llama-3-8b.gguf -p "你好"
```

## 九、MoE(混合专家模型)深度

### 9.1 基础定义

**全称:** Mixture of Experts

**定义:** 一种模型架构,由多个专家子网络组成,每次推理只激活部分专家,实现"参数量大但计算量小"。

### 9.2 MoE 的原理

```
输入
  ↓
[路由网络 Router] → 选择 Top-K 个专家
  ↓
[专家 1][专家 2] ... [专家 N]
  (只激活 K 个, K < N)
  ↓
加权求和(权重 = Router 输出的 softmax)
  ↓
输出
```

**核心思想:**
- 总参数量大(所有专家),但每次推理只用少数专家(计算量小)
- 不同 Token 路由到不同专家,实现专业化
- 训练时所有专家并行训练

### 9.3 MoE 在 Transformer 中的位置

MoE 通常替换 FFN 层:

```
标准 Transformer 层:
    Attention → FFN → 输出

MoE Transformer 层:
    Attention → MoE(多专家 FFN)→ 输出
```

### 9.4 MoE 的优势和挑战

| 优势 | 挑战 |
|------|------|
| 参数量大,能力强 | 路由策略很关键 |
| 推理速度快(只激活部分) | 专家负载均衡 |
| 训练更快(数据并行+专家并行) | 训练不稳定 |
| 成本效益高 | 部分专家可能没用 |
| 专业化(不同专家学不同知识) | 显存占用大(所有专家都要加载) |
| 可解释性(看路由了解决策) | 通信开销大(多卡) |

### 9.5 路由策略

| 策略 | 说明 | 特点 |
|------|------|------|
| **Top-K Routing** | 选概率最高的 K 个专家 | 标准 MoE |
| **Top-2 Routing** | K=2,最常用 | 平衡负载和精度 |
| **Expert Choice** | 专家选 Token | 反向路由 |
| **Soft Routing** | 所有专家加权 | 类似 dense |
| **Hash Routing** | 哈希路由 | 简单,无学习 |

### 9.6 负载均衡

**问题:** 路由可能让少数专家被过度使用,其他专家闲置。

**解决:**
- **Auxiliary Loss:** 加一个辅助损失,鼓励均匀路由
- **Capacity Factor:** 限制每个专家的最大 Token 数
- **Expert Balancing:** 动态调整路由概率

### 9.7 主流 MoE 模型

| 模型 | 厂商 | 总参数/激活参数 | 特点 |
|------|------|----------------|------|
| **Mixtral 8x7B** | Mistral | 47B / 12.9B | 开源 MoE 标杆 |
| **Mixtral 8x22B** | Mistral | 141B / 39B | 更强 MoE |
| **DeepSeek-V2** | 深度求索 | 236B / 21B | MLA + MoE |
| **DeepSeek-V3** | 深度求索 | 671B / 37B | 高效训练、性价比 |
| **DeepSeek-R1** | 深度求索 | 671B / 37B | 推理模型 |
| **Qwen3-235B-A22B** | 阿里 | 235B / 22B | 中文 MoE |

## 十、推理优化技术

### 10.1 KV Cache

**定义:** 缓存自回归生成中已计算的 Key 和 Value,避免重复计算。

**原理:**
```
[无 KV Cache]
每生成一个 Token,都要重新计算所有历史 Token 的 K/V
复杂度: O(n²) per token

[有 KV Cache]
缓存历史 K/V,只计算新 Token 的 K/V
复杂度: O(n) per token
```

**节省:** 对于长序列,KV Cache 可加速数十倍。

### 10.2 KV Cache 内存占用

```
KV Cache 大小 = 2 × n_layers × n_heads × d_head × seq_len × batch × dtype_size

示例(Llama-3-8B,32K 上下文,BF16):
= 2 × 32 × 32 × 128 × 32768 × 1 × 2
= 16.8 GB(超过模型权重!)
```

**优化:**
- GQA/MQA/MLA 减少 KV 头数
- KV Cache 量化(INT8/INT4)
- PagedAttention 内存碎片化
- Sliding Window Attention

### 10.3 FlashAttention

**论文:** Dao et al., "FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness", 2022-06
**论文链接:** https://arxiv.org/abs/2205.14135

**核心:** 通过分块计算和 IO 感知,在 GPU SRAM 中完成 Attention,减少 HBM 访问。

**优势:**
- 速度快(2-4x)
- 内存省(5-10x,无需 n² 中间矩阵)
- 精确(不是近似)

**版本:**
- FlashAttention v1(2022):基础版
- FlashAttention v2(2023):更好的并行
- FlashAttention v3(2024):Hopper 架构优化

### 10.4 PagedAttention(vLLM)

**论文:** Kwon et al., "Efficient Memory Management for Large Language Model Serving with PagedAttention", 2023-06
**论文链接:** https://arxiv.org/abs/2306.03069

**核心:** 借鉴操作系统的虚拟内存分页机制,管理 KV Cache。

**优势:**
- 减少 KV Cache 内存碎片
- 支持更高的 batch size
- 提升吞吐量 2-4 倍
- 支持动态 batch(连续批处理)

### 10.5 Speculative Decoding(推测解码)

**论文:** Leviathan et al., "Fast Inference from Transformers via Speculative Decoding", 2022-11
**论文链接:** https://arxiv.org/abs/2211.17192

**核心:** 用小模型(草稿模型)快速生成多个 Token,大模型批量验证。

```
1. 小模型(draft)生成 K 个 Token: [t1, t2, t3]
2. 大模型(target)一次前向验证这 K 个 Token
3. 接受匹配的 Token,拒绝的 Token 重新生成
4. 接受的 Token 多个一起输出,加速推理
```

**优势:**
- 推理速度提升 2-3 倍
- 输出质量与大模型完全一致
- 适合延迟敏感场景

**变体:**
- **Medusa:** 多头并行预测
- **Eagle:** 自回归草稿
- **Lookahead:** 大模型自身预测

### 10.6 Continuous Batching(连续批处理)

**传统 Batch:**
- 等所有请求生成完才换新 batch
- 短请求被长请求拖累

**Continuous Batching:**
- 任何请求生成完,立刻换新请求
- 动态调整 batch 组成
- 显著提升 GPU 利用率

### 10.7 推理优化技术总结

| 技术 | 加速 | 适用场景 |
|------|------|---------|
| KV Cache | 10x+ | 所有自回归 |
| FlashAttention | 2-4x | 所有 Attention |
| PagedAttention | 2-4x(吞吐) | 服务端 |
| Speculative Decoding | 2-3x | 延迟敏感 |
| Continuous Batching | 2-4x(吞吐) | 服务端 |
| 量化(INT8/INT4) | 1.5-2x | 边缘/服务 |
| Tensor Parallelism | 线性 | 多 GPU |
| Pipeline Parallelism | 线性 | 多 GPU |

### 11.1 基础定义

**定义:** 模型一次能处理的最大 Token 数量。

### 11.2 上下文窗口的演进

### 11.3 长上下文的挑战

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **计算量** | Attention 是 O(n²),长上下文计算量大 | FlashAttention、稀疏注意力 |
| **记忆衰减** | "Lost in the Middle",中间信息容易被忽略 | 重新排序、RoPE 改进 |

| **LongRoPE** | 微软的位置插值 | 2M 上下文 |
| **Ring Attention** | 分布式长上下文 | 1M+ 上下文 |

### 11.5 "Lost in the Middle" 问题

**论文:** Liu et al., "Lost in the Middle: How Language Models Use Long Contexts", 2023-07
**论文链接:** https://arxiv.org/abs/2307.03172

**核心发现:** LLM 对长上下文的利用呈现 U 形——开头和结尾的信息利用好,中间的信息利用差。

**应对:**
- 把关键信息放在开头和结尾
- 用 RAG 替代长上下文
- 重新排序检索结果

- 后续请求复用,降低 90% 成本

## 十二、Streaming(流式输出)

### 12.1 基础定义

**定义:** 模型边生成边输出,用户可以实时看到生成过程,而不是等全部完成后才显示。

### 12.2 流式输出的实现方式

| 方式 | 说明 | 场景 |
|------|------|------|
| **SSE** | Server-Sent Events | Web 应用最常用 |
| **WebSocket** | 双向通信 | 实时交互 |
| **HTTP Chunked** | 分块传输 | 简单场景 |
| **gRPC Stream** | gRPC 流 | 微服务 |

### 12.3 SSE 流式示例

```python
# 服务端(Python + FastAPI)
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import anthropic

app = FastAPI()
client = anthropic.Anthropic()

@app.get("/chat")
def chat():
    def stream():
        with client.messages.stream(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1024,
            messages=[{"role": "user", "content": "你好"}]
        ) as stream:
            for text in stream.text_stream:
                yield f"data: {text}\n\n"
    return StreamingResponse(stream(), media_type="text/event-stream")

# 客户端(JavaScript)
const eventSource = new EventSource("/chat");
eventSource.onmessage = (e) => {
    document.getElementById("output").innerHTML += e.data;
};
```

### 12.4 流式输出的优势

- **用户体验好:** 可以看到实时进度,减少等待感
- **可中断:** 用户可以中途停止
- **节省时间:** 不需要等全部生成完
- **首字节延迟低:** 第一个 Token 很快返回

### 12.5 流式的延迟指标

| 指标 | 说明 |
|------|------|
| **TTFT(Time To First Token)** | 首 Token 延迟,用户体验关键 |
| **TPOT(Time Per Output Token)** | 每个输出 Token 的平均时间 |
| **总延迟** | TTFT + (n_tokens × TPOT) |

## 十四、Distillation(模型蒸馏)

### 14.1 基础定义

**定义:** 用大模型(教师)的输出训练小模型(学生),让小模型学习大模型的能力。

### 14.2 蒸馏的原理

```
大模型(教师)
    ↓ 输出软标签(概率分布,带温度)
小模型(学生)
    ↓ 学习拟合教师的输出(KL 散度)
得到一个更小但能力接近的模型
```

**损失函数:**

```
L = α · L_hard(y_true, y_pred) + (1-α) · L_soft(y_teacher, y_student)

其中:
- L_hard: 硬标签损失(交叉熵)
- L_soft: 软标签损失(KL 散度)
- α: 平衡系数
- 温度 T: softmax(logits/T)
```

### 14.3 蒸馏的价值

- **减小模型体积:** 大模型的能力,小模型的体积
- **加速推理:** 小模型推理更快
- **降低成本:** 部署成本更低
- **边缘部署:** 可以在端侧部署

### 14.4 蒸馏的类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **知识蒸馏** | 学习教师模型的输出分布 | 经典蒸馏 |
| **注意力蒸馏** | 学习教师的注意力模式 | MiniLM |
| **逐层蒸馏** | 每一层都做蒸馏 | TinyBERT |
| **自蒸馏** | 自己教自己(如模型集成) | Self-Distillation |
| **数据蒸馏** | 用大模型生成数据训小模型 | Alpaca |
| **推理蒸馏** | 学习大模型的推理过程 | DeepSeek-R1-Distill |

### 14.5 DeepSeek-R1 蒸馏案例

DeepSeek-R1(671B)通过推理蒸馏,训练了多个小模型:

| 学生模型 | 基座 | 效果 |
|---------|------|------|
| DeepSeek-R1-Distill-Qwen-1.5B | Qwen 2.5 | 1.5B 接近 GPT-4 推理 |
| DeepSeek-R1-Distill-Qwen-7B | Qwen 2.5 | 7B 超越 GPT-4 推理 |
| DeepSeek-R1-Distill-Qwen-32B | Qwen 2.5 | 32B 接近 o1 |
| DeepSeek-R1-Distill-Llama-8B | Llama 3.1 | 8B 强推理 |
| DeepSeek-R1-Distill-Llama-70B | Llama 3.3 | 70B 接近 R1 |

## 十五、Toolformer(工具专用模型)

### 15.1 基础定义

**论文:** Schick et al., "Toolformer: Language Models Can Teach Themselves to Use Tools", 2023-02
**论文链接:** https://arxiv.org/abs/2302.04761

**定义:** Meta 提出的让模型自己学会使用工具的方法,模型可以自主决定何时使用工具、使用哪个工具。

### 15.2 Toolformer 的核心思想

- 模型自己决定何时调用工具
- 不需要人类标注工具使用示例
- 通过自监督学习学会使用工具
- 可以用多种工具(计算器、搜索引擎、翻译等)

### 15.3 Toolformer 训练流程

```
[1] 工具调用候选生成
    在数据集中插入工具调用位置
    ↓
[2] 自监督筛选
    执行工具调用,看结果是否有助于下游任务
    ↓
[3] 用筛选后的数据微调
    让模型学会在合适位置调用合适工具
```

### 15.4 与 Function Calling 的区别

| 维度 | Toolformer | Function Calling |
|------|-----------|-------------------|
| **训练方式** | 自监督学习 | 监督微调 |
| **是否需要标注** | 不需要 | 需要 |
| **工具选择** | 模型自主决定 | 按格式输出 |
| **灵活性** | 更高 | 取决于设计 |
| **可控性** | 较低 | 高(开发者定义工具) |
| **商用** | 研究为主 | 主流(GPT-4、Claude) |
