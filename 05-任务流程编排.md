## 一、Workflow vs Agent(Anthropic 官方核心区分)

### 1.1 官方定义(Anthropic《Building Effective Agents》)

> **Agentic Systems(智能体系统)** 是 Anthropic 对上述变体的统称,但区分为两类架构:
>
> - **Workflows(工作流):** 通过**预定义代码路径**编排 LLM 与工具的系统
> - **Agents(智能体):** LLM **动态主导**自身流程与工具使用,自主控制如何完成任务

**核心选型原则(Anthropic 官方建议):**

> 找到**最简单的可行方案**,只在需要时增加复杂度。有时甚至不需要构建 agentic 系统——单次 LLM 调用加优化可能就够。
>
> - 任务明确、需可预测性 → **Workflows**
> - 需要灵活性、模型驱动决策 → **Agents**

### 1.2 Workflow vs Agent 全面对比

| 维度 | Workflow(工作流) | Agent(智能体) |
|------|-------------------|------------------|
| **流程控制** | 预定义代码路径 | LLM 自主决策 |
| **可控性** | 高(开发者掌控) | 较低(LLM 决策) |
| **可预测性** | 高(每次走相同路径) | 低(每次可能不同) |
| **灵活性** | 低 | 高 |
| **延迟** | 低(固定步骤) | 高(循环+推理) |
| **成本** | 可预估 | 不可预估 |
| **调试难度** | 容易(已知路径) | 难(动态路径) |
| **适用任务** | 流程明确、可拆解 | 开放性、探索性 |
| **典型场景** | 翻译、分类、ETL | Bug 修复、研究、创作 |
| **失败模式** | 单步失败可定位 | 可能死循环、跑题 |

### 1.3 五大工作流模式总览

```
┌──────────────────────────────────────────────────────────────┐
│ Workflow(预定义路径)——4 种                                   │
│  ├── Prompt Chaining   串行链式调用                          │
│  ├── Routing           分类后路由到专用处理                  │
│  ├── Evaluator-Optimizer  生成-评估-优化循环                 │
│  └── Orchestrator-Workers  动态拆分+分配+汇总                │
│                                                              │
│ Agent(LLM 动态主导)——1 种                                   │
│  └── Agent              自主循环,工具与环境反馈驱动          │
└──────────────────────────────────────────────────────────────┘
```

**复杂度递增顺序:**

```
单次 LLM 调用
    ↓ 复杂度↑
Prompt Chaining(串行链)
    ↓
Routing(路由分发)
    ↓
Evaluator-Optimizer(评估优化)
    ↓
Orchestrator-Workers(编排工人)
    ↓
Agent(纯智能体,最复杂)
```

## 二、Anthropic 五大工作流模式深度解析

> 来源:Anthropic《Building Effective Agents》
> 官方链接:https://www.anthropic.com/research/building-effective-agents

### 2.1 Prompt Chaining(提示链)

**定义:** 把任务拆成**串行**的多个 LLM 调用,每步输出作为下一步输入,中间可插入确定性代码(如格式化、检索)。

```
┌────────┐    ┌─────────┐    ┌──────────┐    ┌─────────┐    ┌────────┐
│ 输入   │───▶│ LLM 1   │───▶│ Gate/代码│───▶│ LLM 2   │───▶│ 输出   │
│        │    │ 生成大纲│    │ 校验     │    │ 写正文  │    │        │
└────────┘    └─────────┘    └──────────┘    └─────────┘    └────────┘
```

**核心特征:**
- 串行执行,前一步输出 → 后一步输入
- 中间可插入"门控"(Gate),用于校验/过滤/路由
- 整个链路任一步失败,整个流程失败
- 适合可拆解为**固定子任务**的场景

**典型应用:**
- **翻译流水线:** 翻译 → 校对 → 润色
- **写作流水线:** 大纲 → 草稿 → 修订 → 终稿
- **代码生成:** 设计 → 实现 → 测试 → 文档
- **数据处理:** 抽取 → 清洗 → 转换 → 加载

**Anthropic 官方示例(Prompt Chaining):**

```python
import anthropropic

client = anthropic.Anthropic()

# 第一步:生成大纲
outline_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "生成一篇关于 AI Agent 的文章大纲,5 个章节"}]
)
outline = outline_response.content[0].text

# 第二步:根据大纲写正文
article_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=4096,
    messages=[{"role": "user", "content": f"根据以下大纲写正文:\n{outline}"}]
)
article = article_response.content[0].text

# 第三步:校对
final_response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=4096,
    messages=[{"role": "user", "content": f"校对并优化以下文章:\n{article}"}]
)
print(final_response.content[0].text)
```

**优点:**
- 流程清晰,每步可单独优化
- 可在中间步骤做校验(Gate)
- 容易调试,定位问题

**缺点:**
- 串行延迟累积
- 任一步失败全失败
- 灵活性低,无法动态调整

### 2.2 Routing(路由分发)

**定义:** 先用 LLM/分类器判定输入类型,路由到不同的专用处理流程,每个流程有独立 Prompt 和工具。

```
        ┌────────┐
        │  输入  │
        └───┬────┘
            ↓
     ┌──────────────┐
     │ Router LLM   │  ← 分类(意图识别)
     │ "这是哪类?" │
     └──────┬───────┘
            ↓
   ┌────────┼────────┬────────┐
   ↓        ↓        ↓        ↓
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│退款  │ │技术  │ │咨询  │ │投诉  │  ← 专用 Agent
│Agent │ │Agent │ │Agent │ │Agent │
└──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘
   ↓        ↓        ↓        ↓
   └────────┴────────┴────────┘
            ↓
        ┌────────┐
        │  输出  │
        └────────┘
```

**典型应用:**
- **客服系统:** 退款/技术/咨询/投诉分流
- **多领域问答:** 医疗/法律/金融分类
- **不同复杂度分流:** 简单问题用小模型,复杂问题用大模型
- **不同语言分流:** 中文/英文/日文走不同翻译流程

**Anthropic 官方示例(Routing):**

```python
def route_request(user_input: str):
    """路由分类"""
    classify_response = client.messages.create(
        model="claude-haiku-4-5",  # 小模型分类,快
        max_tokens=20,
        messages=[{"role": "user", "content": f"""
        把以下用户请求分类为:refund(退款)、tech(技术)、consult(咨询)。
        只输出类别名。
        用户请求:{user_input}
        """}]
    )
    category = classify_response.content[0].text.strip()
    
    # 路由到不同 Agent
    routes = {
        "refund": handle_refund,
        "tech": handle_tech,
        "consult": handle_consult
    }
    return routes.get(category, handle_default)(user_input)
```

**优点:**
- 每个 Agent 专注单一领域,Prompt 更精简
- 可用小模型做分类(省钱),大模型做处理
- 关注点分离,易维护

**缺点:**
- 分类错误会路由到错误流程
- 类别需要预先定义
- 多了一步分类延迟

### 2.3 Evaluator-Optimizer(评估-优化)

**定义:** 一个 LLM 生成输出,另一个 LLM 评估,根据评估反馈迭代改进,直到满足标准。

```
       ┌──────────────────────────────────────┐
       │                                      ↓
┌──────┴─────┐    ┌──────────┐    ┌──────────┴───┐
│ Generator  │───▶│Evaluator │───▶│ 满意?        │
│ LLM 生成   │    │ LLM 评估 │    │ Yes → 输出   │
└──────┬─────┘    └──────────┘    │ No → 反馈    │
       ▲                          └──────────────┘
       │                                 │
       └─────────── 反馈 ────────────────┘
```

**典型应用:**
- **文学翻译:** 翻译 → 校对 → 润色,反复打磨
- **代码生成:** 生成 → 跑测试 → 修复,直到通过
- **文案撰写:** 生成 → 评分 → 修改,直到达标
- **安全审计:** 生成 → 安全检查 → 修正,直到无漏洞

**Anthropic 官方示例(Evaluator-Optimizer):**

```python
def generate_and_optimize(task: str, max_iterations: int = 3):
    """生成-评估-优化循环"""
    current_output = ""
    feedback = ""
    
    for i in range(max_iterations):
        # 生成(带反馈)
        gen_prompt = f"任务:{task}\n"
        if feedback:
            gen_prompt += f"\n上一版反馈:{feedback}\n请改进。"
        gen_response = client.messages.create(
            model="claude-sonnet-4-5",
            max_tokens=2048,
            messages=[{"role": "user", "content": gen_prompt}]
        )
        current_output = gen_response.content[0].text
        
        # 评估
        eval_response = client.messages.create(
            model="claude-sonnet-4-5",
            max_tokens=1024,
            messages=[{"role": "user", "content": f"""
            评估以下输出,给出 0-100 分和改进建议。
            如果 ≥85 分,输出 "PASS"。
            
            任务:{task}
            输出:{current_output}
            """}]
        )
        eval_result = eval_response.content[0].text
        if "PASS" in eval_result:
            return current_output
        feedback = eval_result
    
    return current_output  # 达到最大迭代次数返回最后版本
```

**优点:**
- 自我改进,质量高
- 有明确目标(评估标准)
- 可解释(每次迭代有反馈)

**缺点:**
- 迭代次数多,成本高
- 评估器本身可能不准
- 可能陷入"局部最优"

### 2.4 Orchestrator-Workers(编排-工人)

**定义:** 中央编排 LLM 动态拆分任务,分发给多个 Worker 并行处理,再汇总结果。**与 Routing 的关键区别:任务拆分是动态的,而非预先定义。**

```
        ┌─────────────────┐
        │   Orchestrator  │  ← 中央编排 LLM
        │   (编排者)      │
        └────────┬────────┘
                 │
   ┌─────────────┼─────────────┐
   │  动态拆分   │  分配       │  汇总
   ↓             ↓             ↓
┌──────┐    ┌──────┐    ┌──────┐
│Worker│    │Worker│    │Worker│  ← 并行 Worker
│  1   │    │  2   │    │  3   │
└──┬───┘    └──┬───┘    └──┬───┘
   │           │           │
   └───────────┼───────────┘
               ↓
        ┌─────────────┐
        │  Orchestrator │  ← 汇总整合
        │   整合输出    │
        └─────────────┘
```

**典型应用:**
- **复杂代码修改:** 多文件并行修改(每个 Worker 改一个文件)
- **调研任务:** 多角度并行调研(每个 Worker 调研一个方面)
- **报告生成:** 分章节并行撰写(每个 Worker 写一章)
- **多语言翻译:** 多语言并行翻译(每个 Worker 翻译一种语言)

**Anthropic 官方示例(Orchestrator-Workers):**

```python
def orchestrator_workers(task: str):
    """编排-工人模式"""
    # 第一步:Orchestrator 拆分任务
    plan_response = client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=1024,
        messages=[{"role": "user", "content": f"""
        把以下任务拆分为可并行的子任务,输出 JSON 数组:
        任务:{task}
        
        格式:[{{"subtask": "...", "type": "..."}}]
        """}]
    )
    subtasks = parse_json(plan_response.content[0].text)
    
    # 第二步:并行执行 Worker
    results = []
    with ThreadPoolExecutor() as executor:
        futures = [executor.submit(worker_execute, st) for st in subtasks]
        results = [f.result() for f in futures]
    
    # 第三步:Orchestrator 汇总
    synth_response = client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=4096,
        messages=[{"role": "user", "content": f"""
        整合以下子任务结果为最终输出:
        原任务:{task}
        子结果:{json.dumps(results, ensure_ascii=False)}
        """}]
    )
    return synth_response.content[0].text
```

**优点:**
- 并行加速
- 适应性强(动态拆分)
- 每个 Worker 上下文精简

**缺点:**
- Orchestrator 是瓶颈(单点)
- 汇总整合复杂
- 子任务间可能有依赖

### 2.5 Agent(纯智能体,LLM 主导)

**定义:** LLM 在 Agent Loop 中自主决策,动态选择工具和步骤,直到完成任务。

```
用户目标
    ↓
┌────────────────────────────────────────┐
│            Agent Loop                  │
│                                        │
│   ┌──────────────────────────────┐     │
│   │  while not done:             │     │
│   │      LLM.Think()             │     │
│   │      LLM.Act(tool)           │     │
│   │      Observe(result)         │     │
│   │      Judge(done?)            │     │
│   └──────────────────────────────┘     │
│                                        │
└────────────────────────────────────────┘
    ↓
最终输出
```

**典型应用:**
- **开放式任务:** 没有固定流程(如"找出这个 bug 的根因")
- **探索性任务:** 如"调研这个新兴技术"
- **创造性任务:** 如"设计一个新产品方案"
- **复杂修复:** 如"修复这个生产事故"

**Anthropic 官方示例(Agent):**

```python
def agent_loop(task: str, max_iterations: int = 20):
    """Agent 循环"""
    messages = [{"role": "user", "content": task}]
    tools = [read_file, edit_file, run_command, search]
    
    for i in range(max_iterations):
        # LLM 决策
        response = client.messages.create(
            model="claude-sonnet-4-5",
            max_tokens=4096,
            tools=tools,
            messages=messages
        )
        messages.append({"role": "assistant", "content": response.content})
        
        # 判断是否结束
        if response.stop_reason == "end_turn":
            return extract_final_answer(response)
        
        # 执行工具调用
        for tool_use in response.content:
            if tool_use.type == "tool_use":
                result = execute_tool(tool_use.name, tool_use.input)
                messages.append({
                    "role": "user",
                    "content": [{"type": "tool_result",
                                "tool_use_id": tool_use.id,
                                "content": result}]
                })
    
    return "达到最大迭代次数,任务未完成"
```

**优点:**
- 最灵活,适应未知情况
- 可处理开放性任务
- 模仿人类解决问题方式

**缺点:**
- 不可预测,可能死循环
- 成本不可预估
- 难以调试

### 2.6 五大模式对比表

| 模式 | 复杂度 | 可控性 | 灵活性 | 延迟 | 成本 | 典型应用 |
|------|--------|--------|--------|------|------|----------|
| **Prompt Chaining** | 低 | 高 | 低 | 中(串行) | 中 | 翻译、写作流水线 |
| **Routing** | 中 | 高 | 中 | 低(并行) | 低(小模型分类) | 客服分流 |
| **Evaluator-Optimizer** | 中 | 中 | 中 | 高(迭代) | 高 | 翻译打磨、代码生成 |
| **Orchestrator-Workers** | 高 | 中 | 高 | 低(并行) | 高 | 多文件代码修改 |
| **Agent** | 最高 | 低 | 最高 | 高(循环) | 最高 | 开放性探索任务 |

### 2.7 选型决策树

```
你的任务是什么类型?
│
├─ 流程明确,可拆解为固定步骤?
│   ├─ 是 → 用 Prompt Chaining
│   └─ 否 ↓
│
├─ 输入可分类,每类有专用处理?
│   ├─ 是 → 用 Routing
│   └─ 否 ↓
│
├─ 有明确评估标准,可迭代改进?
│   ├─ 是 → 用 Evaluator-Optimizer
│   └─ 否 ↓
│
├─ 任务复杂,可动态拆分并行处理?
│   ├─ 是 → 用 Orchestrator-Workers
│   └─ 否 ↓
│
└─ 开放性任务,无法预知步骤?
    └─ 是 → 用 Agent
```

## 三、Pipeline(执行管道)

### 3.1 基础定义

**定义:** 串行处理流水线,数据依次经过多个处理阶段,每个阶段有明确的输入输出格式。

```
输入 → 阶段1 → 阶段2 → 阶段3 → ... → 输出
       (抽取)  (清洗)  (转换)        (加载)
```

### 3.2 常见 Pipeline 模式

| Pipeline 类型 | 阶段示例 | 说明 |
|---------------|----------|------|
| **数据处理管道(ETL)** | Extract → Clean → Transform → Load | 经典 ETL 流程 |
| **RAG 管道** | Load → Split → Embed → Store → Retrieve → Generate | 知识库问答 |
| **LLM 推理管道** | Tokenize → Embed → Infer → Decode → Post-process | 模型推理 |
| **CI/CD 管道** | Lint → Test → Build → Deploy | 持续集成 |
| **NLP 处理管道** | Tokenize → POS → NER → Parse → Analyze | 自然语言处理 |
| **数据处理 ML 管道** | Ingest → Preprocess → Feature → Train → Eval → Deploy | 机器学习 |

### 3.3 RAG Pipeline 详解(以 RAG 为例)

```
┌──────────────────────────────────────────────────────────────┐
│                    RAG Pipeline                              │
│                                                              │
│  [索引阶段(离线)]                                            │
│  文档 → 加载 → 分块 → 嵌入 → 存储                            │
│   │      │      │      │      │                             │
│   │      │      │      │      └─→ 向量数据库(Milvus/PG)    │
│   │      │      │      └─→ Embedding 模型(text-embedding-3)│
│   │      │      └─→ Chunking(500 token / 50 重叠)          │
│   │      └─→ PyPDFLoader / WebBaseLoader                    │
│   └─→ .pdf / .docx / .html / .md                            │
│                                                              │
│  [查询阶段(在线)]                                            │
│  问题 → 嵌入 → 检索 → 重排 → 拼接 → LLM 生成 → 答案         │
│   │      │      │      │      │      │      │               │
│   │      │      │      │      │      │      └─→ Claude/GPT  │
│   │      │      │      │      │      └─→ Prompt 模板        │
│   │      │      │      │      └─→ Top-K 上下文              │
│   │      │      │      └─→ Cohere Rerank                   │
│   │      │      └─→ 向量检索(Top-K=10)                    │
│   │      └─→ Embedding 模型(同索引阶段)                   │
│   └─→ 用户输入                                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.4 Pipeline vs Workflow 区别

| 维度 | Pipeline | Workflow |
|------|----------|----------|
| **核心特征** | 数据流式串行 | 任务流程编排 |
| **分支** | 通常无 | 可有分支 |
| **循环** | 通常无 | 可有循环 |
| **并行** | 通常无 | 可有并行 |
| **结构** | 线性 | 图状(DAG) |
| **典型代表** | ETL、RAG | Airflow、n8n、LangGraph |

### 3.5 LangChain LCEL(LangChain Expression Language)

LCEL 是 LangChain 的声明式管道语法,用 `|` 连接组件:

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_anthropic import ChatAnthropic
from langchain_community.retrievers import BM25Retriever

# 定义组件
prompt = ChatPromptTemplate.from_template("根据以下上下文回答:\n{context}\n\n问题:{question}")
model = ChatAnthropic(model="claude-sonnet-4-5")
parser = StrOutputParser()
retriever = BM25Retriever.from_documents(docs)

# 用 LCEL 组合管道
chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | model
    | parser
)

# 调用
result = chain.invoke("什么是 MCP?")
```

**LCEL 优势:**
- 用 `|` 语法直观表达流水线
- 支持流式、异步、批量
- 自动追踪(Tracing)
- 内置重试、回退

## 四、Orchestrator(编排器)深度

### 4.1 编排器的核心职责

| 职责 | 说明 |
|------|------|
| **任务拆解** | 把大任务拆分为可执行的子任务 |
| **任务分配** | 把子任务分配给合适的 Worker |
| **进度跟踪** | 跟踪各子任务执行状态(pending/running/done/failed) |
| **结果汇总** | 收集各子任务结果,整合为最终输出 |
| **异常处理** | 处理失败的子任务(重试、降级、上报) |
| **依赖管理** | 处理子任务间的依赖关系 |
| **资源调度** | 控制并发数、负载均衡 |
| **状态管理** | 维护整个流程的状态(支持恢复) |

### 4.2 编排器的两种形态

| 形态 | 说明 | 特点 | 典型实现 |
|------|------|------|----------|
| **代码编排器** | 用代码定义流程 | 固定流程,可控性高 | LangGraph、n8n、Airflow |
| **Agent 编排器** | LLM 自主决定如何拆分和分配 | 灵活,适应性强 | Claude Code、OpenAI Swarm |

### 4.3 LangGraph 编排器示例

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    task: str
    subtasks: list
    results: Annotated[list, operator.add]
    final_output: str

def orchestrator(state: State) -> dict:
    """编排器:拆分任务"""
    response = llm.invoke(f"把任务拆分为 3 个子任务:{state['task']}")
    return {"subtasks": parse_subtasks(response)}

def worker(state: State) -> dict:
    """工人:执行所有子任务(并行)"""
    results = []
    for subtask in state["subtasks"]:
        result = llm.invoke(f"执行子任务:{subtask}")
        results.append(result)
    return {"results": results}

def synthesizer(state: State) -> dict:
    """汇总器:整合结果"""
    final = llm.invoke(f"整合以下结果:{state['results']}")
    return {"final_output": final}

# 构建图
workflow = StateGraph(State)
workflow.add_node("orchestrator", orchestrator)
workflow.add_node("worker", worker)
workflow.add_node("synthesizer", synthesizer)

workflow.set_entry_point("orchestrator")
workflow.add_edge("orchestrator", "worker")
workflow.add_edge("worker", "synthesizer")
workflow.add_edge("synthesizer", END)

app = workflow.compile()
```

### 4.4 编排器设计模式

#### 4.4.1 中心化编排(Manager Pattern)

```
        ┌──────────┐
        │ Manager  │  ← 唯一编排者
        │ Agent    │
        └────┬─────┘
             │
    ┌────────┼────────┐
    ▼        ▼        ▼
┌──────┐ ┌──────┐ ┌──────┐
│Coder │ │Tester│ │Review│
└──────┘ └──────┘ └──────┘
```

#### 4.4.2 去中心化协作(Decentralized Pattern)

```
┌──────┐ ←────── → ┌──────┐
│Agent │           │Agent │
│  A   │ ←────── → │  B   │
└──────┘           └──────┘
    ↑                  ↑
    └────── ←────── ───┘
       (共享黑板/消息总线)
```

#### 4.4.3 层级编排(Hierarchical Pattern)

```
        ┌────────────┐
        │ Top Manager│  ← 顶层
        └─────┬──────┘
              │
       ┌──────┴──────┐
       ▼             ▼
    ┌──────┐     ┌──────┐
    │ Mgr1 │     │ Mgr2 │  ← 中层
    └──┬───┘     └──┬───┘
       │            │
   ┌───┴───┐    ┌───┴───┐
   ▼       ▼    ▼       ▼
┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│ W1  │ │ W2  │ │ W3  │ │ W4  │  ← 底层 Worker
└─────┘ └─────┘ └─────┘ └─────┘
```

### 4.5 三种编排模式对比

| 模式 | 控制 | 灵活性 | 复杂度 | 典型场景 |
|------|------|--------|--------|----------|
| **中心化** | 集中 | 中 | 低 | 任务明确的小团队 |
| **去中心化** | 分布 | 高 | 高 | 创造性任务、Agent 协作 |
| **层级** | 分层 | 中 | 中 | 大型组织、复杂项目 |

## 五、Chain(推理执行链)

### 5.1 基础定义

**定义:** 多步串行调用,上一步输出作为下一步输入。Chain 是 LangChain 早期的核心概念,后被 LCEL 取代。

### 5.2 LangChain Chain 类型(经典)

| Chain 类型 | 说明 | 用途 |
|-----------|------|------|
| **LLMChain** | 单次 LLM 调用 + Prompt 模板 | 最基础的链 |
| **SequentialChain** | 多个 Chain 顺序执行 | 串行流水线 |
| **RouterChain** | 根据输入选择不同 Chain | 路由分发 |

| **ConversationalRetrievalChain** | 对话 + 检索 | 带历史的 RAG |
| **StuffDocumentsChain** | 文档打包 | 长文本处理 |
| **MapReduceDocumentsChain** | Map-Reduce | 大量文档处理 |

## 六、Plan(任务规划)深度

### 6.1 基础定义

**定义:** 执行前的步骤设计,把复杂任务分解为有序的执行计划。

### 6.2 规划的三个层次

| 层次 | 粒度 | 说明 | 示例 |
|------|------|------|------|
| **战略规划** | 大方向 | 任务目标、整体策略 | "做一个用户认证系统" |
| **战术规划** | 步骤级 | 具体执行步骤 | "1. 设计模型 2. 实现接口 3. 写测试" |
| **操作规划** | 动作级 | 每个工具调用细节 | "调用 edit_file('models/user.py', ...)" |

### 6.3 Plan-and-Execute 范式

**"先思后行"——先制定全局规划,再按计划逐步执行。**

```
[规划阶段 Planner]
   ↓
[计划 Plan]
   1. 步骤 1
   2. 步骤 2
   3. 步骤 3
   ↓
[执行阶段 Executor]
   逐步执行,每步完成后更新状态
   ↓
[最终输出]
```

### 6.4 Plan-and-Execute 代码示例

```python
def plan_and_execute(task: str):
    """先规划后执行"""
    # 规划阶段
    plan_response = llm.invoke(f"""
    把任务拆解为详细步骤,每步包含:
    - step: 步骤描述
    - tool: 使用的工具
    - input: 工具输入
    
    任务:{task}
    
    输出 JSON 数组。
    """)
    plan = parse_json(plan_response)
    
    # 执行阶段
    state = {"done": [], "context": ""}
    for i, step in enumerate(plan):
        result = execute_step(step, state["context"])
        state["done"].append({"step": step, "result": result})
        state["context"] += f"\n步骤 {i+1} 结果:{result}"
        
        # 中途可重新规划
        if need_replan(state):
            new_plan = replan(task, state)
            plan = new_plan
    
    return state["context"]
```

### 6.5 Plan-and-Execute 变体:ReWOO

**ReWOO(Reasoning WithOut Observation):** 规划阶段**不调用工具**,只生成"占位符",执行阶段才真正调用工具。

**论文:** "ReWOO: Decoupling Reasoning from Tools for Large Language Models"(Xu et al., 2023)

```
[规划阶段(无工具调用)]
Plan:
  Step 1: 搜索 X → #E1
  Step 2: 搜索 Y → #E2
  Step 3: 综合 #E1 和 #E2 → #E3

[执行阶段]
  执行 Step 1,填入 #E1
  执行 Step 2,填入 #E2
  Step 3 用 #E1 + #E2 综合

[解决阶段]
  LLM 根据所有证据给出最终答案
```

**优势:**
- 规划阶段 Token 消耗低(不调工具)
- 可并行执行所有独立步骤
- 减少 LLM 上下文往返

### 6.6 Plan-and-Execute 变体:LLMCompiler

**LLMCompiler:** 借鉴传统编译器思想,把 LLM 任务编译为"任务 DAG",并行执行。

**论文:** "An LLM Compiler for Parallel Function Calling"(Kim et al., 2023-12)

```
[Task Planner(任务规划器)]
   生成可并行的任务图(DAG)
   ↓
[Task Fetching Unit(任务获取单元)]
   按依赖顺序获取任务
   ↓
[Task Executor(并行执行器)]
   多线程执行无依赖任务
   ↓
[Joiner(汇总器)]
   汇总结果,判断是否完成
```

**优势:**
- 并行执行多个工具调用
- 显著降低延迟
- 自动处理依赖关系

### 6.7 ReAct vs Plan-Execute vs ReWOO vs LLMCompiler

| 范式 | 规划 | 执行 | 并行 | Token | 适用 |
|------|------|------|------|-------|------|
| **ReAct** | 不预先规划,边想边做 | 单步串行 | 否 | 高 | 探索性任务 |
| **Plan-Execute** | 先全局规划 | 逐步执行 | 否 | 中 | 明确多步任务 |
| **ReWOO** | 规划不调工具 | 占位符填充 | 部分 | 低 | 工具调用密集任务 |
| **LLMCompiler** | 生成 DAG | 并行执行 | 是 | 中 | 可并行多工具任务 |

## 七、Task Decomposition(任务拆解)深度

### 7.1 拆解原则

| 原则 | 说明 |
|------|------|
| **独立性** | 子任务尽可能独立,减少依赖 |
| **可验证** | 每个子任务有明确的完成标准 |
| **合适粒度** | 既不过粗(无法独立完成)也不过细(增加协调成本) |
| **可组合** | 子任务结果可组合为最终结果 |
| **可恢复** | 子任务失败可独立重试,不影响其他 |
| **原子性** | 子任务要么完全成功,要么完全失败 |

### 7.2 拆解方法

| 方法 | 说明 | 适用 |
|------|------|------|
| **按功能拆解** | 按功能模块拆分 | 系统开发 |
| **按步骤拆解** | 按执行顺序拆分 | 流程类任务 |
| **按数据拆解** | 按数据分片拆分 | 数据处理 |
| **按角色拆解** | 按专业角色拆分 | 多领域任务 |
| **按层级拆解** | 递归拆分(树形) | 复杂层级任务 |
| **按时间拆解** | 按时间阶段拆分 | 长周期项目 |

### 7.3 LLM 辅助拆解示例

```
用户任务: 为项目添加用户认证功能

LLM 拆解:
├─ 1. 设计用户模型和数据库表
│   └─ 1.1 设计 User 模型(id, email, password_hash, ...)
│   └─ 1.2 创建迁移文件
│
├─ 2. 实现注册接口
│   ├─ 2.1 /api/register POST 路由
│   ├─ 2.2 密码哈希(bcrypt)
│   └─ 2.3 邮箱验证
│
├─ 3. 实现登录接口
│   ├─ 3.1 /api/login POST 路由
│   ├─ 3.2 JWT 签发
│   └─ 3.3 Refresh Token
│
├─ 4. 添加 JWT 中间件
│   ├─ 4.1 Token 验证
│   └─ 4.2 用户上下文注入
│
├─ 5. 编写测试
│   ├─ 5.1 单元测试(模型、中间件)
│   └─ 5.2 集成测试(注册→登录→访问)
│
└─ 6. 更新文档
    ├─ 6.1 API 文档
    └─ 6.2 README 更新
```

### 7.4 任务拆解的反模式

| 反模式 | 说明 | 后果 |
|--------|------|------|
| **过细拆解** | 子任务太多 | 协调成本超过执行成本 |
| **过粗拆解** | 子任务太大 | 难以独立完成 |
| **强依赖** | 子任务间依赖过多 | 无法并行,失败连锁 |
| **无验收标准** | 不知道何时完成 | 难以判断成功 |
| **拆解后不可恢复** | 失败后无法重试 | 整体流程崩溃 |

## 八、Scheduler(调度器)深度

### 8.1 调度策略

| 策略 | 说明 | 适用 | 优缺点 |
|------|------|------|--------|
| **FIFO** | 先进先出 | 简单任务队列 | 简单,但无优先级 |
| **LIFO** | 后进先出 | 紧急任务优先 | 简单,但可能饿死老任务 |
| **Priority Queue** | 按优先级排序 | 有紧急任务的场景 | 灵活,但需定义优先级 |
| **Round Robin** | 轮询 | 公平调度 | 公平,但延迟高 |
| **依赖调度** | 按依赖关系调度 | 有前置任务的场景 | 保证正确性,复杂 |
| **定时调度** | 按时间触发 | 定时任务 | 可预测,不灵活 |
| **事件驱动** | 按事件触发 | 响应式系统 | 实时,需事件源 |
| **权重调度** | 按权重分配 | 多租户 | 公平,需配权重 |

### 8.2 多 Agent 调度

在多 Agent 系统中,调度器负责:

| 职责 | 说明 |
|------|------|
| **任务分配** | 把任务分配给合适的 Agent |
| **并发控制** | 限制同时运行的 Agent 数 |
| **负载均衡** | 平衡各 Agent 的负载 |
| **超时管理** | 单任务超时强制终止 |
| **重试策略** | 失败任务的自动重试 |
| **优先级调度** | 高优先级任务优先 |
| **资源隔离** | 不同 Agent 资源隔离 |

### 8.3 调度器实现示例

```python
import asyncio
from asyncio import Queue, Task
from typing import Callable

class AgentScheduler:
    def __init__(self, max_concurrency: int = 5):
        self.max_concurrency = max_concurrency
        self.queue = Queue()
        self.agents = {}  # agent_id -> load
    
    async def submit(self, task: dict, priority: int = 0):
        """提交任务"""
        await self.queue.put((priority, task))
    
    async def run(self):
        """调度器主循环"""
        sem = asyncio.Semaphore(self.max_concurrency)
        
        async def execute(task):
            async with sem:
                agent = self.select_agent(task)
                result = await agent.run(task)
                self.update_load(agent, -1)
                return result
        
        while True:
            priority, task = await self.queue.get()
            asyncio.create_task(execute(task))
    
    def select_agent(self, task):
        """负载均衡:选择负载最低的 Agent"""
        return min(self.agents.values(), key=lambda a: a.load)
```

## 九、Trigger(触发器)深度

### 9.1 触发器类型

| 类型 | 说明 | 示例 | 典型工具 |
|------|------|------|----------|

### 9.2 Cron 表达式详解

## 十、Dependency(任务依赖)与 DAG

### 10.1 依赖类型

| 类型 | 说明 | 图示 | 示例 |
|------|------|------|------|
| **串行依赖** | 一个接一个执行 | A → B → C | 编译→打包→部署 |
| **并行依赖** | 多个任务并行,全部完成后继续 | A, B, C 并行 → D | 多模块测试后整合 |
| **条件依赖** | 根据条件选择路径 | A → (条件) → B 或 C | 测试通过→部署,失败→告警 |
| **循环依赖** | 需要避免的反模式 | A → B → A | ❌ 死锁 |
| **数据依赖** | B 需要 A 的输出 | A → 数据 → B | 抽取→分析 |
| **资源依赖** | 共享资源互斥 | A 锁住资源,B 等待 | 数据库锁 |

### 10.2 DAG(有向无环图)

**定义:** Directed Acyclic Graph,任务依赖的常用表示。
- **节点:** 任务
- **边:** 依赖关系
- **无环:** 避免循环依赖

```
        任务A
       /    \
   任务B    任务C
      \      /
       任务D
       /    \
   任务E    任务F
       \    /
        任务G
```

**拓扑排序(Topological Sort):**

对 DAG 节点进行线性排序,保证每个节点的依赖都在它之前。

```
原 DAG:        拓扑排序:
  A             [A, B, C, D, E, F, G]
 / \            或
B   C           [A, C, B, D, F, E, G]
 \ /
  D            (任何满足依赖关系的线性序都合法)
```

### 10.3 Airflow DAG 示例

```python
from airflow import DAG

    dag_id="etl_pipeline",

    extract = PythonOperator(

        python_callable=extract_data
    )
    
    transform = PythonOperator(
        task_id="transform",
```

### 10.4 LangGraph 中的 DAG

LangGraph 用 StateGraph 表示任务图,支持条件边:

```python
from langgraph.graph import StateGraph, END

workflow = StateGraph(State)

# 添加节点
workflow.add_node("extract", extract_fn)
workflow.add_node("transform", transform_fn)
```

### 11.3 Reflexion(反思)机制

**论文:** "Reflexion: Language Agents with Verbal Reinforcement Learning Self-Reflection"(Shinn et al., 2023)

```
[第 1 次尝试]
任务: 写一个快速排序
行动: 生成代码
结果: 测试失败 ❌

[反思]
"我忘记处理空数组的情况"
→ 加入长期记忆

[第 2 次尝试(带反思上下文)]
任务: 写一个快速排序
注意: 上次忘记处理空数组
行动: 生成新代码(含边界处理)
结果: 测试通过 ✓
```

**Reflexion 代码示例:**

```python
def reflexion_loop(task: str, max_attempts: int = 3):
    reflections = []  # 累积反思
    
    for i in range(max_attempts):
        # 带 reflection 的生成
        result = llm.invoke(f"""
        任务:{task}
        
        过去的尝试和反思:
        {chr(10).join(reflections)}
        
        请基于反思改进。
        """)
        
        # 评估
        score = evaluate(result)
        if score >= 0.9:
            return result
        
        # 反思
        reflection = llm.invoke(f"""
        评估这次尝试,分析失败原因,给出改进建议:
        尝试:{result}
        评分:{score}
        """)
        reflections.append(reflection)
    
    return result
```

## 十二、Iteration(迭代推理)深度

### 12.1 迭代推理的价值

- 复杂问题难以一步到位
- 每轮迭代可以修正之前的错误
- 逐步逼近最优解
- 允许"试错"

### 12.2 常见迭代模式

| 模式 | 说明 | 示例 |
|------|------|------|
| **Refinement(细化)** | 逐步细化,从粗到精 | 大纲 → 草稿 → 终稿 |
| **Correction(修正)** | 发现错误后修正 | 跑测试 → 修 bug → 再跑 |
| **Expansion(扩展)** | 逐步扩展内容 | 一段 → 一节 → 一章 |
| **Simplification(简化)** | 逐步简化优化 | 复杂版 → 简化版 → 精炼版 |
| **Self-Consistency(自洽)** | 多次采样取多数 | 多次推理 → 投票 |
| **Debate(辩论)** | 多 Agent 辩论收敛 | 多轮辩论 → 共识 |

### 12.3 迭代终止条件

| 条件 | 说明 |
|------|------|
| **最大迭代次数** | 防止无限循环 |
| **结果满足评估标准** | 质量达标 |
| **连续两轮无改进** | 已收敛 |
| **资源耗尽** | 时间、Token 用完 |
| **用户中止** | 人工干预 |
| **错误模式重复** | 陷入死循环 |

### 12.4 Self-Consistency(自洽性)示例

**论文:** "Self-Consistency Improves Chain of Thought Reasoning in Language Models"(Wang et al., 2022-03)

```
任务: 一个篮子有 5 个苹果,吃了 2 个,买了 3 个,有多少个?

[采样 1] 5 - 2 + 3 = 6
[采样 2] 5 - 2 + 3 = 6
[采样 3] 5 + 3 - 2 = 6
[采样 4] 5 - 2 + 3 = 6
[采样 5] 5 + 3 - 2 = 6

[多数投票] 答案 = 6(5 票)
```

## 十三、三大设计范式对比

### 13.1 ReAct(Reasoning + Acting)

**核心:** LLM 在每一步交替输出"思考"与"行动",基于观察调整下一步。

```
Thought: 我需要先读取文件了解结构
Action: read_file("src/main.py")
Observation: [文件内容]
Thought: 这是一个 FastAPI 应用,需要添加日志中间件
Action: edit_file(...)
Observation: [修改成功]
Thought: 任务完成
Final Answer: 已添加日志中间件
```

**特点:**
- 边想边做,反应式
- 每步有推理,可解释
- 适合探索性、步骤不确定的任务
- **缺点:** 串行,无法并行;Token 消耗大

### 13.2 Plan-and-Execute(规划 + 执行)

**核心:** "先思后行"——先制定全局规划,再按计划逐步执行。

```
[规划阶段]
1. 读取文件结构
2. 识别需要修改的文件
3. 逐个修改
4. 运行测试
5. 提交

[执行阶段]
Step 1: 执行读取...
Step 2: 执行识别...
...
```

**特点:**
- 规划与执行分离
- 全局视角,可控性高
- 适合步骤明确、复杂的多步任务
- **缺点:** 规划可能失效,需要重规划

### 13.3 Multi-Agent(多智能体协作)

**核心:** 多个有不同角色、各有专长的 Agent 互相沟通配合完成任务。

```
[Manager Agent]
    ↓ 分配
[Coder Agent] → [Tester Agent] → [Reviewer Agent]
    ↓                              ↓
    └────────── 汇总 ──────────────┘
                ↓
            最终输出
```

**特点:**
- 角色分工、并行/协作
- 每个 Agent 上下文精简
- 适合超复杂、多领域任务
- **缺点:** 协调复杂,通信开销大

### 13.4 Reflection(反思)

**核心:** 执行后自我评估,失败则调整策略再执行。

```
执行 → 结果 → 自我评估 → 改进 → 再执行
                              ↑      │
                              └──────┘
                              (循环)
```

**特点:**
- 自我改进
- 错误自动修复
- 适合代码、写作
- **缺点:** 多次尝试成本高

### 13.5 四大范式对比表

| 范式 | 决策方式 | 适合任务 | 成本 | 可控性 | 典型代表 |
|------|----------|----------|------|--------|----------|
| **ReAct** | 边走边想 | 探索性 | 高 | 低 | ReAct 论文 |
| **Plan-Execute** | 先全局规划 | 明确多步 | 中 | 高 | LangChain Plan-Execute |
| **Multi-Agent** | 角色分工 | 超复杂 | 高 | 中 | CrewAI、AutoGen |
| **Reflection** | 自我反思 | 代码、写作 | 高 | 中 | Reflexion |

### 13.6 混合范式(现代 Agent 实践)

实际生产中,Agent 常采用混合范式:

```
[Plan-and-Execute 主框架]
    ↓
[每步执行用 ReAct]
    ↓
[失败时触发 Reflection]
    ↓
[复杂子任务委派 Multi-Agent]
    ↓
[结果用 Evaluator-Optimizer 优化]
```

**Claude Code 的实际做法(参考):**
1. 接到任务 → 先制定 TodoList(Plan-Execute)
2. 逐项执行,每步可调用工具(ReAct)
3. 工具失败 → 反思原因,重试(Reflection)
4. 复杂子任务 → 委派给 Subagent(Multi-Agent)
5. 完成后 → 自我校验(Evaluator)

## 十四、状态机(State Machine)视角

### 14.1 任务流程作为状态机

```
┌────────────────────────────────────────────────────┐
│                                                    │
│   ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    │
│   │PENDING│───▶│RUNNING│───▶│SUCCESS│   │FAILED│    │
│   └──────┘    └──┬───┘    └──────┘    └──────┘    │
│                  │                                │
│                  ├─→ PAUSED                       │
│                  │                                │
│                  ├─→ RETRY                        │
│                  │     ↓                          │
│                  │     └→ RUNNING                 │
│                  │                                │
│                  └─→ CANCELED                     │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 14.2 状态转移表

| 当前状态 | 事件 | 目标状态 | 动作 |
|----------|------|----------|------|
| PENDING | start | RUNNING | 启动执行 |
| RUNNING | success | SUCCESS | 返回结果 |
| RUNNING | fail | FAILED | 记录错误 |
| RUNNING | pause | PAUSED | 暂停执行 |
| RUNNING | cancel | CANCELED | 终止执行 |
| FAILED | retry | RUNNING | 重新执行 |
| PAUSED | resume | RUNNING | 恢复执行 |

### 14.3 LangGraph 状态机实现

LangGraph 用 `StateGraph` 显式表示状态机:

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

class TaskState(TypedDict):
    status: str
    result: str
    error: str

def execute(state: TaskState) -> dict:
    try:
        result = do_work(state)
        return {"status": "success", "result": result}
    except Exception as e:
        return {"status": "failed", "error": str(e)}

def retry(state: TaskState) -> dict:
    return {"status": "running"}

def finish(state: TaskState) -> dict:
    return {"status": "done"}

# 构建状态机
workflow = StateGraph(TaskState)
workflow.add_node("execute", execute)
workflow.add_node("retry", retry)
workflow.add_node("finish", finish)

workflow.set_entry_point("execute")

# 条件转移
workflow.add_conditional_edges(
    "execute",
    lambda s: "success" if s["status"] == "success" else "fail",
    {
        "success": "finish",
        "fail": "retry"
    }
)
workflow.add_edge("retry", "execute")
workflow.add_edge("finish", END)

app = workflow.compile()
```

## 十五、并发(Concurrency)与并行

### 15.1 并发 vs 并行

| 概念 | 说明 | 实现 |
|------|------|------|
| **并发(Concurrency)** | 多任务交替执行(看起来同时) | asyncio、协程 |
| **并行(Parallelism)** | 多任务真正同时执行(多核) | multiprocessing、ThreadPool |
| **分布式(Distributed)** | 多机执行 | Celery、Ray |

### 15.2 并发实现方式

| 方式 | 说明 | Python 实现 | 适用 |
|------|------|------------|------|
| **多线程** | 共享内存,受 GIL 限制 | threading | I/O 密集 |
| **多进程** | 独立内存,真正并行 | multiprocessing | CPU 密集 |
| **协程** | 单线程异步,高效 | asyncio | I/O 密集 |
| **线程池** | 固定数量线程 | ThreadPoolExecutor | 混合 |
| **进程池** | 固定数量进程 | ProcessPoolExecutor | CPU 密集 |

### 15.3 并行执行多个 LLM 调用示例

```python
import asyncio
import anthropic

async def call_llm(prompt: str) -> str:
    client = anthropic.AsyncAnthropic()
    response = await client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text

async def parallel_tasks(prompts: list[str]) -> list[str]:
    """并行执行多个 LLM 调用"""
    tasks = [call_llm(p) for p in prompts]
    return await asyncio.gather(*tasks)

# 使用
results = await parallel_tasks([
    "翻译 Hello 到中文",
    "翻译 Hello 到日文",
    "翻译 Hello 到韩文"
])
```

```
最终输出: 完整博客
```

```
    ↓
```

## 十七、最佳实践

### 17.1 Anthropic 官方建议

| 建议 | 说明 |
|------|------|
| **从简单开始** | 先用单次 LLM 调用,不够再加 Workflow,还不够才用 Agent |
| **可预测优先** | 能用 Workflow 就不用 Agent |
| **小模型优先** | 简单任务用 Haiku,复杂任务用 Sonnet/Opus |
| **工具描述清晰** | 工具的 description 要清晰,Schema 要完整 |
| **提供示例** | Few-shot 示例比纯文字描述更有效 |
| **限制迭代** | 设置最大迭代数,防止死循环 |
| **保留人工介入** | 关键决策提供 HITL 接口 |

### 17.2 OpenAI 官方建议

| 建议 | 说明 |
|------|------|
| **明确 Agent 边界** | 不要让 Agent 做超出能力范围的事 |
| **从简单 Agent 开始** | 先做单 Agent,再考虑 Multi-Agent |
| **重视 Orchestration** | 编排层是 Agent 的关键 |
| **Guardrails 必备** | 输入输出都要校验 |
| **可观测性** | Tracing、Logging、Metrics 必备 |
| **失败可恢复** | 设计重试、回滚机制 |

### 17.3 编排设计清单

- [ ] 是否真的需要 Workflow/Agent?(简单 LLM 调用够不够?)
- [ ] 选对模式?(五大模式中选最简单的可行方案)
- [ ] 工具描述清晰?
- [ ] 设置了最大迭代/超时?
- [ ] 关键步骤有 HITL?
- [ ] 有 Tracing?
- [ ] 失败可恢复?
- [ ] 成本可预估?
- [ ] 测试覆盖?
