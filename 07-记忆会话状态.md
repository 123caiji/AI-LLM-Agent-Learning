## 一、Memory(记忆模块)基础

### 1.1 基础定义

**定义:** Agent 的信息存储系统,用于保存和检索知识、历史、状态等信息。

### 1.2 为什么需要记忆

| 原因 | 说明 | 示例 |
|------|------|------|
| **上下文保持** | 记住对话历史,理解指代关系 | "他"指代谁? |
| **知识积累** | 跨会话学习和积累知识 | 用户偏好、项目规范 |
| **状态维护** | 跟踪任务进度和中间结果 | "已完成第 3 步,进行第 4 步" |
| **个性化** | 记住用户偏好和习惯 | "用户偏好 Python 而非 Java" |
| **错误避免** | 记住过去的错误,避免重复 | "上次这样写报错了" |
| **长程推理** | 在多步任务中保持一致性 | 多步代码修改要保持风格一致 |

### 1.3 人类记忆 vs Agent 记忆

| 维度 | 人类记忆 | Agent 记忆 |
|------|----------|------------|
| **类型** | 感觉/短期/长期 | 工作/短期/长期 |
| **存储** | 神经元连接 | 向量数据库/消息列表 |
| **检索** | 联想回忆 | 相似度搜索 |
| **遗忘** | 自然遗忘 | 主动裁剪 |
| **容量** | 有限但高效 | 大容量,但受 Token 限制 |
| **反思** | 自动反思 | 需要显式触发 |

## 三、Working Memory(工作记忆)深度

### 3.1 基础定义

**定义:** Agent 当前任务的临时工作区,记录思考过程、中间结果、计划等。

### 3.2 工作记忆的内容

| 内容 | 说明 | 示例 |
|------|------|------|
| **思考过程** | Thought 内容 | "我需要先读取文件..." |
| **行动计划** | 下一步要做什么 | "1. 读文件 2. 修改 3. 测试" |
| **中间结果** | 工具调用的临时结果 | "文件大小: 1200 行" |
| **任务状态** | 当前任务进度 | "已完成 3/5 步" |
| **当前假设** | 推理中的假设 | "假设用户用的是 Python 3.12" |

### 3.3 ReAct 中的 Scratchpad

在 ReAct 模式中,Scratchpad 记录每一步的 Thought/Action/Observation:

```
[Scratchpad]

Thought 1: 我需要先读取文件了解结构
Action 1: read_file("src/main.py")
Observation 1: [文件内容,1200 行]

Thought 2: 这是一个 FastAPI 应用,需要添加日志中间件
Action 2: read_file("src/middleware.py")
Observation 2: [中间件文件内容]

Thought 3: 在 middleware.py 添加 LoggingMiddleware 类
Action 3: edit_file(...)
Observation 3: [修改成功]

Thought 4: 任务完成
Action 4: Finish("已添加日志中间件")
```

### 3.4 工作记忆的管理

```python
class WorkingMemory:
    def __init__(self):
        self.thoughts = []      # 思考过程
        self.actions = []       # 已执行动作
        self.observations = []  # 观察结果
        self.plan = None        # 当前计划
        self.state = {}         # 临时状态
    
    def add_thought(self, thought: str):
        self.thoughts.append(thought)
    
    def add_action(self, action: dict):
        self.actions.append(action)
    
    def add_observation(self, obs: str):
        self.observations.append(obs)
    
    def get_context(self) -> str:
        """获取工作记忆的上下文"""
        context = ""
        for i, (t, a, o) in enumerate(zip(self.thoughts, self.actions, self.observations)):
            context += f"\nStep {i+1}:\n  Thought: {t}\n  Action: {a}\n  Observation: {o}"
        return context
```

## 四、Short-term Memory(短期记忆)深度

### 4.1 基础定义

**定义:** 当前会话的上下文,通常以消息列表的形式保存在内存中。

### 4.2 短期记忆的内容

| 内容 | 说明 | 示例 |
|------|------|------|
| **用户消息** | 用户的所有输入 | "帮我写个函数" |
| **模型回复** | 模型的所有输出 | "好的,这是函数..." |
| **工具调用** | 工具调用请求和结果 | `read_file("a.py")` → 内容 |
| **系统提示** | 持续生效的系统指令 | "你是 Python 专家" |
| **会话元数据** | 时间戳、用户ID 等 | `{"ts": "2025-01-01", "user": "abc"}` |

### 4.3 短期记忆的限制

| 限制 | 说明 | 应对策略 |
|------|------|----------|
| **上下文窗口限制** | 受模型最大上下文长度限制 | 滑动窗口/摘要 |
| **Token 成本** | 上下文越长,Token 消耗越高 | 压缩历史 |
| **遗忘问题** | 早期消息可能被裁剪掉 | 重要性排序 |
| **检索精度** | 长上下文中关键信息可能被"忽略" | "Lost in the Middle" |

### 4.4 短期记忆的管理策略

#### 4.4.1 滑动窗口(Sliding Window)

```python
def sliding_window(messages: list, max_messages: int = 20):
    """只保留最近 N 条消息"""
    return messages[-max_messages:]
```

**适用场景:** 简单对话,不需要长上下文。

#### 4.4.2 摘要压缩(Summarization)

完整可运行示例(含 tiktoken token 计数):

```python
# 依赖:pip install tiktoken openai
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o-mini")

def count_tokens(messages: list) -> int:
    """粗略统计消息列表的 token 数"""
    return sum(len(enc.encode(m["content"])) + 4 for m in messages)  # +4 角色开销

def summarize_old_messages(messages: list, keep_recent: int = 5,
                           max_tokens: int = 2000):
    """token 超限时,用 LLM 摘要早期消息,保留最近几条原文"""
    if count_tokens(messages) <= max_tokens:
        return messages

    old_messages = messages[:-keep_recent]
    recent_messages = messages[-keep_recent:]

    old_text = "\n".join(f"{m['role']}: {m['content']}" for m in old_messages)
    summary = llm.invoke(
        f"摘要以下对话的关键信息(用户目标、已确认事实、待办事项):\n{old_text}"
    )

    return [
        {"role": "system", "content": f"对话摘要:{summary}"}
    ] + recent_messages
```

**适用场景:** 长对话,需要保留历史关键信息。

**记忆写入策略与遗忘/污染防护(工程要点):**
- **写前判断:** 不是所有对话都值得写入长期记忆,先让 LLM 判断"这是持久事实还是一次性闲聊"(Mem0 等框架内置此步);
- **去重与合并:** 写入前检索已有相似记忆,选择新增/更新/跳过,防止同一事实存 10 份;
- **污染防护:** 用户教给 Agent 的"事实"可能是错的或恶意的,写入时打来源标签,检索时低置信来源降权;
- **主动遗忘:** 支持用户级删除(隐私合规"被遗忘权"),给记忆加时间戳,TTL 过期或长期未命中的冷记忆归档;
- **摘要保真:** 压缩时保留结构化关键信息(路径、ID、决策),避免"细节在摘要中丢失后被模型编造"。

#### 4.4.3 重要性排序

```python
def importance_based_selection(messages: list, max_tokens: int):
    """按重要性保留消息"""
    scored_messages = []
    for msg in messages:
        score = calculate_importance(msg)  # 计算重要性分数
        scored_messages.append((score, msg))
    
    # 按分数排序,选 top N
    scored_messages.sort(key=lambda x: x[0], reverse=True)
    selected = scored_messages[:max_tokens // 1000]  # 粗略估算
    return [msg for _, msg in selected]
```

**适用场景:** 任务型对话,某些消息明显更重要。

#### 4.4.4 分块管理

```python
def topic_based_chunking(messages: list):
    """按主题分块,按需加载"""
    chunks = []
    current_chunk = []
    current_topic = None
    
    for msg in messages:
        topic = detect_topic(msg)  # 检测主题
        if topic != current_topic:
            if current_chunk:
                chunks.append({"topic": current_topic, "messages": current_chunk})
            current_chunk = [msg]
            current_topic = topic
        else:
            current_chunk.append(msg)
    
    return chunks
```

**适用场景:** 知识库对话,多主题切换。

### 4.5 LangChain Memory 实现(已弃用,附 LangGraph 替代)

> ⚠️ **弃用警告(2025 必读):** `langchain.memory` 模块自 LangChain 0.3 起被标记为 **deprecated**,并在 LangChain 1.0(2025-10)中移除。官方迁移方向是 **LangGraph 持久化(Checkpointer / Store)**,见 10.6 / 10.7 节与 10.8 端到端示例。迁移指南:https://python.langchain.com/docs/versions/migrating_memory/
>
> 以下代码仅在 `langchain<1.0`(如 0.3.x)下可运行,**新项目请直接用 LangGraph**。

```python
# 旧方案(仅 langchain 0.3.x,已 deprecated,仅存量代码维护参考)
from langchain.memory import (
    ConversationBufferMemory,
    ConversationBufferWindowMemory,
    ConversationSummaryMemory,
    ConversationSummaryBufferMemory,
    ConversationTokenBufferMemory
)

# 1. 完整保留(适合短对话)
memory = ConversationBufferMemory()

# 2. 滑动窗口(保留最近 K 轮)
memory = ConversationBufferWindowMemory(k=5)

# 3. 摘要(对历史做摘要)
memory = ConversationSummaryMemory(llm=llm)

# 4. 摘要 + 缓冲(混合)
memory = ConversationSummaryBufferMemory(
    llm=llm,
    max_token_limit=2000
)

# 5. Token 限制(按 Token 数裁剪)
memory = ConversationTokenBufferMemory(
    llm=llm,
    max_token_limit=1000
)
```

**新方案(LangGraph,官方推荐):** 会话历史由 Checkpointer 按 `thread_id` 自动持久化,裁剪用 `trim_messages`:

```python
# pip install langgraph langchain-core(示例基于 langgraph 0.2+/1.x)
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langchain_core.messages import trim_messages

def call_model(state: MessagesState):
    # 只把最近 10 条消息传给 LLM,完整历史仍由 checkpointer 保存
    recent = trim_messages(state["messages"], strategy="last",
                           max_tokens=10, token_counter=len)
    return {"messages": [llm.invoke(recent)]}

builder = StateGraph(MessagesState)
builder.add_node("chat", call_model)
builder.add_edge(START, "chat")
builder.add_edge("chat", END)

graph = builder.compile(checkpointer=InMemorySaver())
config = {"configurable": {"thread_id": "session-1"}}  # 同一 thread_id = 同一会话
graph.invoke({"messages": [("user", "你好")]}, config)
graph.invoke({"messages": [("user", "我刚才说了什么?")]}, config)  # 记得上文
```

### 4.6 各策略对比

| 策略 | 实现复杂度 | Token 节省 | 信息保留 | 适用场景 |
|------|-----------|-----------|----------|----------|
| **完整保留** | 极低 | 0 | 100% | 短对话 |
| **滑动窗口** | 低 | 高 | 中(丢失早期) | 简单对话 |
| **摘要压缩** | 中(需 LLM) | 中 | 中(摘要损失) | 长对话 |
| **重要性排序** | 高(需打分) | 中 | 高(保留关键) | 任务型对话 |
| **分块管理** | 高(需分块) | 高 | 高(按需加载) | 知识库对话 |
| **Token 缓冲** | 低 | 中 | 中 | 成本敏感 |

## 五、Long-term Memory(长期记忆)深度

### 5.1 基础定义

**定义:** 跨会话的知识存储,通常使用向量数据库 + RAG 技术实现。

### 5.2 长期记忆的特点

| 特点 | 说明 |
|------|------|
| **持久性** | 会话结束后仍然保留 |
| **大容量** | 可以存储大量知识(TB 级) |
| **检索式** | 通过相似度搜索获取相关信息 |
| **可积累** | 可以不断添加新的知识 |
| **可更新** | 旧知识可以被更新或删除 |
| **可分类** | 不同类型知识可分类存储 |

### 5.3 长期记忆的分类(类比人类)

| 类型 | 说明 | 示例 | 实现 |
|------|------|------|------|
| **Episodic Memory(情节记忆)** | 记录具体事件和经历 | "上次用户让我改 bug" | 向量数据库 |
| **Semantic Memory(语义记忆)** | 记录概念和事实的知识 | "Python 用缩进表示代码块" | 知识库 |
| **Procedural Memory(程序记忆)** | 记录如何做事的技能 | "如何调试代码" | Skills/规则 |

### 5.4 长期记忆的实现方式

| 实现方式 | 说明 | 优缺点 | 典型工具 |
|----------|------|--------|----------|
| **向量数据库** | 文本向量化后存储,相似度检索 | 灵活、语义检索,但有精度损失 | Pinecone、Milvus、Qdrant |
| **关系数据库** | 结构化存储,SQL 查询 | 精确查询,但不支持语义检索 | PostgreSQL、MySQL |
| **图数据库** | 实体-关系图结构 | 关系推理强,但构建成本高 | Neo4j、ArangoDB |
| **文档数据库** | JSON 文档存储 | 灵活,检索能力中等 | MongoDB、CouchDB |
| **文件系统** | 直接存储文档文件 | 简单直观,但检索效率低 | 本地文件系统 |
| **混合存储** | 多种存储组合 | 各取所长,复杂 | 向量库 + 关系库 |

### 5.5 向量数据库详解

#### 5.5.1 向量数据库工作流程

```
[写入]
原始文本 → Embedding 模型 → 向量 → 存入向量数据库
                                  ↓
                              元数据(text, id, source, ...)

[查询]
查询文本 → Embedding 模型 → 查询向量 → 相似度搜索 → Top-K 结果
                                                      ↓
                                                  原始文本
```

#### 5.5.2 主流向量数据库(精简版)

| 数据库 | 类型 | 一句话定位 |
|--------|------|-----------|
| **Pinecone** | 云服务 | 全托管,免运维,企业首选 |
| **Milvus** | 开源 | 大规模生产级,云原生 |
| **Qdrant** | 开源 + 云 | Rust 高性能,过滤能力强 |
| **Chroma** | 开源 | 轻量原型,Python 友好 |
| **pgvector** | PG 扩展 | 已用 PostgreSQL 的项目 |
| **FAISS** | 库(Meta) | 嵌入式,无服务端 |

> 📖 完整对比(含 Weaviate、LanceDB、Vespa、ES 等 12 个数据库的选型决策树与代码示例)详见《08-RAG知识库体系》**8.4 / 8.5 节**,此处不再重复。

#### 5.5.3 相似度搜索算法

| 算法 | 说明 | 精度 | 速度 |
|------|------|------|------|
| **余弦相似度(Cosine)** | 向量夹角 | 高 | 中 |
| **欧氏距离(Euclidean)** | 直线距离 | 高 | 中 |
| **内积(Inner Product)** | 点积 | 中 | 快 |
| **HNSW** | 图索引近似 | 高(近似,典型召回 >95%) | 极快 |
| **IVF** | 倒排索引 | 中 | 快 |
| **PQ** | 乘积量化 | 低 | 极快(压缩) |

### 5.8 记忆框架选型:Mem0 / Letta / Zep(2024-2025 生产级)

论文(MemGPT、Generative Agents)给出了思想,但落地时通常直接采用成熟的**记忆层(Memory Layer)框架**。三者是 2024-2025 年长期记忆的事实标准:

| 维度 | **Mem0** | **Letta(MemGPT 商业化)** | **Zep** |
|------|----------|--------------------------|---------|
| **定位** | 轻量"记忆中间件",嵌入现有 LLM 应用 | 完整 Agent 运行时 + 自管理记忆(OS 思想) | 时序知识图谱记忆引擎,面向企业对话 |
| **核心机制** | 从对话抽取事实 → 向量化存储 → 相似度检索 | Main/External Context 分页调度,LLM 自主管理 | 对话构建时序知识图谱(Graphiti),实体+关系+时间 |
| **部署** | 开源库 / 托管云服务 / 自托管 | 开源自托管(letta server)/ Letta Cloud | 开源(Graphiti)/ Zep Cloud 托管 |
| **API 风格** | `add()` / `search()` 两个核心方法 | 创建有状态 Agent,REST/SDK 交互 | `memory.add()` / `memory.get()` + 图谱查询 |
| **适用** | 快速给现有应用加用户偏好记忆 | 需要完整 Agent 生命周期管理 | 长对话历史、关系演变、时效性敏感场景 |
| **链接** | https://github.com/mem0ai/mem0 | https://github.com/letta-ai/letta | https://github.com/getzep/graphiti |

```
[Context Window = 200k tokens]
├─ System Prompt     (1k tokens)

├─ Assistant Reply 1 (500 tokens)
├─ User Message 2    (100 tokens)
├─ Assistant Reply 2 (200 tokens)
├─ ...
├─ User Message N    (50 tokens)
```

### 6.5 "Lost in the Middle" 问题

**论文:** "Lost in the Middle: How Language Models Use Long Contexts"(Liu et al., 2023-07)
https://arxiv.org/abs/2307.03172

**核心发现:** LLM 在长上下文中,对**中间**位置的信息利用率最低,**开头和结尾**的信息利用率最高。

```
信息利用率:
高 ──────────────────────────────────────── 低
│                                            │
├─ 开头 ──────── 中间 ──────── 结尾 ──┤
   高              低              高
```

**应对策略:**
- 关键信息放开头或结尾
- 用 RAG 检索最相关内容,而非塞入所有内容
- 长文档先做摘要

```
┌─────────────────────────────────────────┐

│                                         │

│    │  User → Assistant    │             │
```

### 7.5 会话的存储

```python
# 会话数据结构
class Session:
    def __init__(self, session_id: str, user_id: str):
        self.session_id = session_id
        self.user_id = user_id
        self.created_at = datetime.now()
        self.messages = []  # 消息列表
        self.state = {}     # 会话状态
        self.metadata = {}  # 元数据
    
    def add_message(self, role: str, content: str):
        self.messages.append({
            "role": role,
            "content": content,
            "timestamp": datetime.now()
        })
    
    def save(self, storage):
        storage.save(self.session_id, self.to_dict())
    
    @classmethod
    def load(cls, session_id: str, storage):
        data = storage.load(session_id)
        return cls.from_dict(data)
```

## 十、State Machine(状态机)深度

### 10.1 基础定义

**定义:** 显式状态转移模型,定义状态之间的转换规则。

### 10.2 为什么用状态机

| 优势 | 说明 |
|------|------|
| **可控性** | 状态转换有明确规则 |
| **可预测性** | 知道每个状态下会发生什么 |
| **可调试性** | 状态清晰,便于排查问题 |
| **可维护性** | 状态转换逻辑集中管理 |
| **可视化** | 状态图直观 |
| **可测试** | 状态转移可单元测试 |

### 10.3 状态机的要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **State(状态)** | 系统当前所处的模式 | Idle/Thinking/Done |
| **Event(事件)** | 触发状态转换的动作 | user_input/tool_result |
| **Transition(转移)** | 从一个状态到另一个状态 | Idle → Thinking |
| **Action(动作)** | 状态转换时执行的操作 | 调用 LLM |
| **Guard(守卫)** | 转移条件 | if has_tool: → Acting |

### 10.4 状态转移表

| 当前状态 | 事件 | 目标状态 | 动作 |
|----------|------|----------|------|
| Idle | user_input | Thinking | 调用 LLM |
| Thinking | need_tool | ToolCalling | 调用工具 |
| Thinking | no_tool | Done | 返回答案 |
| ToolCalling | tool_result | Thinking | 处理结果 |
| ToolCalling | tool_error | Error | 记录错误 |
| Error | retry | Thinking | 重新调用 |
| Done | user_input | Thinking | 新一轮 |
| Any | timeout | Error | 超时处理 |

### 10.5 LangGraph 状态机

LangGraph 是基于状态机的 Agent 框架:
- **节点(Node):** 状态和操作
- **边(Edge):** 状态转换
- **状态(State):** 共享的数据结构
- **条件边:** 根据条件选择下一个状态

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

class AgentState(TypedDict):
    messages: list
    current_task: str
    tool_results: list
    final_answer: str

def think(state: AgentState) -> dict:
    """思考节点"""
    response = llm.invoke(state["messages"])
    return {"messages": state["messages"] + [response]}

def act(state: AgentState) -> dict:
    """行动节点"""
    tool_result = execute_tool(state)
    return {"tool_results": state["tool_results"] + [tool_result]}

def should_continue(state: AgentState) -> str:
    """条件转移"""
    if state["messages"][-1].get("tool_calls"):
        return "act"
    return "end"

# 构建状态机
workflow = StateGraph(AgentState)
workflow.add_node("think", think)
workflow.add_node("act", act)

workflow.set_entry_point("think")

workflow.add_conditional_edges(
    "think",
    should_continue,
    {"act": "act", "end": END}
)
workflow.add_edge("act", "think")  # 行动后回到思考

app = workflow.compile()
```

**官方文档:** https://langchain-ai.github.io/langgraph/

### 10.7 LangGraph Store:跨会话长期记忆

Checkpointer 按 `thread_id` 隔离会话,**跨会话共享的长期记忆**用 Store(`BaseStore`):一个键值存储,按 namespace(如 `("memories", user_id)`)组织,支持语义检索(配置 embedding 后)。

```python
# 依赖:pip install langgraph

store = InMemoryStore()

# 写入:namespace + key + value(JSON)
store.put(("memories", "u001"), "pref-lang",
          {"content": "用户偏好 Python", "type": "preference"})

# 精确读取
item = store.get(("memories", "u001"), "pref-lang")

# 列出某用户全部记忆
items = store.search(("memories", "u001"))

# 语义检索(需在 InMemoryStore(index={"embed": ..., "dims": 1536}) 配置嵌入)
# results = store.search(("memories", "u001"), query="编程语言偏好")
```

**与 Checkpointer 的分工:** Checkpointer = 短期记忆(会话内对话状态,按 thread_id);Store = 长期记忆(跨会话用户知识,按 user_id)。两者可同时在 `compile()` 时挂载:`graph = builder.compile(checkpointer=..., store=...)`,节点内通过 `store` 参数读写。官方文档:https://langchain-ai.github.io/langgraph/concepts/memory/

## 十一、Scratchpad(工作记忆区)深度

### 11.1 基础定义

**定义:** Agent 的临时工作区,用于记录思考过程、中间结果、计划等。

### 11.2 Scratchpad vs 对话历史

| 维度 | Scratchpad | 对话历史 |
|------|-----------|----------|
| **内容** | 思考过程、计划 | 用户消息、模型回复 |
| **可见性** | 通常对用户不可见 | 对用户可见 |
| **持久性** | 任务结束清空 | 会话内保留 |
| **用途** | 临时计算 | 上下文记忆 |

### 11.3 Scratchpad 在 ReAct 中的角色

```
[Scratchpad 内容]

Question: 用户的问题

Thought 1: 第一步推理
Action 1: 工具调用 1
Observation 1: 结果 1

Thought 2: 第二步推理
Action 2: 工具调用 2
Observation 2: 结果 2

...

Thought N: 最终推理
Action N: Finish(答案)
```

### 11.4 Scratchpad 的实现

```python
class Scratchpad:
    def __init__(self):
        self.entries = []
    
    def add(self, thought: str, action: dict, observation: str):
        self.entries.append({
            "thought": thought,
            "action": action,
            "observation": observation,
            "timestamp": datetime.now()
        })
    
    def to_prompt(self) -> str:
        """转为 LLM 提示"""
        prompt = ""
        for i, entry in enumerate(self.entries, 1):
            prompt += f"\nThought {i}: {entry['thought']}"
            prompt += f"\nAction {i}: {entry['action']}"
            prompt += f"\nObservation {i}: {entry['observation']}\n"
        return prompt
    
    def clear(self):
        self.entries = []
```

## 十二、MemGPT:无限上下文

### 12.1 基础定义

**论文:** "MemGPT: Towards LLMs as Operating Systems"(Packer et al., 2023-10)
https://arxiv.org/abs/2310.08560

**核心思想:** 把 LLM 上下文管理类比为操作系统的内存管理,实现"无限"上下文。

### 12.2 MemGPT 架构

```
┌─────────────────────────────────────────┐
│            MemGPT Architecture          │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │  Main Context(主上下文)        │    │
│  │  - System Instructions          │    │  ← LLM 直接访问
│  │  - Working Context(工作区)     │    │
│  │  - FIFO Queue(最近消息)        │    │
│  └─────────────────────────────────┘    │
│              ↑ ↓ (页面调度)             │
│  ┌─────────────────────────────────┐    │
│  │  External Context(外部上下文)  │    │
│  │  - 长期记忆                     │    │  ← 需要主动检索
│  │  - 历史会话                     │    │
│  │  - 知识库                       │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 12.3 类比操作系统

| OS 概念 | MemGPT 对应 | 说明 |
|---------|-------------|------|
| **RAM** | Main Context | LLM 直接访问 |
| **Disk** | External Context | 需要主动加载 |
| **Page Fault** | 上下文溢出 | 触发页面调度 |
| **Page In** | 加载到主上下文 | 从外部拉取 |
| **Page Out** | 移出主上下文 | 保存到外部 |
| **System Call** | Function Call | LLM 主动调用 |

### 12.4 MemGPT 的工作流程

```
1. 用户消息 → 进入 FIFO Queue
2. LLM 处理 Main Context
3. 如果 Main Context 满:
   a. 把早期消息 Page Out 到 External Context
   b. 摘要后保留在 Working Context
4. 如果需要历史信息:
   a. LLM 主动调用 search_memory()
   b. 从 External Context 检索
   c. Page In 到 Main Context
5. LLM 生成回复
```

### 12.5 MemGPT 的价值

| 价值 | 说明 |
|------|------|
| **突破上下文限制** | 理论上无限上下文 |
| **自主管理** | LLM 自己决定何时检索 |
| **保留关键信息** | 通过摘要保留关键信息 |
| **类比 OS** | 借鉴成熟 OS 思想 |

### 15.2 上下文管理策略

#### 15.2.1 滑动窗口

```python
def sliding_window(messages, max_tokens=100000):
    """保留最近 max_tokens 的消息"""
    total = 0
    kept = []
    for msg in reversed(messages):
        msg_tokens = count_tokens(msg)
        if total + msg_tokens > max_tokens:
            break
        kept.insert(0, msg)
        total += msg_tokens
    return kept
```

#### 15.2.2 摘要 + 最近

```python
def summary_plus_recent(messages, recent_count=10):
    """早期摘要 + 最近几条"""
    if len(messages) <= recent_count:
        return messages
    
    old = messages[:-recent_count]
    recent = messages[-recent_count:]
    
    summary = llm.summarize(old)
    return [
        {"role": "system", "content": f"历史摘要:{summary}"}
    ] + recent
```

#### 15.2.3 选择性保留

```python
def selective_retention(messages, max_tokens=50000):
    """选择性保留重要消息"""
    scored = [(importance(m), m) for m in messages]
    scored.sort(key=lambda x: x[0], reverse=True)
    
    kept = []
    total = 0
    for score, msg in scored:
        msg_tokens = count_tokens(msg)
        if total + msg_tokens > max_tokens:
            continue
        kept.append(msg)
        total += msg_tokens
    
    # 按时间排序
    kept.sort(key=lambda m: m.get("timestamp", 0))
    return kept
```

### 15.4 Context Engineering 与上下文压缩(2025 通用话语)

2025 年起,业界把"给模型准备上下文的一切工作"(系统提示、记忆、RAG、工具结果、历史裁剪的总和)统称为 **Context Engineering(上下文工程)**——它取代了狭义的"记忆管理",成为 Agent 工程的核心议题。一句话:**模型效果 ≈ f(模型能力, 上下文质量)**,而上下文质量是你唯一能工程化的部分。

**Context Engineering 的组成部分:**

```
Context = System Prompt + 长期记忆检索结果 + RAG 文档
        + 对话历史(裁剪/压缩后) + 工具调用结果 + 当前输入
目标:在固定窗口内,让每一 token 都尽可能"有用"
```

**上下文压缩(Compaction)实操:**

1. **Claude Code `/compact`:** 手动触发把长会话历史压缩成摘要继续工作;`/compact 重点保留数据库相关决策` 可带指令定制保留重点。长任务接近上下文上限时也会自动压缩(auto-compact)。
2. **手写压缩策略(通用实现):** 超过阈值时用 LLM 摘要早期消息,保留最近 N 条原文:

```python
def compress_context(messages, max_tokens=8000, keep_recent=6):
    """超过 token 阈值时,摘要早期消息 + 保留最近原文"""
    if count_tokens(messages) <= max_tokens:
        return messages
    old, recent = messages[:-keep_recent], messages[-keep_recent:]
    summary = llm.invoke(
        "把以下对话压缩为 200 字摘要,保留:用户目标、已确认事实、"
        "关键决策、未完成事项。\n\n" + format_messages(old)
    )
    return [{"role": "system", "content": f"[前期对话摘要]{summary}"}] + recent
```

3. **分层压缩:** 工具结果(如文件全文、命令输出)优先压缩为要点,用户消息与助手结论尽量保留原文——工具结果是上下文膨胀的最大来源。
4. **压缩≠无损:** 摘要会丢细节,关键数据(路径、ID、配置值)应转入结构化 State / 长期记忆,而非依赖摘要保留。

## 十八、易混淆概念对比

| 维度 | Working Memory | Short-term Memory |

| **内容** | 当前任务的思考 | 对话历史 |
| **生命周期** | 任务内 | 会话内 |
| **用途** | 临时计算 | 上下文 |

## 十九、最佳实践

### 19.1 记忆系统设计原则

| 原则 | 说明 |
|------|------|
| **分层存储** | 工作/短期/长期三层分明 |
| **按需检索** | 不一次性加载所有记忆 |
| **主动遗忘** | 定期清理不重要信息 |
| **重要性打分** | 重要信息优先保留 |
| **可恢复** | 关键状态可恢复 |
| **可观测** | 记忆操作可追踪 |

### 19.2 上下文管理清单

- [ ] 是否设置了上下文窗口限制?
- [ ] 是否实现了历史裁剪策略?
- [ ] 是否考虑了 Token 成本?
- [ ] 关键信息是否放开头/结尾?
- [ ] 是否使用 Prompt Caching?
- [ ] 是否提供会话保存/恢复?
- [ ] 是否有长期记忆存储?
- [ ] 是否有 RAG 检索机制?

### 19.3 Anthropic 官方建议

| 建议 | 说明 |
|------|------|
| **充分利用长上下文** | Claude 200k,可以塞入很多内容 |
| **用 Prompt Caching** | 长系统提示缓存,省钱 |
| **关键信息放开头/结尾** | 避免 "Lost in the Middle" |
| **用 RAG 而非塞入** | 大量知识用 RAG 检索 |
