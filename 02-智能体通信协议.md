# 02 - 智能体通信协议(Agent Protocol)

> 让 Agent 与外部世界 / 其他 Agent 通信的标准化协议。涵盖 MCP(Agent↔工具)、A2A(Agent↔Agent)、Function Calling、消息队列、Handoff 任务移交等核心概念。本章是整个 AI Agent 体系的"通信枢纽",理解协议 = 理解 Agent 之间如何"对话"。

## 一、MCP(模型上下文协议)

### 1.0 基础定义

**全称:** Model Context Protocol(模型上下文协议)

**官方定义(2025-06-18 规范):**
> MCP is an open-source standard for connecting AI applications to external systems. Using MCP, AI applications like Claude or ChatGPT can connect to data sources, tools and workflows—enabling them to access key information and perform tasks.

**发布方:** Anthropic,2024 年 11 月 25 日开源

**定位:** 被誉为"AI 应用的 USB-C 接口",把 M × N 集成问题降为 M + N。

**官方资源:**
- 规范文档:https://spec.modelcontextprotocol.io/
- 官方主页:https://modelcontextprotocol.io
- GitHub 组织:https://github.com/modelcontextprotocol
- 官方 Server 仓库:https://github.com/modelcontextprotocol/servers

### 1.1 为什么需要 MCP

#### 1.1.1 传统痛点(M × N 问题)

```
传统集成方式:
        OpenAI FC   自定义JSON   REST API   GraphQL   ...
App1 ────┬───────────┬───────────┬─────────┬─────
App2 ────┼───────────┼───────────┼─────────┼─────   ← M 个应用
App3 ────┼───────────┼───────────┼─────────┼─────
  ↓        ↓           ↓           ↓         ↓
Tool1   Tool2       Tool3       Tool4    Tool5     ← N 个工具

集成代码量 = M × N 套
```

**具体问题:**
- 每个 LLM 应用对接 N 个工具,需要 N 套集成代码
- 工具接入方式各不相同(OpenAI Function Calling、自定义 JSON、REST API)
- 切换 LLM 供应商时,所有工具集成都要重写
- 工具方要为每个 LLM 平台单独开发适配

#### 1.1.2 MCP 的解决方案(M + N 问题)

```
MCP 方式:
        ┌─────────────┐
App1 ───┤             │
App2 ───┤  MCP 协议   ├─── Tool1 (MCP Server)
App3 ───┤             ├─── Tool2 (MCP Server)
        └─────────────┤─── Tool3 (MCP Server)
                        └── Tool4 (MCP Server)

集成代码量 = M + N 套
```

- 提供统一的 Client/Server 协议
- 工具方写一次 MCP Server,所有支持 MCP 的 Host 都能用
- 把 M × N 问题降为 M + N 问题
- 协议层屏蔽 LLM 供应商差异

#### 1.1.3 生态支持现状(2026 年)

| 类别 | 支持方 |
|------|--------|
| **AI 助手** | Claude Desktop、ChatGPT(OpenAI 2025-03 宣布支持) |
| **开发工具** | VS Code、Cursor、Windsurf、Trae、Zed、JetBrains |
| **CLI Agent** | Claude Code、Codex CLI、Gemini CLI |
| **企业平台** | Cloudflare、Vercel、Stripe、Block |
| **官方 Server** | 7 个参考实现(filesystem/git/fetch/memory/time 等,2025 年大整理后) |
| **社区 Server** | 数千个(mcp.so、Glama、PulseMCP 等聚合) |

### 1.2 双层架构(2025-06-18 规范关键概念)

```
┌─────────────────────────────────────────────────┐
│  Transport Layer(传输层,外层)                  │
│  ┌───────────────────────────────────────────┐  │
│  │  Data Layer(数据层,内层)                 │  │
│  │  - JSON-RPC 2.0 消息                      │  │
│  │  - 生命周期管理(Initialize/Shutdown)     │  │
│  │  - Primitives 原语(Tools/Resources/...)  │  │
│  │  - 能力协商(Capabilities)                │  │
│  └───────────────────────────────────────────┘  │
│  - 连接建立、消息分帧、授权(OAuth 2.1)         │
│  - 多种传输机制可替换(stdio/Streamable HTTP)  │
└─────────────────────────────────────────────────┘
```

| 层级 | 职责 | 内容 |
|------|------|------|
| **Data Layer(数据层)** | 定义 JSON-RPC 协议、原语、生命周期 | Tools / Resources / Prompts / Sampling / Elicitation / Tasks / Lifecycle / Notifications |
| **Transport Layer(传输层)** | 管理通信通道与认证 | stdio / Streamable HTTP / OAuth 2.1 |

**设计意义:** 同一套 JSON-RPC 2.0 消息可在任意传输机制上运行,实现传输层与协议层解耦。比如同一个 Server 既能用 stdio 本地运行,也能用 Streamable HTTP 远程部署,业务逻辑不变。

### 1.3 三角参与者(Participants)

```
┌──────────────────┐
│    MCP Host      │  ← 承载 LLM 的应用(Claude Desktop / Codex CLI / Cursor / VS Code)
│  ┌────────────┐  │
│  │    LLM     │  │
│  └─────┬──────┘  │
│        │         │
│  ┌─────▼──────┐  │
│  │ Host Logic │  │  ← 用户授权、UI、路由
│  └─────┬──────┘  │
└────────┼─────────┘
         │ 内部实例化(1 个 Server = 1 个 Client)
   ┌─────┴───────────┬─────────────────┐
   │                 │                 │
┌──▼───────┐   ┌─────▼────┐    ┌──────▼──────┐
│ Client 1 │   │ Client 2 │    │ Client 3    │
└──┬───────┘   └─────┬────┘    └──────┬──────┘
   │ stdio            │ stdio           │ Streamable HTTP
   │                  │                 │ + OAuth 2.1
┌──▼───────┐   ┌─────▼────┐    ┌──────▼──────┐
│ Server A │   │ Server B │    │ Server C    │
│Filesystem│   │ Database │    │ Sentry(远程)│
└──────────┘   └──────────┘    └─────────────┘
```

| 角色 | 职责 | 举例 |
|------|------|------|
| **Host** | 运行 LLM,管理 Client 生命周期,处理用户授权 | Claude Desktop、Codex CLI、Cursor、VS Code |
| **Client** | 与单个 Server 建立 1:1 专用连接,转发请求/响应 | Host 内部的协议适配层(对用户透明) |
| **Server** | 暴露 Resources / Tools / Prompts 三类能力 | filesystem、sqlite、github、Sentry |

**连接特性:**
- **本地 stdio Server:** 通常服务**单个** Client(1:1),进程隔离
- **远程 Streamable HTTP Server:** 通常服务**多个** Client(1:N),可水平扩展

**安全隔离原则:**
- Host 必须显式获得用户授权才能调用任何 Tool
- Host 必须控制 Server 的访问边界(Roots)
- Host 必须对 Server 返回内容进行审计

### 1.4 Primitives(原语)完整列表

> **原语(Primitive):** MCP 协议预定义的标准操作类型,每个原语有对应的方法名(JSON-RPC method)。

#### 1.4.1 Server 端原语(Client → Server 方向,Server 实现、Client 调用)

| 原语 | 中文 | 控制方 | 用途 | 类比 | 关键方法 |
|------|------|--------|------|------|----------|
| **Tools** | 工具 | 模型主导(LLM 自主调用) | 可执行操作(写文件、API 调用、DB 查询) | POST 请求 | `tools/list`、`tools/call` |
| **Resources** | 资源 | 用户主导(应用控制) | 只读上下文数据(文件、DB 行、API 响应) | GET 请求 | `resources/list`、`resources/read`、`resources/subscribe` |
| **Prompts** | 提示词模板 | 用户主导 | 预定义交互模板(系统提示、Few-shot) | 函数模板 | `prompts/list`、`prompts/get` |

> **控制方差异是关键:** Resources 和 Prompts 由用户主动选择触发,Tools 由 LLM 自主决定调用。这避免了 LLM 自主读取敏感资源或自主注入提示词的风险。

#### 1.4.2 Client 端原语(Server → Client 方向,Server 发起、Client 实现)

> **方向注意:** 这组原语由 **Server 主动发起请求**、Client(及其背后的 Host/用户)负责实现响应,与 §1.4.1 的调用方向相反。

| 原语 | 用途 | 调用方法 | 说明 |
|------|------|----------|------|
| **Sampling** | Server 请求 Client 的 LLM 完成推理 | `sampling/createMessage` | 模型无关,Server 不直接访问 LLM |
| **Elicitation** | Server 向用户请求额外信息或操作确认 | `elicitation/create` | 2025-06-18 新增,如询问"请输入数据库密码" |
| **Logging** | Server 向 Client 发送日志便于调试 | `notifications/message` | 多级别日志(DEBUG/INFO/WARN/ERROR) |
| **Roots** | Server 询问 Client 可访问的文件系统边界 | `roots/list` | 安全机制,Server 不得越界 |

#### 1.4.3 实用工具原语(Experimental)

| 原语 | 用途 |
|------|------|
| **Tasks**(实验性) | 持久化执行包装,支持延迟结果获取与状态追踪(批量处理、长任务、工作流) |

每个原语都有对应方法:`*/list`(发现)、`*/get`(获取)、`tools/call`(执行)。Client 先 `tools/list` 发现工具,再 `tools/call` 执行。

#### 1.4.4 原语控制流总览

```
┌─────────────────────────────────────────────────────────┐
│                       MCP Host                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────┐    │
│  │   LLM    │   │   User   │   │  Host Application │    │
│  └────┬─────┘   └────┬─────┘   └────────┬─────────┘    │
│       │              │                   │              │
│       │ 自主调用      │ 用户触发           │ 应用控制       │
│       ▼              ▼                   ▼              │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Tools       Resources    Prompts   Roots       │    │
│  │  (LLM控)    (用户控)     (用户控)  (Host控)     │    │
│  └────────────────────┬────────────────────────────┘    │
│                       │                                 │
└───────────────────────┼─────────────────────────────────┘
                        │
                        │ JSON-RPC 2.0
                        ▼
              ┌──────────────────┐
              │   MCP Server     │
              │  ┌────────────┐  │
              │  │  Tools     │  │  ← Server 实现
              │  │  Resources │  │
              │  │  Prompts   │  │
              │  └────────────┘  │
              └──────────────────┘
                        │
                        │ 反向请求(可选)
                        ▼
              Sampling / Elicitation / Logging
              (Server → Client 反向调用)
```

### 1.5 通信协议

#### 1.5.1 底层消息格式:JSON-RPC 2.0

**JSON-RPC 2.0 三种消息类型:**

```json
// 1. Request(请求,带 id,需响应)
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "read_file", "arguments": { "path": "/tmp/test.txt" } }
}

// 2. Response(响应,带相同 id)
// 2.1 成功响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "content": [{ "type": "text", "text": "文件内容..." }] }
}

// 2.2 错误响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": { "code": -32601, "message": "Method not found" }
}

// 3. Notification(通知,无 id,无需响应)
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
```

#### 1.5.2 两种传输方式对比

| 传输方式 | 适用场景 | 特点 | 安全 |
|----------|----------|------|------|
| **stdio** | 本地 Server | 进程间通信,无网络开销,性能最优 | 进程隔离,信任边界清晰 |
| **Streamable HTTP** | 远程 Server | HTTP POST + 可选 SSE 流式;支持 Bearer Token / API Key / OAuth | HTTPS + OAuth 2.1 |

#### 1.5.3 Streamable HTTP 详解(2025-03-26 取代旧 HTTP+SSE)

**旧方案(2024-11):** 两个端点
- POST `/mcp` - Client → Server 请求
- GET `/mcp/sse` - Server → Client 流式推送(长连接),首帧为 `event: endpoint` 告知回发地址

**新方案(2025-03-26+):** 单一端点
- POST `/mcp` - 双向通信
- 响应可以是普通 JSON,也可以是 `Content-Type: text/event-stream`(SSE 流)
- 服务端**可选**通过 `Mcp-Session-Id` 头维护会话状态——会话管理是可选的,无状态部署(每个请求自包含上下文)自 2025-03-26 起即可行,可直接上 Serverless

```
Client                              Server
  │                                   │
  │  POST /mcp                        │
  │  Mcp-Session-Id: session-xxx      │
  │  Accept: application/json,        │
  │          text/event-stream        │
  │ ─────────────────────────────────→│
  │                                   │
  │  HTTP 200 OK                      │
  │  Content-Type: text/event-stream  │
  │  ←─────────┬─────────────────────│
  │            │ event: message       │
  │            │ data: {...}          │
  │            │ event: message       │
  │            │ data: {...}          │
  │  ←─────────┴─────────────────────│
  │                                   │
```

> 注意:Streamable HTTP 的 SSE 事件类型是 `event: message`(每个 JSON-RPC 消息一帧),旧 HTTP+SSE 方案的 `event: endpoint` 握手帧已废弃。

### 1.6 生命周期(Lifecycle Management)

MCP 是**有状态协议**(但 Streamable HTTP 的会话管理自 2025-03-26 起即为可选,可构建无状态部署),需生命周期管理,目的是协商双方能力。

```
1. initialize       → Client 发起握手,发送 protocolVersion + capabilities + clientInfo
2. initialized      → Client 收到 Server 响应后发送通知,进入正常通信
3. Operation        → 调用 tools/list、tools/call、resources/read 等
4. (Notifications)  → 双方互发通知(tools/list_changed、resources/updated)
5. Shutdown         → 优雅关闭连接
```

**初始化请求示例:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "elicitation": {},
      "roots": { "listChanged": true },
      "sampling": {}
    },
    "clientInfo": { "name": "example-client", "version": "1.0.0" }
  }
}
```

**初始化响应示例:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": { "subscribe": true, "listChanged": true },
      "prompts": { "listChanged": true },
      "logging": {},
      "tasks": {}
    },
    "serverInfo": { "name": "filesystem-server", "version": "1.0.0" }
  }
}
```

### 1.7 协议版本演进

| 版本 | 发布时间 | 关键变化 |
|------|----------|----------|
| 2024-11-05 | 2024-11 | 首版规范,JSON-RPC 2.0,有状态连接,stdio + HTTP+SSE 双通道 |
| 2025-03-26 | 2025-03 | **Streamable HTTP 取代 HTTP+SSE**、OAuth 2.1 授权框架、JSON-RPC 批处理、工具 annotations |
| 2025-06-18 | 2025-06 | 引入 **Elicitation**、**structuredContent** 结构化输出、Resource Links,移除 JSON-RPC 批处理 |
| 2025-11-25 | 2025-11 | 引入 **Tasks**(SEP-1686,实验性)、URL 模式 Elicitation、icons 元数据、CIMD、OIDC Discovery |

> 规范以 `YYYY-MM-DD` 日期版本号迭代,各版本不完全向后兼容,部署时应锁定版本。截至 2026-07,最新正式版为 **2025-11-25**;之后的版本以官方 changelog 为准(https://modelcontextprotocol.io/specification/changelog)。

**2025-03-26 规范要点:**
- **Streamable HTTP** 取代 HTTP+SSE 双通道方案,单一端点,会话管理可选(无状态部署自此可行)
- 引入 **OAuth 2.1** 授权框架,强制 PKCE
- 新增 **工具注释(annotations)**:`readOnlyHint` / `destructiveHint` 等,声明工具是只读/破坏性/需用户确认
- 新增 JSON-RPC 批处理、音频内容类型、`completions` 能力

**2025-06-18 规范要点:**
- 新增 **Elicitation** 原语:Server 可向用户请求额外信息(如密码、确认)
- 新增 **structuredContent**:工具可返回符合 outputSchema 的结构化数据
- 新增 Resource Links(工具结果中引用资源)
- **移除** JSON-RPC 批处理(2025-03-26 引入后又移除)
- OAuth 资源服务器归类(RFC 8707),HTTP 传输强制 `MCP-Protocol-Version` 头

**2025-11-25 规范要点:**
- 新增 **Tasks** 原语(SEP-1686,实验性):持久化异步请求,轮询获取延迟结果,支撑长任务
- 新增 **URL 模式 Elicitation**(SEP-1036):敏感交互(如 OAuth 授权)跳转到外部 URL 完成,不经过 Client
- 新增 **icons 元数据**(SEP-973):工具/资源/提示词可带图标
- Sampling 支持携带工具定义(SEP-1577),可在 Server 侧构建 agentic loop
- OAuth 增强:Client ID Metadata Documents(CIMD,SEP-991)、client-credentials M2M、OpenID Connect Discovery 1.0

### 1.9 Tools(工具)原语详解

#### 1.9.1 Tool 定义结构

```json
{
  "name": "read_file",
  "description": "读取指定路径的文件内容",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "文件绝对路径"
      }
    },
    "required": ["path"]
  },
  "annotations": {
    "readOnlyHint": true,
    "destructiveHint": false,
    "idempotentHint": true,
    "openWorldHint": false
  }
}
```

#### 1.9.2 Tool Annotations(工具注释,2025-03 新增)

| 注释字段 | 类型 | 说明 |
|----------|------|------|
| `readOnlyHint` | bool | 工具只读取不修改(安全等级最高) |
| `destructiveHint` | bool | 工具可能破坏数据(需用户确认) |
| `idempotentHint` | bool | 重复调用幂等(多次调用结果一致) |
| `openWorldHint` | bool | 工具与外部世界交互(网络调用) |

Host 可根据这些注释决定是否自动批准或要求用户确认。

#### 1.9.3 Tool 调用流程

```
1. LLM 决定调用工具
   ↓
2. Host 检查权限(annotations + 用户配置)
   ↓
3. Client 发送 tools/call 请求
   ↓
4. Server 执行工具
   ↓
5. Server 返回 content[] (支持多模态)
   ↓
6. Client 把结果交给 LLM
   ↓
7. LLM 基于结果继续推理
```

#### 1.9.4 Tool 返回内容类型

```json
{
  "content": [
    { "type": "text", "text": "文件内容..." },
    { "type": "image", "data": "base64...", "mimeType": "image/png" },
    { "type": "resource", "resource": { "uri": "file:///...", "text": "..." } }
  ],
  "isError": false
}
```

### 1.14 OAuth 2.1 + PKCE 认证流程

#### 1.14.1 为什么是 OAuth 2.1

OAuth 2.1 是 OAuth 2.0 的整合版本,合并了多个补丁(RFC 6749 + 多个 RFC):
- 强制 PKCE(防授权码拦截)
- 移除 Implicit Grant 和 Password Grant(不安全)
- 统一 Bearer Token 用法

#### 1.14.2 PKCE 流程详解

**PKCE(Proof Key for Code Exchange):** 防止授权码被中间人截获。

```
1. Client 生成 code_verifier(随机字符串)
   ↓
2. 计算 code_challenge = SHA256(code_verifier) + Base64URL
   ↓
3. 发起授权请求:
   GET /authorize?
     response_type=code&
     client_id=xxx&
     redirect_uri=xxx&
     code_challenge=xxx&
     code_challenge_method=S256
   ↓
4. 用户登录并授权
   ↓
5. Auth Server 重定向到 redirect_uri,带 code
   ↓
6. Client 用 code + code_verifier 换取 access_token:
   POST /token
     grant_type=authorization_code&
     code=xxx&
     code_verifier=原随机字符串
   ↓
7. Auth Server 验证:
   SHA256(code_verifier) == 之前收到的 code_challenge?
   ✓ 通过 → 返回 access_token
   ✗ 不匹配 → 拒绝(可能是中间人攻击)
```

#### 1.14.3 MCP 中的 OAuth 2.1 应用

- 远程 MCP Server 应支持 OAuth 2.1 授权流程
- **MCP Client 是 OAuth Client**(代表用户向 Authorization Server 请求令牌)
- **MCP Server 是 Resource Server**(验证请求携带的 Bearer Token,不代替用户登录)
- Host 启动时检测 `WWW-Authenticate` 头触发 OAuth 流程

| 症状 | 原因与修法 |
|------|-----------|
| Windows 下 `npx` 类 Server 启动失败 | Windows 的 npx 是 `.cmd` 脚本,需用 `cmd /c` 包装:`"command": "cmd", "args": ["/c", "npx", "-y", "xxx"]` |

| 改了代码不生效 | stdio Server 由 Host 托管进程,改完需重启 Host 或 `claude mcp` 重连 |

#### 1.15.5(进阶)底层 Server API 完整示例

需要精细控制(自定义 annotations、资源订阅等)时,用底层 `Server` API。注意启动写法:**`stdio_server()` 是异步上下文管理器**,必须在 `async with` 中拿到读写流后再 `server.run()`:

```python
# server_lowlevel.py
import asyncio
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent, ToolAnnotations

server = Server("my-filesystem-server")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="read_file",
            description="读取指定路径的文件内容",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {"type": "string", "description": "文件绝对路径"}
                },
                "required": ["path"]
            },
            annotations=ToolAnnotations(
                readOnlyHint=True,
                destructiveHint=False,
                idempotentHint=True,
                openWorldHint=False
            )
        )
    ]

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "read_file":
        with open(arguments["path"], "r", encoding="utf-8") as f:
            return [TextContent(type="text", text=f.read())]
    raise ValueError(f"未知工具: {name}")

async def main():
    async with stdio_server() as (read_stream, write_stream):
        await server.run(
            read_stream,
            write_stream,
            server.create_initialization_options()
        )

if __name__ == "__main__":
    asyncio.run(main())
```

### 1.17 安全注意事项

#### 1.17.1 常见风险

| 风险 | 说明 | 后果 |
|------|------|------|
| **工具滥用** | LLM 被注入攻击诱导调用危险工具 | 数据破坏、未授权操作 |
| **越权访问** | Server 暴露了过宽的文件系统/数据库权限 | 敏感数据泄露 |
| **凭证泄露** | 配置文件中的 API Key 被误传到远程 Server | 凭证被滥用 |
| **供应链攻击** | 第三方 Server 包含恶意代码 | 后门、数据窃取 |
| **Prompt 注入** | 工具返回内容包含恶意指令 | LLM 被劫持 |
| **Toxic Tools** | 工具元数据欺骗(标注 readOnly 实际写文件) | 绕过用户确认 |

#### 1.17.2 防护建议

- **审查源码:** 始终审查 Server 源码,优先使用官方/可信源
- **最小权限:** 使用最小权限原则配置 Server(限定目录、只读账户)
- **用户确认:** 启用工具调用的用户确认机制(Host 层)
- **传输安全:** 远程 Server 必须使用 OAuth 2.1 + HTTPS
- **配置审计:** 定期审查 `.mcp.json` 和 `config.toml` 中的配置
- **沙箱隔离:** 优先使用容器/沙箱运行不可信 Server
- **Roots 限制:** Client 主动声明可访问的文件系统根

#### 1.17.3 延伸阅读

- 官方安全最佳实践:https://modelcontextprotocol.io/specification/draft/basic/security_best_practices(随规范持续更新)
- 各类第三方的 MCP 安全报告/白皮书请以其官方发布页为准,注意甄别发布日期与内容时效。

## 二、A2A(智能体互通信)

### 2.0 基础定义

**全称:** Agent2Agent Protocol(智能体间协议)

**发布方:** Google,2025-04-10 Cloud Next 大会联合 50+ 厂商开源

**核心维护方:** Linux Foundation 旗下的 A2A 项目

**定位:** Agent ↔ Agent 跨框架协作协议

**官方资源:**
- 官方博客:https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/
- GitHub:https://github.com/google/A2A
- 文档:https://a2a-protocol.org/

### 2.1 A2A 解决的问题

#### 2.1.1 现状痛点

```
不同 Agent 框架各自为政:

┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│   LangGraph     │  │     CrewAI      │  │     AutoGen     │
│   Agent         │  │     Agent       │  │     Agent       │
└─────────────────┘  └─────────────────┘  └─────────────────┘
        ↑                    ↑                    ↑
   各自的协议            各自的协议            各自的协议
   各自的格式            各自的格式            各自的格式
```

- 不同 Agent 框架(LangGraph/CrewAI/AutoGen 等)各自为政
- Agent 之间无法直接交换任务和结果
- 缺少跨框架协作的统一标准
- 企业用 A 框架,合作伙伴用 B 框架,无法协作

#### 2.1.2 A2A 的解决方案

```
┌─────────────────┐                ┌─────────────────┐
│   LangGraph     │                │     CrewAI      │
│   Agent         │                │     Agent       │
└────────┬────────┘                └────────┬────────┘
         │                                  │
         │     A2A 协议(统一标准)         │
         └──────────────────────────────────┘
                     ↕
         ┌─────────────────┐
         │   AutoGen       │
         │   Agent         │
         └─────────────────┘
```

### 2.2 A2A 核心概念

| 概念 | 说明 |
|------|------|
| **Agent Card** | Agent 的能力描述名片(JSON),托管在 `/.well-known/agent-card.json` |
| **Client/Server 模型** | 同 MCP,但通信对象是 Agent 而非工具 |
| **Task(任务)** | A2A 中的核心工作单元,有完整生命周期 |
| **Message(消息)** | 任务中的对话消息,有 role(user/agent) |
| **Part(部分)** | 消息的组成部分(TextPart/FilePart/DataPart) |
| **Artifact(产物)** | 任务产生的最终输出(生成的图片、报告、代码) |
| **Task State(任务状态)** | submitted/working/input-required/completed/failed/canceled |
| **Push Notification** | 长任务的回调通知机制(Webhook) |

### 2.3 Agent Card(智能体名片)

#### 2.3.1 Agent Card 结构

A2A 规定每个 Agent 必须在 `https://{agent-host}/.well-known/agent-card.json` 暴露 Agent Card(0.3.0 起;更早版本为 `/.well-known/agent.json`):

```json
{
  "$schema": "https://a2a-protocol.org/schemas/agent-card.json",
  "name": "Code Review Agent",
  "description": "专业的代码审查助手,支持多种编程语言",
  "version": "1.0.0",
  "protocolVersion": "0.3.0",
  "url": "https://api.example.com/a2a",
  "preferredTransport": "JSONRPC",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true,
    "stateTransitionHistory": true
  },
  "defaultInputModes": ["text", "file"],
  "defaultOutputModes": ["text", "file"],
  "skills": [
    {
      "id": "code-review-python",
      "name": "Python 代码审查",
      "description": "审查 Python 代码的质量、安全性、可读性",
      "tags": ["python", "code-review", "security"],
      "examples": [
        "审查这段 Python 代码: def add(a, b): return a+b",
        "检查这个 Flask 应用的安全问题"
      ],
      "inputModes": ["text", "file"],
      "outputModes": ["text"]
    },
    {
      "id": "code-review-javascript",
      "name": "JavaScript 代码审查",
      "description": "审查 JavaScript/TypeScript 代码",
      "tags": ["javascript", "typescript", "code-review"]
    }
  ],
  "securitySchemes": {
    "bearerAuth": {
      "type": "http",
      "scheme": "bearer"
    }
  },
  "security": [{ "bearerAuth": [] }],
  "documentationUrl": "https://docs.example.com/code-review-agent"
}
```

#### 2.3.2 Agent Card 字段详解

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | ✓ | Agent 名称 |
| `description` | ✓ | Agent 能力描述 |
| `version` | ✓ | Agent 版本 |
| `protocolVersion` | ✓ | 支持的 A2A 协议版本 |
| `url` | ✓ | Agent 的 A2A 端点 |
| `preferredTransport` | ✗ | 首选传输方式(JSONRPC/GRPC/HTTP) |
| `capabilities` | ✓ | 能力声明 |
| `defaultInputModes` | ✓ | 默认接受的输入类型 |
| `defaultOutputModes` | ✓ | 默认输出类型 |
| `skills` | ✓ | 技能列表(每个技能是 Agent 的一项专长) |
| `securitySchemes` | ✗ | 安全方案定义 |
| `security` | ✗ | 应用哪些安全方案 |
| `documentationUrl` | ✗ | 文档链接 |

### 2.4 A2A 任务模型

#### 2.4.1 任务生命周期

```
            ┌───────────┐
            │ submitted │ ← 任务提交
            └─────┬─────┘
                  │
                  ▼
            ┌───────────┐
            │  working  │ ← 执行中
            └─────┬─────┘
                  │
        ┌─────────┼─────────┐
        │         │         │
        ▼         ▼         ▼
┌───────────┐ ┌─────────┐ ┌──────────┐
│input-required│ │ failed │ │ canceled │
└─────┬─────┘ └─────────┘ └──────────┘
      │
      ▼
┌───────────┐
│ completed │ ← 成功完成
└───────────┘
```

#### 2.4.2 任务创建请求

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "message/send",
  "params": {
    "message": {
      "messageId": "msg-uuid-xxx",
      "contextId": "context-yyy",
      "role": "user",
      "parts": [
        {
          "kind": "text",
          "text": "请审查这段 Python 代码: def add(a, b): return a+b"
        }
      ]
    }
  }
}
```

> `contextId` 是 A2A 0.3.0 的会话标识(0.2.x 叫 `sessionId`,已废弃);同一 `contextId` 下的多个任务共享对话上下文。

#### 2.4.3 任务响应

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "id": "task-uuid-xxx",
    "contextId": "context-yyy",
    "status": {
      "state": "completed",
      "timestamp": "2026-07-19T10:30:00Z"
    },
    "artifacts": [
      {
        "name": "审查报告",
        "parts": [
          {
            "kind": "text",
            "text": "代码审查结果:\n1. 函数命名规范 ✓\n2. 缺少类型注解 ⚠\n3. 缺少文档字符串 ⚠"
          }
        ]
      }
    ]
  }
}
```

### 2.5 A2A 长任务与 Push Notification

#### 2.5.1 Push Notification 注册(独立方法,不内嵌于发送请求)

长任务(可能持续数分钟到数小时)不能一直占用 HTTP 连接,A2A 提供 Push Notification 机制。注意:**推送配置不内嵌在 `message/send` 请求里**,需对同一任务单独调用 `tasks/pushNotificationConfig/set`:

```json
// 第 1 步:先发任务
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "message/send",
  "params": {
    "message": {
      "messageId": "msg-uuid-xxx",
      "role": "user",
      "parts": [{ "kind": "text", "text": "分析这个 10GB 日志文件" }]
    }
  }
}
// 响应返回 task.id = "long-task-xxx",state = "working"

// 第 2 步:为该任务注册 webhook
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tasks/pushNotificationConfig/set",
  "params": {
    "taskId": "long-task-xxx",
    "pushNotificationConfig": {
      "url": "https://client.example.com/webhook/a2a",
      "token": "client-validation-token",
      "authentication": {
        "schemes": ["bearer"]
      }
    }
  }
}
```

配套方法:`tasks/pushNotificationConfig/get` / `.../list` / `.../delete`(查询、列出、删除配置)。

#### 2.5.2 Webhook 回调流程

```
1. Client 发起长任务(message/send),Server 返回 task.id + state = working
   ↓
2. Client 调用 tasks/pushNotificationConfig/set 注册 webhook
   ↓
3. Client 释放连接,做其他事
   ↓
4. Server 后台执行任务
   ↓
5. 任务完成后,Server POST 到 webhook url
   POST https://client.example.com/webhook/a2a
   Authorization: Bearer xxx
   Body: { "taskId": "long-task-xxx", "state": "completed" }
   ↓
6. Client 收到回调,调用 tasks/get 获取结果
```

### 2.6 A2A 安全模型

| 安全方案 | 说明 |
|----------|------|
| **HTTP Basic** | 用户名密码(基础) |
| **HTTP Bearer** | Bearer Token(常用) |
| **OAuth2** | OAuth 2.0 授权码流程(v1.0 起移除 implicit/password 流,新增 Device Code 流与 PKCE 支持) |
| **OpenID Connect** | OAuth2 + 身份认证 |
| **API Key** | Header / Query 参数 |
| **mutual TLS** | v1.0 新增,双向证书认证 |

### 2.7 A2A 1.0(2026 年发布,首个稳定版)

A2A 在 2026 年发布 **1.0.0**,这是脱离 0.x 试验期的首个稳定版,含多处 **breaking changes**。上文的 0.3.0 示例在 1.0 下需要按下表迁移;**迁移细节务必以官方迁移指南为准**(https://a2a-protocol.org/latest/whats-new-v1/)。

#### 2.7.1 关键 breaking changes

| 维度 | 0.3.0 | 1.0 |
|------|-------|-----|
| **Agent Card 端点声明** | `url` + `preferredTransport` + `additionalInterfaces` | 统一为 `supportedInterfaces[]`,每项含 `url` / `protocolBinding` / `protocolVersion` |
| **协议版本位置** | 卡片顶层 `protocolVersion: "0.3.0"` | 移到每个 `AgentInterface` 内,支持一卡多版本并存 |
| **任务状态枚举** | 小写/kebab:`"working"`、`"input-required"` | SCREAMING_SNAKE 带前缀:`"TASK_STATE_WORKING"`、`"TASK_STATE_INPUT_REQUIRED"` |
| **消息角色枚举** | `"user"` / `"agent"` | `"ROLE_USER"` / `"ROLE_AGENT"` |
| **Part 结构** | `TextPart` / `FilePart` / `DataPart`,靠 `kind` 字段判别 | 统一为单个 `Part`,靠成员字段(`text`/`url`/`raw`/`data`)判别;`mimeType` → `mediaType` |
| **操作命名** | JSON-RPC 风格:`message/send`、`tasks/get`、`tasks/cancel` | gRPC 统一为 PascalCase:`SendMessage`、`GetTask`、`CancelTask`(JSON-RPC 绑定同步调整) |
| **流式事件** | 带 `kind` 判别字段与 `final` 布尔位 | 去掉 `kind`/`final`,用包装成员(`taskStatusUpdate` / `taskArtifactUpdate`)判别,流关闭表示终态 |
| **扩展卡片能力** | 顶层 `supportsAuthenticatedExtendedCard` | 移入 `capabilities.extendedAgentCard` |
| **HTTP+JSON 路径** | 带 `/v1` 前缀(如 `POST /v1/message:send`) | 去掉前缀(如 `POST /message:send`) |
| **错误模型** | RFC 9457 Problem Details | `google.rpc.Status` + `ErrorInfo`(`domain: "a2a-protocol.org"`) |
| **任务列表** | 无 `tasks/list` | 新增 `ListTasks`,cursor 分页,按调用者鉴权过滤 |

#### 2.7.3 1.0 新增能力

- **版本协商:** 客户端每个请求带 `A2A-Version: 1.0` 头,Server 校验不支持的版本直接拒绝;一卡多接口可实现平滑灰度迁移
- **Agent Card 签名:** 基于 JWS(RFC 7515)+ JSON 规范化(RFC 8785),Client 可验证卡片未被篡改

- **扩展机制:** 卡片可声明 `extensions[]`(含 `required` 标记),客户端可据此判断兼容性

#### 2.7.4 最小 echo Agent(a2a-sdk,Python)

官方 Python SDK 是 `a2a-sdk`(仓库 a2aproject/a2a-python,2025 年验证 0.3.x API;1.0 API 以官方文档为准):

```bash
pip install a2a-sdk uvicorn
```

```python
# echo_agent.py —— 最小 A2A echo agent
import uvicorn
from a2a.server.apps import A2AStarletteApplication
from a2a.server.request_handlers import DefaultRequestHandler
from a2a.server.tasks import InMemoryTaskStore
from a2a.server.agent_execution import AgentExecutor, RequestContext
from a2a.server.events import EventQueue
from a2a.types import AgentCard, AgentSkill, AgentCapabilities
from a2a.utils import new_agent_text_message

class EchoExecutor(AgentExecutor):
    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None:
        text = context.get_user_input()          # 取用户输入文本
        await event_queue.enqueue_event(
            new_agent_text_message(f"echo: {text}")
        )

    async def cancel(self, context, event_queue) -> None:
        raise NotImplementedError("不支持取消")

skill = AgentSkill(id="echo", name="回声", description="原样返回输入",
                   tags=["echo"], examples=["hello"])
card = AgentCard(
    name="Echo Agent", description="最小 A2A 示例", version="0.1.0",
    url="http://localhost:9999/", protocol_version="0.3.0",
    capabilities=AgentCapabilities(streaming=False),
    default_input_modes=["text"], default_output_modes=["text"],
    skills=[skill],
)

handler = DefaultRequestHandler(agent_executor=EchoExecutor(),
                                task_store=InMemoryTaskStore())
app = A2AStarletteApplication(agent_card=card, http_handler=handler)

if __name__ == "__main__":
    uvicorn.run(app.build(), host="0.0.0.0", port=9999)
```

启动后 `curl http://localhost:9999/.well-known/agent-card.json` 应返回卡片 JSON;用 SDK 的 `A2AClient` 或任意 JSON-RPC 客户端发 `message/send` 即可收到 `echo: ...`。

## 三、MCP vs A2A 关键差异(2026 官方定位)

### 3.1 核心对比

| 维度 | MCP | A2A |
|------|-----|-----|
| **维护方** | Anthropic + 社区 | Google + 50+ 厂商(Linux Foundation) |
| **通信方向** | Agent ↔ 工具/数据 | Agent ↔ Agent |
| **解决问题** | Agent 接入外部能力 | Agent 间跨框架协作 |
| **发布时间** | 2024-11 | 2025-04-10 |
| **架构** | Host/Client/Server 三角 | Client/Server + Agent Card |
| **核心原语** | Tools/Resources/Prompts/Sampling/Elicitation | Task/Message/Part/Artifact |
| **传输** | stdio / Streamable HTTP | JSONRPC / gRPC / HTTP+JSON 三种绑定,流式走 SSE(无 WebSocket) |
| **认证** | OAuth 2.1 + PKCE | 多方案(Bearer/OAuth2/API Key/mTLS) |
| **状态** | 有状态(2025-03-26 起会话管理可选,可无状态部署) | 有状态(任务有完整生命周期) |
| **发现机制** | 配置文件(.mcp.json)+ 官方 MCP Registry(2025 年上线) | `/.well-known/agent-card.json` 自动发现 |
| **典型场景** | Agent 调用数据库/API/文件 | 多 Agent 协作完成复杂任务 |

### 3.2 互补关系

MCP 和 A2A 是**互补**而非竞争关系:

```
┌──────────────────────────────────────────────────────────┐
│                  用户请求                                │
│                  "分析 GitHub 仓库并生成报告"            │
└────────────────────────┬─────────────────────────────────┘
                         │
                         ▼
              ┌──────────────────┐
              │   主 Agent       │
              │  (Claude Code)   │
              └────────┬─────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
        ▼ (MCP)        ▼ (A2A)        ▼ (MCP)
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ GitHub MCP   │ │ 代码审查     │ │ 文件系统     │
│ Server       │ │ Agent (A2A)  │ │ MCP Server   │
│ (Agent↔工具) │ │ (Agent↔Agent)│ │ (Agent↔工具) │
└──────────────┘ └──────────────┘ └──────────────┘
```

- **MCP 负责 Agent 接入外部工具和数据**(横向扩展能力)
- **A2A 负责 Agent 之间的协作**(纵向扩展智能)
- 一个完整的多 Agent 系统可能同时使用两者

**同生态的其他协议(一句话了解):**
- **AG-UI(Agent↔前端):** CopilotKit 主导的 Agent 与用户界面交互协议,把 Agent 的流式输出、工具调用、状态同步以事件流形式推给前端,解决"Agent↔UI"而非"Agent↔工具/Agent"的标准化。
- **ACP(Agent Communication Protocol):** IBM/BeeAI 发起的 Agent 通信协议,2025 年已并入 Linux Foundation 旗下的 A2A 项目,生态收敛到 A2A 上,新项目直接选 A2A 即可。

### 3.3 协议层次图

```
┌────────────────────────────────────────────┐
│   应用层(用户业务)                       │
├────────────────────────────────────────────┤
│   Agent 层(智能体)                       │
│   ┌────────────────────────────────┐      │
│   │   A2A 协议(Agent ↔ Agent)    │      │  ← 跨 Agent 协作
│   └────────────────────────────────┘      │
├────────────────────────────────────────────┤
│   能力层(工具/资源)                      │
│   ┌────────────────────────────────┐      │
│   │   MCP 协议(Agent ↔ 工具)     │      │  ← Agent 调用工具
│   └────────────────────────────────┘      │
```

## 四、Function Calling(MCP 的前身)

### 4.1 基础定义

**全称:** Function Calling(函数调用)

**发布方:** OpenAI,2023-06 在 Chat Completions API 中推出

**定义:** LLM 根据用户意图和函数描述,输出结构化 JSON 调用外部函数的机制。

### 4.2 Function Calling 工作流程

```
1. 开发者定义可用函数(名称/描述/参数 schema)
   ↓
2. 用户提问
   ↓
3. LLM 判断:需要调用哪个函数?用什么参数?
   ↓
4. LLM 输出结构化 JSON(tool_calls)
   ↓
5. 开发者代码执行对应函数
   ↓
6. 把函数结果作为 tool 角色消息回传给 LLM
   ↓
7. LLM 基于结果继续生成最终回答
```

### 4.3 Function Calling 示例

```python
from openai import OpenAI

client = OpenAI()

# 1. 定义工具
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询某城市的天气",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名"}
                },
                "required": ["city"]
            }
        }
    }
]

# 2. 用户提问
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "北京天气怎么样?"}],
    tools=tools
)

# 3. LLM 输出 tool_calls
tool_call = response.choices[0].message.tool_calls[0]
# tool_call.function.name = "get_weather"
# tool_call.function.arguments = '{"city": "北京"}'

# 4. 执行函数
import json
args = json.loads(tool_call.function.arguments)
result = get_weather(args["city"])  # 实际函数实现

# 5. 把结果回传给 LLM
response2 = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "user", "content": "北京天气怎么样?"},
        response.choices[0].message,
        {"role": "tool", "tool_call_id": tool_call.id, "content": result}
    ]
)
# 6. LLM 生成最终回答
print(response2.choices[0].message.content)
```

### 4.4 Function Calling vs MCP 对比

| 维度 | Function Calling | MCP |
|------|------------------|-----|
| **维护方** | 各 LLM 厂商各自实现 | Anthropic + 社区(开放标准) |
| **协议层** | 无,直接写在 API 请求里 | 完整的 JSON-RPC 协议 |
| **传输方式** | 内嵌 LLM API | stdio / Streamable HTTP |
| **工具发现** | 每次请求都传完整工具列表 | `tools/list` 动态发现 |
| **协议原语** | 仅 tools | Tools / Resources / Prompts / Sampling / Elicitation / Tasks |
| **状态** | 无状态 | 有状态(支持生命周期) |
| **跨厂商** | 各厂商格式略有差异 | 统一标准 |
| **典型用法** | 简单工具调用 | 复杂 Agent 系统 |

**关系:** MCP 是 Function Calling 的"协议化升级",底层仍依赖 LLM 的 Function Calling 能力。Host 在内部把 MCP 工具转换为 LLM 可识别的 Function Calling 格式。

## 五、Handoff(任务移交)

### 5.1 基础定义

**定义:** 把任务交给更合适的 Agent 处理的机制,是多 Agent 系统中的核心协作模式。

### 5.2 典型实现

| 实现 | 来源 | 特点 |
|------|------|------|
| **OpenAI Agents SDK Handoffs** | OpenAI 2025-03 | 三原语之一,框架级支持 |
| **Claude Code Subagents** | Anthropic | 自动委派机制,3 个内置 Subagent(Explore / Plan / general-purpose) |
| **A2A 协议** | Google | 跨框架任务转交 |
| **LangGraph Supervisor** | LangChain | 中心化调度,Supervisor 决定移交 |
| **CrewAI Process** | CrewAI | 顺序/层级流程,内置移交 |

### 5.3 OpenAI Agents SDK Handoff 详解

```python
from agents import Agent, Runner

# 定义两个 Agent
booking_agent = Agent(
    name="预订专家",
    instructions="专门负责机票、酒店预订",
    tools=[search_flights, book_flight, search_hotels]
)

refund_agent = Agent(
    name="退款专家",
    instructions="专门负责退款流程",
    tools=[check_order, process_refund]
)

# 主 Agent 配置 handoffs
triage_agent = Agent(
    name="客服前台",
    instructions="根据用户意图移交给对应专家",
    handoffs=[booking_agent, refund_agent]  # 声明可移交的 Agent
)

# 运行
result = Runner.run_sync(triage_agent, "我想退上周的机票订单")
# 1. triage_agent 判断这是退款请求
# 2. 自动 handoff 给 refund_agent
# 3. refund_agent 处理退款
# 4. 返回最终结果
```

### 5.4 Handoff 流程图

```
主 Agent 接收任务
    ↓
判断:我能处理吗?
    ├── 是 → 自己处理
    └── 否 → 寻找更合适的 Agent
              ↓
          Handoff 移交(带上下文摘要)
              ↓
          目标 Agent 接管会话
              ↓
          执行任务(可继续 handoff)
              ↓
          返回结果
              ↓
          主 Agent 整合回复用户
```

### 5.5 Handoff 的核心价值

| 价值 | 说明 |
|------|------|
| **专业化分工** | 每个 Agent 专注自己擅长的领域 |
| **上下文隔离** | 任务在独立上下文中处理,不污染主会话 |
| **可扩展性** | 动态添加新的专业 Agent |
| **Token 优化** | 主 Agent 上下文更精简 |
| **工具隔离** | 不同 Agent 有不同工具集,避免误调用 |

## 六、Message Queue(消息队列)在 Agent 系统中的应用

### 6.1 基础定义

**定义:** 异步消息传递中间件,用于系统间解耦通信。

### 6.2 主流消息队列对比

| 代表 | 特点 | 适用场景 |
|------|------|----------|
| **Kafka** | 高吞吐分布式消息队列,持久化 | 大数据、事件溯源、日志聚合 |
| **RabbitMQ** | 功能丰富的 AMQP 消息代理 | 企业级、复杂路由、任务队列 |
| **Redis Stream** | 轻量级消息流,低延迟 | 实时场景、简单任务队列 |
| **NATS** | 极简高性能 | 微服务通信、IoT |
| **AWS SQS** | 托管消息队列 | 云原生 AWS 应用 |
| **Pulsar** | 多租户、可扩展 | 大规模企业 |

### 6.3 在 Agent 系统中的作用

| 作用 | 说明 |
|------|------|
| **异步任务处理** | Agent 任务提交到队列,Worker 异步执行 |
| **解耦** | Agent 之间通过队列通信,不直接依赖 |
| **削峰填谷** | 缓冲请求峰值,保护下游系统 |
| **事件驱动** | 消息触发 Agent 执行 |
| **重试机制** | 失败自动重试,保证可靠性 |
| **可观测性** | 消息流可追溯 |

### 6.4 典型架构

```
用户请求
    ↓
[API Gateway]
    ↓
[任务队列 (Kafka)]  ← 削峰填谷
    ↓
┌─────────────┬─────────────┬─────────────┐
│ Agent       │ Agent       │ Agent       │
│ Worker 1    │ Worker 2    │ Worker 3    │  ← 水平扩展
└─────────────┴─────────────┴─────────────┘
    ↓
[结果队列]
    ↓
[结果聚合/通知]
```

## 七、其他相关通信协议

### 7.1 Protocol(通信协议)层次

**定义:** 通信双方必须遵循的规则集合。

| 协议层 | 典型协议 | 作用 |
|--------|----------|------|
| **传输层** | HTTP/WebSocket/SSE | 数据传输通道 |
| **RPC 层** | JSON-RPC/gRPC | 远程过程调用 |
| **应用层** | MCP/A2A/OpenAI API | AI 专用协议 |
| **认证层** | OAuth 2.1/API Key/JWT | 身份验证与授权 |

### 7.2 gRPC 详解

**全称:** gRPC Remote Procedure Calls

**发布方:** Google,2015 年

**特点:**
- 基于 HTTP/2,支持双向流
- 使用 Protocol Buffers(Protobuf)二进制编码,高效
- 强类型,自动生成多语言客户端
- 适合微服务间通信

**对比 REST:**
| 维度 | gRPC | REST |
|------|------|------|
| 编码 | Protobuf(二进制) | JSON(文本) |
| 传输 | HTTP/2 | HTTP/1.1 或 HTTP/2 |
| 性能 | 高(二进制+HTTP/2) | 中 |
| 流式 | 原生支持双向流 | SSE/WebSocket 补充 |
| 浏览器 | 需要 grpc-web 代理 | 原生支持 |
| 适用 | 服务间通信 | 公开 API |

### 7.3 Webhook(网络钩子)

**定义:** 事件驱动的反向 HTTP 回调机制,某系统在事件发生时主动 POST 到预先注册的 URL。

**A2A 中的应用:** 长任务完成后,Server 主动 POST 通知 Client。

**典型 Webhook 流程:**
```
1. Client 向 Server 注册 webhook(url + secret)
   ↓
2. Client 发起长任务
   ↓
3. Server 立即返回任务 ID
   ↓
4. Client 释放连接
   ↓
5. Server 后台执行任务
   ↓
6. 任务完成,Server POST 到 webhook url
   Headers:
     X-Webhook-Signature: HMAC-SHA256(body, secret)
   Body: { "event": "task.completed", "taskId": "xxx", ... }
   ↓
7. Client 验签并处理
```

### 7.4 JSON-RPC 2.0 错误码

| 错误码 | 含义 | 说明 |
|--------|------|------|
| -32700 | Parse error | JSON 解析错误(格式不对) |
| -32600 | Invalid Request | 无效请求(不是合法的 JSON-RPC 请求) |
| -32601 | Method not found | 方法不存在(Server 不支持该 method) |
| -32602 | Invalid params | 参数无效 |
| -32603 | Internal error | Server 内部错误 |
| -32000 to -32099 | Server error | 保留给 Server 自定义错误 |

## 八、协议选型决策树

```
你需要解决什么问题?
│
├─ Agent 需要调用外部工具/数据?
│  ├─ 是 → MCP
│  │       ├─ 本地工具 → stdio 传输
│  │       └─ 远程工具 → Streamable HTTP + OAuth 2.1
│  │
│  └─ 否 ↓
│
├─ 多个 Agent 之间需要协作?
│  ├─ 是 → A2A
│  │       ├─ 同框架 → 框架原生 API(LangGraph/CrewAI)
│  │       └─ 跨框架 → A2A 标准协议
│  │
│  └─ 否 ↓
│
├─ 仅需要 LLM 调用简单函数?
│  ├─ 是 → Function Calling(OpenAI/Anthropic API)
│  │
│  └─ 否 ↓
│
├─ 长任务需要异步处理?
│  ├─ 是 → Message Queue(Kafka/RabbitMQ) + Webhook
│  │
│  └─ 否 ↓
│
└─ 服务间高性能通信? → gRPC
```

```
Triage Agent (前台分流)
```
