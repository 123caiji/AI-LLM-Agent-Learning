# 04 - 能力与约束体系(Capability & Constraint)

> Agent 的能力封装与行为约束机制。涵盖 Skills 技能、CLAUDE.md 项目规则、AGENTS.md 跨工具标准、Rules 运行约束、Hooks 生命周期钩子、Plugin 插件、Tool Calling、Tool Schema、沙箱等核心概念。本章是 Agent 的"能力扩展层 + 行为边界层",理解它 = 知道 Agent "能做什么"和"不能做什么"。

## 一、Skill(技能/单项能力)

### 1.1 基础定义

**定义:** 可复用的能力包,将特定领域的知识、流程、模板打包为可被 Agent 调用的模块化单元。

**标准:** AgentSkills.io (https://agentskills.io)

### 1.2 Skill 的核心价值

| 价值 | 说明 |
|------|------|
| **可复用** | 一次编写,多个项目/多个 Agent 使用 |
| **专业化** | 每个 Skill 聚焦一个能力领域 |
| **可分享** | 团队间共享最佳实践 |
| **可组合** | 多个 Skill 组合完成复杂任务 |
| **降低成本** | 避免重复定义相同能力 |
| **标准化** | 跨项目/跨工具复用 |

### 1.3 Claude Code Skills 系统

Skill 是封装好的能力包,位于 `.claude/skills/` 目录:

```
.claude/skills/
└── code-review/
    ├── SKILL.md          # 技能定义(YAML front matter + Markdown 内容)
    ├── scenarios.md      # 场景示例
    └── templates/        # 模板文件
        ├── review-template.md
        └── checklist.md
```

**SKILL.md 示例:**

```markdown
---
name: code-review
description: 执行深度代码审查,检查安全、性能、可维护性。当用户要求审查代码时主动使用。
allowed-tools: [Read, Grep, Glob]
---

# 代码审查技能

## 审查维度
1. **安全性**:注入、XSS、CSRF、权限绕过
2. **性能**:N+1 查询、内存泄漏、阻塞调用
3. **可维护性**:命名、复杂度、重复代码
4. **测试覆盖**:边界情况、异常路径

## 输出格式
- 严重等级:🔴 高 / 🟡 中 / 🟢 低
- 文件位置:`file:line`
- 问题描述
- 修复建议(含代码示例)
```

### 1.4 SKILL.md 关键字段

> 官方规范(agentskills.io)+ Claude Code 扩展字段。**注意是 `allowed-tools`(连字符),不是 `allowed_tools`;不存在 `trigger`、`priority` 字段**——触发完全靠 LLM 读 `description` 自行判断。

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | ✓ | 技能唯一标识(kebab-case,须与目录名一致) |
| `description` | ✓ | 技能描述,LLM 据此决定何时使用(含 `when_to_use` 在内有长度上限) |
| `when_to_use` | ✗ | 触发场景/示例短语,追加到 description 中展示 |
| `allowed-tools` | ✗ | 该技能可使用的工具列表(连字符命名) |
| `disable-model-invocation` | ✗ | 为 true 时 Claude 不能自动调用,仅用户可手动 `/skill` 调用 |
| `user-invocable` | ✗ | 为 false 时不在斜杠菜单显示(但仍可被模型调用) |
| `model` | ✗ | 指定使用的模型(如 haiku) |
| `context` | ✗ | `fork` 时技能在 fork 出的 subagent 上下文中执行 |
| `agent` | ✗ | 配合 `context: fork` 指定 subagent 类型 |
| `argument-hint` | ✗ | 参数自动补全提示 |
| `hooks` | ✗ | 技能生命周期内生效的钩子 |
| `license` / `metadata` | ✗ | agentskills.io 规范中的元数据字段 |

### 1.5 Codex CLI Skills & Plugins

- **Skills:** 可复用指令包,纯 Markdown 定义
- **Plugins:** 接入团队工具的扩展模块,可包含代码

**Codex Skills 目录:** `~/.codex/skills/` 或项目 `.codex/skills/`

### 1.6 官方 Skills 资源

| 资源 | 网址 |
|------|------|
| **Claude Code Skills 文档** | https://code.claude.com/docs/en/skills |
| **Anthropic 官方 Skills 仓库** | https://github.com/anthropics/skills |
| **AgentSkills.io 标准** | https://agentskills.io |
| **Awesome Codex Skills** | https://github.com/ComposioHQ/awesome-codex-skills |

### 1.7 Skill 的执行流程

```
用户输入
    ↓
[Claude 加载所有 Skills 的 metadata]
    ↓
[LLM 根据 description 判断是否需要使用某 Skill]
    ↓
    ├── 不需要 → 正常处理
    └── 需要 → 加载完整 SKILL.md 到上下文
                ↓
            [按 Skill 指令执行]
                ↓
            [使用 Skill 允许的工具]
                ↓
            [按 Skill 输出格式返回]
```

## 二、CLAUDE.md(Claude 项目持久规则文档)

### 2.1 基础定义

**定义:** Claude Code 启动时自动加载的项目规则文件,定义项目级行为规范。

**位置:** 项目根目录(可在子目录嵌套)

### 2.2 CLAUDE.md 的作用

| 作用 | 说明 |
|------|------|
| **项目说明** | 描述项目类型、技术栈、目录结构 |
| **编码规范** | 代码风格、命名约定、测试要求 |
| **禁止事项** | 明确 Agent 不能做的操作 |
| **工作流** | 常用命令、提交流程、验证步骤 |
| **业务规则** | 业务逻辑、领域知识、术语表 |
| **架构说明** | 系统架构、模块职责、依赖关系 |

### 2.3 CLAUDE.md 示例

```markdown
# Claude Code 项目规则

## 项目概述
- 类型: TypeScript + React 单页应用
- 构建: Vite
- 测试: Vitest
- 部署: Vercel

## 目录结构
- src/components/   React 组件
- src/hooks/        自定义 Hooks
- src/utils/        工具函数
- src/api/          API 调用层
- src/types/        TypeScript 类型定义

## 编码规范
- 函数式组件 + Hooks,不用 class
- 使用 Tailwind CSS,不写自定义 CSS
- 必须处理错误边界
- 文件命名:kebab-case(如 `user-profile.tsx`)
- 组件命名:PascalCase(如 `UserProfile`)
- 必须为 public API 添加 TypeScript 类型

## 禁止事项
- 不要修改 package-lock.json
- 不要引入新的 UI 库
- 不要使用 any 类型
- 不要直接修改 dist/ 目录
- 不要在 components 中直接调用 fetch

## 工作流
- 修改后必须运行: `npm run lint && npm test`
- 提交前格式化: `npm run format`
- 提交信息使用 Conventional Commits 格式
- 创建新组件时同步创建 .test.tsx 文件

## 业务规则
- 用户角色:admin / editor / viewer
- 权限检查必须使用 useAuth hook
- API 错误统一用 ApiError 类包装
```

### 2.4 CLAUDE.md 的层级覆盖机制

```
项目根/
├── CLAUDE.md              ← 全局规则(所有子目录生效)
├── src/
│   ├── CLAUDE.md          ← src 目录补充规则
│   └── components/
│       └── CLAUDE.md      ← components 目录补充规则
└── tests/
    └── CLAUDE.md          ← tests 目录补充规则
```

**规则合并机制:**
- 启动时按层级加载所有 CLAUDE.md
- 子目录规则**补充**而非替换父目录
- 冲突时子目录规则优先
- 当前工作目录决定加载哪些 CLAUDE.md

### 2.5 用户级 CLAUDE.md

位置:`~/.claude/CLAUDE.md`

**作用:** 跨项目的个人偏好规则,所有项目都生效。

```markdown
# 个人偏好

## 通用规则
- 优先使用中文回复
- 代码注释用中文
- 优先选择简洁的实现
- 不要过度工程化

## 编程语言偏好
- Python: 类型注解必备,使用 black 格式化
- TypeScript: 严格模式,使用 prettier
- Go: 遵循 Effective Go

## 工具偏好
- 测试: 优先 pytest / vitest
- Linter: ruff(Python)/ eslint(TS)
- 包管理: uv(Python)/ pnpm(TS)
```

### 2.6 CLAUDE.md 的特点

| 特点 | 说明 |
|------|------|
| **启动自动加载** | Claude Code 启动时自动读取 |
| **软约束** | LLM 大概率遵守,但非 100% 强制 |
| **层级覆盖** | 子目录的 CLAUDE.md 可补充父目录规则 |
| **跨项目复用** | 用户级 CLAUDE.md 可全局生效 |
| **Token 占用** | 内容会占用上下文 Token,要精简 |
| **可手动编辑** | 用户可随时修改,无需重启 |

**官方文档:** https://code.claude.com/docs/en/memory

## 三、AGENTS.md(跨工具通用标准)

### 3.1 基础定义

**官网:** https://agents.md

**已被 60k+ 项目采用**

**意义:** 跨 CLI Agent 工具的通用配置文件标准,类似 `.gitignore`。

### 3.2 支持的工具

| 工具 | 支持 AGENTS.md |
|------|----------------|
| Codex CLI | ✓ 原生支持 |
| Claude Code | △ 官方承诺读取的是 CLAUDE.md;AGENTS.md 通常靠 symlink(如 `mklink AGENTS.md CLAUDE.md` 反向链接)或 `@AGENTS.md` 导入变通 |
| Cursor | ✓ 支持 |
| Sourcegraph Cody | ✓ 支持 |
| Aider | ✓ 支持 |
| 其他 CLI Agent | 社区推动中 |

> 注:截至 2025 年底,Claude Code 官方只承诺自动加载 `CLAUDE.md`(可用 `@路径` 语法在 CLAUDE.md 中导入其他文件)。社区常见做法是建符号链接让两者内容同源,具体支持情况以官方文档为准。

### 3.3 推荐放置层级

```
项目根/
├── AGENTS.md              # 全局规则(所有 Agent 生效)
├── src/
│   └── AGENTS.md          # src 目录专属规则
└── tests/
    └── AGENTS.md          # 测试目录专属规则
```

### 3.4 AGENTS.md 最小可用模板

```markdown
# AGENTS.md

## 项目类型
[一句话描述项目]

## 技术栈
- 语言: [Python/TypeScript/Go...]
- 框架: [FastAPI/React/Gin...]
- 测试: [pytest/vitest/...]
- 部署: [Docker/K8s/Vercel...]

## 目录结构
- src/          主代码
- tests/        测试
- docs/         文档
- scripts/      脚本

## 禁止操作
- [列出 Agent 不能做的事]
- 例如:不要修改 migrations 目录

## 必做检查
- [列出 Agent 必须执行的验证]
- 例如:修改后必须运行 `make test`

## 编码规范
- [项目特定规范]
```

### 3.5 AGENTS.md 完整示例

```markdown
# AGENTS.md - 电商后端项目

## 项目类型
电商系统后端,基于微服务架构

## 技术栈
- 语言: Python 3.12
- 框架: FastAPI + SQLAlchemy 2.0
- 数据库: PostgreSQL 16
- 缓存: Redis 7
- 消息队列: RabbitMQ
- 测试: pytest + pytest-asyncio
- 部署: Docker + K8s

## 目录结构
- src/api/          API 路由
- src/models/       数据模型
- src/services/     业务逻辑
- src/repositories/ 数据访问层
- src/schemas/      Pydantic schemas
- tests/            测试
- migrations/       数据库迁移(不要手动修改)

## 编码规范
- 必须使用 async/await
- 数据库操作必须用 SQLAlchemy 2.0 风格
- 所有 API 必须有 OpenAPI 文档
- 类型注解必备
- 函数必须有 docstring

## 禁止操作
- 不要手动修改 migrations 目录,用 alembic revision --autogenerate
- 不要直接操作数据库,必须通过 Repository 层
- 不要在 services 层直接返回 ORM 对象,必须通过 schema 转换
- 不要使用 print(),用 logging
- 不要在代码中硬编码配置,用环境变量

## 必做检查
- 修改后必须运行: `pytest tests/ -v`
- 提交前运行: `ruff check . && black --check .`
- API 改动后必须更新: `docs/api.md`

## 常用命令
- 启动开发服务器: `uvicorn src.main:app --reload`
- 运行测试: `pytest tests/ -v --cov=src`
- 数据库迁移: `alembic upgrade head`
- 生成迁移: `alembic revision --autogenerate -m "description"`

## 业务规则
- 订单状态:pending → paid → shipped → delivered → completed
- 取消订单只能在 pending/paid 状态
- 库存检查必须在创建订单时同步完成
- 支付超时:30 分钟未支付自动取消
```

### 3.6 AGENTS.md vs CLAUDE.md

| 维度 | AGENTS.md | CLAUDE.md |
|------|-----------|-----------|
| **标准** | 跨工具通用 | Claude Code 专属 |
| **支持工具** | Codex/Claude/Cursor/Aider 等 | Claude Code |
| **目的** | 行业标准 | Anthropic 生态 |
| **优先级** | 共存(都加载) | 共存(都加载) |
| **未来趋势** | 逐渐成为通用标准 | 与 AGENTS.md 共存 |

**实践建议:** 以 CLAUDE.md 为主、AGENTS.md 为辅(给 Codex/Cursor 等其他工具),两者可用符号链接保持内容同源;CLAUDE.md 里也可用 `@AGENTS.md` 导入语法复用同一份内容。

## 四、Rules(运行约束规则)

### 4.1 基础定义

**定义:** 定义 Agent 行为边界的规则集合,由 Host 程序强制执行。

### 4.2 Rules 与 CLAUDE.md 的区别

| 维度 | CLAUDE.md | Rules |
|------|-----------|-------|
| **性质** | 提示级软规则 | 运行时约束集 |
| **强制性** | 软(LLM 可能忽略) | 强(Host 强制执行) |
| **触发时机** | 启动加载 | 运行时实时校验 |
| **实现方式** | Markdown 文本 | Host 程序内置/配置 |
| **可绕过** | 是(LLM 可能"忘记") | 否(Host 拦截) |

### 4.3 常见的 Rules 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **文件访问规则** | 允许/禁止访问的目录、文件类型 | 禁止访问 `.env` 文件 |
| **命令执行规则** | 允许/禁止执行的命令 | 禁止 `rm -rf /` |
| **网络访问规则** | 允许/禁止的域名、API | 仅允许访问公司 API |
| **权限边界** | Agent 可操作的资源范围 | 只能修改 src/ 目录 |
| **工具调用规则** | 限制可调用的工具 | 禁用 Bash 工具 |
| **时间限制** | 单次执行最大时间 | 30 秒超时 |
| **资源限制** | CPU、内存、Token 上限 | 单次最多 100k Token |

### 4.4 Rules 在不同工具中的实现

| 工具 | Rules 实现 |
|------|------------|
| **Claude Code** | settings.json + 命令行参数 + Permissions |
| **Codex CLI** | config.toml + sandbox 配置 |
| **Cursor** | .cursorrules 文件(已被 .cursor/rules 取代) |
| **Aider** | .aider.conf.yml 配置 |
| **Continue** | config.json |

### 4.5 Claude Code 权限配置示例

`.claude/settings.json`:

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Edit",
      "Write",
      "Bash(npm run *)",
      "Bash(git *)",
      "Bash(pytest *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl * | sh)",
      "Read(.env)",
      "Read(.env.*)",
      "Read(**/secrets/**)"
    ]
  },
  "model": "claude-sonnet-4-5",
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1"
  },
  "cleanupPeriodDays": 20
}
```

> 注:settings.json 没有 `maxTokens`、`timeout` 这类字段。真实常用字段:`permissions`(allow/deny/ask)、`model`、`env`、`hooks`、`statusLine`、`cleanupPeriodDays`、`disableAllHooks` 等,完整列表见官方 Settings 文档(https://code.claude.com/docs/en/settings)。

### 4.6 Codex CLI 沙箱配置

`~/.codex/config.toml`(真实字段,顶层平铺,不存在 `[sandbox]` / `[permissions]` 表):

```toml
# 沙箱模式:read-only / workspace-write / danger-full-access
sandbox_mode = "workspace-write"

# 审批策略:untrusted / on-failure / on-request / never
approval_policy = "on-request"

# workspace-write 模式的细化选项(网络默认关闭)
[sandbox_workspace_write]
network_access = false

# 受信任的 MCP Server / 工具可按官方文档追加,以官方 config 文档为准:
# https://developers.openai.com/codex/config-reference
```

### 4.7 Cursor Rules(`.cursor/rules`)与 GitHub Copilot 指令

Cursor 的规则文件已从单个 `.cursorrules`(已弃用)演进为 `.cursor/rules/*.mdc` 目录式管理。**MDC = YAML frontmatter + Markdown 正文**,frontmatter 三个字段决定触发方式:

```markdown
---
description: React 组件编码规范,写组件时必须遵守
globs:
  - "src/components/**/*.tsx"
  - "src/components/**/*.ts"
alwaysApply: false
---

# React 组件规范

- 一律使用函数式组件 + Hooks,禁止 class 组件
- Props 必须定义 TypeScript interface,命名 `XxxProps`
- 样式用 Tailwind,不写内联 style
- 每个组件文件配一个同名 `.test.tsx`
```

**三种触发方式(由 frontmatter 组合决定):**

| 触发方式 | frontmatter 组合 | 行为 |
|----------|------------------|------|
| **Always(常驻)** | `alwaysApply: true` | 每次请求都注入上下文 |
| **Auto Attached(按文件)** | 设了 `globs`,`alwaysApply: false` | 上下文中有匹配文件时自动附加 |
| **Agent Requested(智能判断)** | 只设 `description` | Agent 按描述判断相关性后自行引用 |

> 无 frontmatter 的 .mdc 默认为 Manual,只有 `@` 提及时才生效,新手最常踩这个坑。

**GitHub Copilot 的等价物:** 仓库根的 `.github/copilot-instructions.md`(全仓生效)与 `.github/instructions/*.instructions.md`(可用 `applyTo` glob 限定范围),VS Code Copilot 自动加载,写法同为 Markdown,作用与 Cursor Rules 类似。

## 五、Hooks(生命周期钩子)

### 5.1 基础定义

**定义:** 在 Agent 生命周期事件(如工具调用前后)触发的确定性脚本,100% 强制执行。

### 5.2 Hooks 的核心特点

| 特点 | 说明 |
|------|------|
| **事件驱动** | 在特定生命周期事件发生时自动触发 |
| **确定性** | 脚本逻辑固定,不受 LLM 决策影响 |
| **强制性** | 无论 LLM 是否愿意都会执行 |
| **可拦截** | Pre 类钩子可阻止操作执行 |
| **可修改** | 钩子可修改工具参数、结果 |
| **可观测** | 钩子可记录日志、上报指标 |

### 5.3 Hooks 事件类型(Claude Code 官方)

| 事件 | 触发时机 | 用途 | 可拦截 |
|------|----------|------|--------|
| `SessionStart` | 会话开始/恢复时 | 注入环境上下文、持久化环境变量 | ✗ |
| `SessionEnd` | 会话终止时 | 清理资源、归档日志 | ✗ |
| `UserPromptSubmit` | 用户提交输入时 | 输入预处理、敏感词过滤 | ✓ |
| `PreToolUse` | 工具调用前 | 拦截危险操作、修改参数、记录日志 | ✓ |
| `PostToolUse` | 工具调用成功后 | 后置处理、结果验证、格式化 | ✗ |
| `Notification` | 系统通知时 | 通知转发、告警 | ✗ |
| `PreCompact` | 上下文压缩前 | 压缩前保存关键状态到文件 | ✓ |
| `Stop` | Agent 结束响应时 | 清理资源、生成报告、阻止结束 | ✓ |
| `SubagentStop` | 子 Agent 结束时 | 子任务结果处理 | ✓ |

> 完整事件还有 `PostToolUseFailure`、`SubagentStart`、`PostCompact`、`Setup` 等,以官方 Hooks 文档为准。事件按频次分三类:每会话一次(SessionStart/SessionEnd)、每轮一次(UserPromptSubmit/Stop)、每次工具调用(PreToolUse/PostToolUse)。

### 5.4 Hooks 配置示例(Claude Code)

在 `.claude/settings.json` 配置事件钩子。**注意:hook 输入一律通过 stdin 传入 JSON,不存在 `${FILE_PATH}` 这类占位符插值**,取参数必须写脚本(或用 jq)解析:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python ~/.claude/hooks/check-bash.py"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python ~/.claude/hooks/format-file.py"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Agent stopped' >> ~/.claude/logs/agent.log"
          }
        ]
      }
    ]
  }
}
```

### 5.5 PreToolUse 钩子脚本示例(从 stdin 读 JSON)

上面 PostToolUse 引用的 `~/.claude/hooks/format-file.py`(编辑后自动格式化)同样从 stdin 取文件路径:

```python
import sys, json, subprocess

data = json.load(sys.stdin)
path = data.get("tool_input", {}).get("file_path", "")
if path.endswith((".js", ".ts", ".tsx")):
    subprocess.run(["npx", "prettier", "--write", path], check=False)
sys.exit(0)
```

`~/.claude/hooks/check-bash.py`:

```python
#!/usr/bin/env python3
# Windows 下 settings.json 里写 "python"(不是 python3)
import sys
import json
import re

# hook 输入一律走 stdin 的 JSON,无环境变量/占位符插值
input_data = json.load(sys.stdin)

tool_name = input_data.get("tool_name", "")
tool_input = input_data.get("tool_input", {})

if tool_name == "Bash":
    command = tool_input.get("command", "")

    # 检查危险命令
    dangerous_patterns = [
        r"rm\s+-rf\s+/",
        r"curl.*\|\s*sh",
        r"wget.*\|\s*sh",
        r"sudo\s+rm",
        r"chmod\s+777",
        r"git\s+push\s+--force"
    ]

    for pattern in dangerous_patterns:
        if re.search(pattern, command):
            # 现行推荐输出:hookSpecificOutput + permissionDecision
            print(json.dumps({
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "permissionDecision": "deny",
                    "permissionDecisionReason": f"检测到危险命令模式: {pattern}"
                }
            }))
            sys.exit(0)

# 不输出任何 JSON、exit 0 = 放行(也可显式 permissionDecision: "allow")
sys.exit(0)
```

**输出与退出码语义(2025 年后现行规范):**

| 方式 | 效果 |
|------|------|
| exit 0 + 无输出 | 放行,无附加信息 |
| exit 0 + stdout JSON | 结构化控制:`permissionDecision` 为 `allow` / `deny` / `ask`(升级给用户确认) |
| exit 2 | **阻断**,stdout 被忽略,**stderr 内容反馈给 Claude** 让它自我修正 |
| 其他退出码 | 非阻断错误,仅记录日志,流程继续 |

**修改工具参数:** PreToolUse 可在 `hookSpecificOutput` 里返回 `updatedInput` 字段改写入参(旧版 `decision: "modify"` 已废弃);PostToolUse 可用 `updatedToolOutput` 改写工具结果。

### 5.6 Hooks 应用场景

| 场景 | 钩子事件 | 实现方式 |
|------|----------|----------|
| 自动格式化代码 | PostToolUse (Edit/Write) | 运行 prettier/black |
| 阻止危险命令 | PreToolUse (Bash) | 检查命令模式 |
| 自动 lint | PostToolUse (Edit) | 运行 eslint/ruff |
| 提交前测试 | PreToolUse (Git commit) | 运行 pytest |
| 敏感数据脱敏 | UserPromptSubmit | 替换 API key 等 |
| 日志记录 | 所有事件 | 写入日志文件 |
| 通知团队 | Stop | 发送 Slack 通知 |
| 自动补全 import | PostToolUse (Edit) | 运行 isort |
| 安全扫描 | PreToolUse (Bash) | 检查命令安全性 |

### 5.7 Hooks 的工作流程

```
用户输入或工具调用
    ↓
[Claude Code 触发对应 Hook 事件]
    ↓
[执行 Hook 脚本(stdin 传入 JSON 上下文)]
    ↓
[脚本输出决策结果]
    ↓
    ├── exit 0 无输出 → 放行
    ├── exit 0 + JSON → permissionDecision: allow / deny / ask
    │                   (PreToolUse 还可用 updatedInput 改参数)
    └── exit 2 → 阻断,stderr 反馈给 Claude 自我修正
```

**官方文档:** https://code.claude.com/docs/en/hooks

### 5.8 五分钟可验证的 Hooks 实战

目标:让 Claude Code 每次要执行含 `rm -rf` 的 Bash 命令时都被拦下,并亲眼看到拦截反馈。

**第 1 步:写钩子脚本**

新建 `~/.claude/hooks/block-rm.py`(Windows 为 `C:\Users\<你>\.claude\hooks\block-rm.py`):

```python
import sys, json

data = json.load(sys.stdin)
cmd = data.get("tool_input", {}).get("command", "")

if "rm -rf" in cmd:
    # exit 2:阻断,stderr 会被反馈给 Claude
    print(f"已阻止危险命令: {cmd}", file=sys.stderr)
    sys.exit(2)

sys.exit(0)  # 其他命令放行
```

**第 2 步:注册钩子**

在 `~/.claude/settings.json` 的 `hooks` 字段中加入(没有该字段就新建):

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "python ~/.claude/hooks/block-rm.py" }
        ]
      }
    ]
  }
}
```

**第 3 步:验证**

1. 重启 Claude Code(或在会话里直接试,settings 变更会被文件监听自动加载)
2. 让它执行一条危险命令,例如输入:`运行 rm -rf /tmp/test`
3. 预期现象:Bash 调用被阻止,Claude 收到 `已阻止危险命令: ...` 的反馈并改口解释"命令被安全钩子拦截"
4. 正常命令(如 `ls`)不受影响

**Windows 注意事项:**

| 坑 | 修法 |
|----|------|
| 没有 `python3` 命令 | `command` 写 `python`(Windows 通常只有 python.exe) |
| 路径分隔符 | 写 `~/.claude/hooks/block-rm.py` 或正斜杠绝对路径均可;避免反斜杠在 JSON 里转义出错 |
| 找不到 python | 写绝对路径,如 `C:/Python312/python.exe` |
| 脚本没触发 | 在 Claude Code 里输入 `/hooks` 查看已注册钩子,确认 matcher 与命令拼写 |

**扩展:** 把 `if "rm -rf" in cmd` 换成正则黑名单、把结果写日志,就是生产可用的命令审计钩子;想改工具参数而不是阻断,返回 exit 0 + `{"hookSpecificOutput": {"hookEventName": "PreToolUse", "updatedInput": {...}}}` 即可。

## 六、CLAUDE.md vs Rules vs Hooks 三层关系

### 6.1 三层架构图

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: CLAUDE.md / AGENTS.md  ← 持久化"软规则"        │
│                                                         │
│  启动时自动加载,定义项目规则、编码规范                  │
│  特点:LLM 可遵守也可忽略(非强制)                       │
│  作用:引导 LLM 行为,提供项目上下文                     │
├─────────────────────────────────────────────────────────┤
│  Layer 2: Rules / Permissions    ← 运行时"约束集"        │
│                                                         │
│  显式声明不可触碰的边界(文件/命令)                     │
│  特点:由 Host 强制执行,LLM 无法绕过                    │
│  作用:硬性边界,防止越权操作                            │
├─────────────────────────────────────────────────────────┤
│  Layer 3: Hooks                  ← 事件"硬触发"          │
│                                                         │
│  在生命周期事件(PreToolUse/PostToolUse 等)             │
│  执行确定性脚本,100% 强制                               │
│  特点:与 CLAUDE.md 互补,补足确定性                     │
│  作用:自动化流程、强制检查、拦截危险操作                │
└─────────────────────────────────────────────────────────┘
```

### 6.2 三层对比

| 维度 | CLAUDE.md(软规则) | Rules(约束集) | Hooks(事件钩子) |
|------|---------------------|----------------|-------------------|
| **强制性** | 低(LLM 可忽略) | 高(Host 强制) | 100%(脚本执行) |
| **触发** | 启动加载 | 运行时校验 | 生命周期事件 |
| **可绕过** | 是 | 否 | 否 |
| **实现** | Markdown 文本 | 配置文件 | 可执行脚本 |
| **作用** | 引导 LLM | 拦截越权 | 自动化 + 拦截 |
| **示例** | "不要使用 any 类型" | 禁止访问 .env | 编辑后自动格式化 |

### 6.3 实践建议

| 场景 | 推荐机制 |
|------|----------|
| 项目编码规范 | CLAUDE.md / AGENTS.md |
| 业务逻辑说明 | CLAUDE.md / AGENTS.md |
| 禁止访问敏感文件 | Rules(权限配置) |
| 禁止危险命令 | Rules + Hooks 双保险 |
| 自动格式化代码 | Hooks(PostToolUse) |
| 提交前测试 | Hooks(PreToolUse) |
| 强制安全检查 | Hooks(PreToolUse) |
| 团队规范共享 | AGENTS.md(提交到 git) |

## 七、Plugin(插件)

### 7.1 基础定义

**定义:** 扩展 Agent 能力的可插拔模块,通常可包含可执行代码。

### 7.2 Plugin 与 Skill 的区别

| 维度 | Skill | Plugin |
|------|-------|--------|
| **形态** | Markdown 指令包 | 可包含代码的扩展模块 |
| **能力** | 知识/流程封装 | 可接入外部系统、自定义逻辑 |
| **复杂度** | 低 | 高 |
| **安装** | 放入目录即可 | 需要安装注册 |
| **执行** | LLM 按 Markdown 执行 | 程序级执行 |
| **安全** | 低风险(纯文本) | 需要审计(可执行代码) |
| **示例** | 代码审查 Skill | GitHub MCP Server |

### 7.3 典型 Plugin 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **MCP Server** | 通过 MCP 协议接入外部能力 | filesystem MCP Server |
| **自定义工具集** | 多个相关工具的打包 | 团队内部 DevOps 工具集 |
| **系统集成** | 与企业内部系统集成 | Jira/Confluence/Slack 集成 |
| **数据连接器** | 接入特定数据源 | Notion/数据库连接器 |
| **代码执行环境** | 提供代码执行沙箱 | Python/Node.js 执行环境 |

### 7.4 官方 Plugin 市场(2025-10 起)

Claude Code 于 2025-10 推出官方 Plugin 体系与市场(marketplace):一个 Plugin 可同时打包 **Subagents + Skills + Hooks + MCP Servers + 斜杠命令**,通过 `/plugin` 命令浏览、安装、更新。

- **安装:** `/plugin marketplace add <repo>` 添加市场源,`/plugin install <name>@<marketplace>` 安装
- **结构:** 插件根目录含 `plugin.json` 清单 + `agents/`、`skills/`、`hooks/`、`commands/` 等子目录
- **意义:** 此前零散的 skills/agents 配置有了官方分发渠道,团队可以把自己的工程实践打包成 Plugin 在组织内统一分发
- **企业治理:** 管理员可通过托管策略限定可用市场与插件白名单

### 7.5 Skill/Plugin 供应链安全(约束视角)

Skill 与 Plugin 是"能力"入口,也是新的供应链攻击面,与本章"约束"定位直接相关:

| 风险 | 说明 | 对策 |
|------|------|------|
| **恶意 SKILL.md** | description/正文里藏提示注入,诱导 Agent 泄露数据或执行危险操作 | 安装前通读全文,警惕"忽略之前指令""发送到某 URL"类语句 |
| **捆绑恶意代码** | Plugin 内 hooks/scripts 可执行任意命令 | 审计 hooks 与脚本;只从可信市场安装 |
| **工具清单投毒** | MCP Server 的工具 description 夹带私货 | 审查 `tools/list` 输出;最小化 allowed-tools |
| **更新劫持** | 已装插件被上游换成恶意版本 | 锁定版本/审计 diff;企业用私有市场 |

**审计清单:** ① 读 SKILL.md 全文;② 检查 hooks 与脚本调用;③ 核对 `allowed-tools` 是否最小;④ 确认来源仓库可信;⑤ 定期复查已装插件更新。

## 八、Tool Calling(工具调用)

### 8.1 基础定义

**定义:** LLM 选择并调用外部工具的通用行为,比 Function Calling 更宽泛。

### 8.2 Function Calling(OpenAI 规范)

**发布方:** OpenAI,2023-06

**原理:** LLM 输出结构化 JSON,告诉调用方"我想执行哪个工具,参数是什么"。

```json
{
  "tool": "search_web",
  "arguments": {
    "query": "MCP 协议最新规范"
  }
}
```

### 8.3 工具定义要素

| 要素 | 说明 |
|------|------|
| `name` | 工具唯一标识 |
| `description` | 清晰描述功能(LLM 据此选择工具) |
| `inputSchema` | JSON Schema 格式的参数定义 |
| `annotations` | 工具元数据(MCP 2025-03 新增) |

### 8.4 工具调用完整流程

```
1. 用户输入任务
   ↓
2. LLM 分析任务,加载可用工具列表
   ↓
3. LLM 决定调用哪个工具
   ↓
4. LLM 生成工具调用参数(JSON)
   ↓
5. Host 接收工具调用请求
   ↓
6. Host 检查权限(Rules + Hooks)
   ↓
7. Host 执行工具(本地函数 / MCP Server / API)
   ↓
8. 工具返回结果
   ↓
9. Host 把结果交给 LLM
   ↓
10. LLM 基于结果继续推理或输出最终答案
```

### 8.5 Tool Calling 在不同平台的实现

| 平台 | 接口名称 | 特点 |
|------|----------|------|
| **OpenAI** | Function Calling / Tools API | 最早标准化,生态最完善 |
| **Anthropic** | Tool Use | Claude 的工具调用接口 |
| **Google Gemini** | Function Calling | Gemini 的工具调用 |
| **Mistral** | Function Calling | 兼容 OpenAI 格式 |
| **本地模型** | 各家自定义 | 通过 vLLM/Ollama 兼容 OpenAI 格式 |

### 8.6 官方文档

- **OpenAI Function Calling:** https://platform.openai.com/docs/guides/function-calling
- **Anthropic Tool Use:** https://docs.anthropic.com/en/docs/build-with-claude/tool-use
- **Gemini Function Calling:** https://ai.google.dev/gemini-api/docs/function-calling

## 九、Tool Schema(工具描述规范)

### 9.1 基础定义

**定义:** 描述工具名称、功能、参数格式的规范,通常使用 JSON Schema。

### 9.2 Tool Schema 核心要素

| 要素 | 类型 | 说明 |
|------|------|------|
| **name** | string | 工具名(唯一标识,kebab-case) |
| **description** | string | 功能描述(LLM 据此选择工具) |
| **inputSchema** | object | 参数定义(JSON Schema 格式) |
| **annotations** | object | 工具元数据(MCP 2025-03 新增) |

### 9.3 JSON Schema 示例

```json
{
  "name": "read_file",
  "description": "读取指定路径的文件内容",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "文件绝对路径"
      },
      "encoding": {
        "type": "string",
        "default": "utf-8",
        "description": "文件编码"
      }
    },
    "required": ["path"]
  },
  "annotations": {
    "readOnlyHint": true,
    "destructiveHint": false,
    "idempotentHint": true,
    "openWorldHint": false
  }
}
```

### 9.4 Tool Annotations(MCP 2025-03 新增)

| 注释字段 | 类型 | 说明 |
|----------|------|------|
| `readOnlyHint` | bool | 工具只读取不修改(安全等级最高) |
| `destructiveHint` | bool | 工具可能破坏数据(需用户确认) |
| `idempotentHint` | bool | 重复调用幂等(多次调用结果一致) |
| `openWorldHint` | bool | 工具与外部世界交互(网络调用) |

Host 可根据这些注释决定是否自动批准或要求用户确认。

### 9.5 Tool 描述的最佳实践

| 实践 | 说明 |
|------|------|
| **描述清晰** | 让 LLM 一眼看出工具用途 |
| **参数有默认值** | 减少必填参数 |
| **必填参数最少** | 只把真正必要的设为 required |
| **参数有描述** | 每个参数都要 description |
| **类型明确** | 用 JSON Schema 严格类型 |
| **提供示例** | 在 description 中给使用示例 |
| **错误处理** | 工具应返回清晰错误信息 |

## 十、API Tool(接口工具)

### 10.1 基础定义

**定义:** 封装 HTTP API 为可被 LLM 调用的工具。

### 10.2 实现方式

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **手写封装** | 把 REST API 包装为函数 | 灵活 | 重复劳动 |
| **MCP Server** | 通过 MCP 协议标准化接入 | 标准化、可复用 | 需要写 Server |
| **OpenAPI 自动生成** | 根据 OpenAPI 规范自动生成工具 | 自动化 | 依赖规范完整性 |
| **Postman 集合转换** | 从 Postman 集合生成工具 | 复用现有 API 测试 | 需要 Postman |

### 10.3 典型 API Tool 示例

```python
# 手写封装示例
def get_weather(city: str) -> dict:
    """查询天气

    Args:
        city: 城市名(中文/英文)

    Returns:
        天气信息(温度、湿度、风力)
    """
    import requests
    response = requests.get(
        f"https://api.weather.com/v1/current",
        params={"city": city, "key": API_KEY}
    )
    return response.json()
```

## 十二、Sandbox(执行沙箱)

### 12.1 基础定义

**定义:** 隔离的代码执行环境,防止恶意代码影响宿主系统。

### 12.2 沙箱实现方式对比

| 方式 | 原理 | 隔离强度 | 性能 | 代表 |
|------|------|----------|------|------|
| **内核级隔离** | OS 内核特性 | 高 | 高 | macOS seatbelt、Linux landlock |
| **容器隔离** | Linux Namespace + cgroups | 中 | 中 | Docker、Podman |
| **进程级隔离** | 独立进程 + 权限限制 | 低 | 高 | seccomp、AppArmor |
| **虚拟机隔离** | 完整 OS 虚拟化 | 最高 | 低 | Firecracker、gVisor |
| **云环境隔离** | 远程执行环境 | 最高 | 低 | AWS Lambda、Modal |
| **WebAssembly** | WASM 沙箱 | 高 | 高 | wasmtime、wasmer |

### 12.3 Codex CLI 沙箱模式

Codex CLI(Codex)默认采用内核级沙箱:

| 模式 | 说明 |
|------|------|
| **read-only** | 只读,不能修改任何文件 |
| **workspace-write** | 只能写工作目录,网络受限 |
| **danger-full-access** | 完全访问(需显式确认) |

**实现细节:**
- macOS:使用 `seatbelt`(`sandbox-exec`)
- Linux:使用 `landlock`(LSM)

### 12.4 沙箱的作用

| 作用 | 说明 |
|------|------|
| **防止系统破坏** | 阻止 `rm -rf /` 等危险操作 |
| **限制文件访问** | 只能访问工作目录 |
| **限制网络访问** | 防止数据外泄 |
| **资源限制** | CPU/内存/时间上限 |
| **进程隔离** | 防止进程逃逸 |
| **审计日志** | 记录所有操作 |

## 十三、能力与约束三层体系总结

### 13.1 关键配置文件对照

| 文件 | 工具 | 位置 | 作用 |
|------|------|------|------|
| `CLAUDE.md` | Claude Code | 项目根 | 项目级行为规则(软约束) |
| `AGENTS.md` | 通用标准 | 项目根 | 跨工具项目规则 |
| `.claude/settings.json` | Claude Code | 项目根 | 权限和 Hooks 配置 |
| `.claude/skills/` | Claude Code | 项目根 | 自定义技能 |
| `.claude/agents/` | Claude Code | 项目根 | 自定义 Subagent |
| `.cursor/rules/` | Cursor | 项目根 | Cursor 规则(新) |
| `.cursorrules` | Cursor | 项目根 | Cursor 规则(旧,已弃用) |
| `config.toml` | Codex CLI | `~/.codex/` | 全局配置 + 沙箱 |
| `.codex/skills/` | Codex CLI | 项目根 | Codex Skills |
| `.aider.conf.yml` | Aider | 项目根 | Aider 配置 |

### 13.2 能力与约束三层体系

| 层级 | 机制 | 强制性 | 触发 | 示例 |
|------|------|--------|------|------|
| **软规则** | CLAUDE.md / AGENTS.md | 低(LLM 可忽略) | 启动加载 | "不要使用 any 类型" |
| **约束集** | Rules / 权限配置 | 高(Host 强制) | 运行时校验 | 禁止访问 .env |
| **硬触发** | Hooks | 100%(脚本执行) | 生命周期事件 | 编辑后自动格式化 |

### 13.3 决策树:该用哪种机制?

```
你想约束/扩展 Agent 行为?
│
├─ 想引导 LLM 行为(可绕过)?
│  └─ → CLAUDE.md / AGENTS.md
│
├─ 想强制禁止某操作?
│  └─ → Rules / 权限配置
│
├─ 想在事件发生时自动执行?
│  └─ → Hooks
│
├─ 想封装可复用的能力?
│  └─ → Skill
│
├─ 想接入外部系统?
│  └─ → Plugin / MCP Server
│
└─ 想隔离代码执行环境?
   └─ → Sandbox
```
